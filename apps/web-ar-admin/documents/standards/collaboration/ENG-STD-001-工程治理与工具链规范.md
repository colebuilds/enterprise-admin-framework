# ENG-STD-001 工程治理与工具链规范

## 1. 文档定位

本文档是工程治理领域的总规范，用于说明：

1. 工程治理要解决什么问题
2. 工具链各自负责什么
3. 本地开发、提交前、CI 分别执行哪些检查
4. 各子规范之间的关系
5. 哪些规则为强制门禁，哪些为配套建议

本文档不承载全部细节规则。提交规范、样式规范、依赖边界规范、检查与发布门禁规范由对应子规范承载。

## 2. 目标

本规范用于解决以下问题：

- 代码风格不统一
- 样式规范失控
- 提交信息不规范
- 分层架构被破坏
- Monorepo 包边界失控
- 本地能过、提交后失败、CI 再返工
- 工具链过多、配置分散、职责重叠

工程治理的目标不是单纯增加工具，而是建立一套清晰、稳定、低重复、可执行的约束系统。

## 3. 核心原则

### 3.1 单一职责

每个工具只解决一类核心问题，不重复覆盖。

### 3.2 统一入口

能收口的静态检查尽量收口，不鼓励开发者记忆过多分散命令。

### 3.3 分层门禁

检查分为三层：

- 本地开发主动检查
- Git 提交前拦截
- CI 全量门禁

### 3.4 规则显式化

规则必须写入文档和配置，不依赖口头约定。

### 3.5 架构优先

工程治理不只管格式，还必须保护架构边界。

### 3.6 渐进治理

老项目允许先建立基线，再逐步收敛，不要求一次性清零全部历史问题。

## 4. 工具链分工

### 4.1 统一工程入口

建议由统一工程入口工具收口以下能力：

- `dev`
- `build`
- `check`
- `test`

该入口适合作为日常脚本入口，但不覆盖样式 lint、提交规范、依赖边界治理。

### 4.2 Stylelint

Stylelint 负责样式质量治理，包括：

- CSS
- SCSS
- Less
- Vue SFC 中的 style 块

样式规则不并入 JS/TS lint，保持职责清晰。

### 4.3 Commitlint

Commitlint 负责 commit message 规范校验，用于：

- 统一提交格式
- 支持后续 changelog / release / 版本治理
- 提升需求追踪和变更审计能力

### 4.4 Dependency Cruiser

Dependency Cruiser 负责依赖关系治理，包括：

- 循环依赖
- 分层依赖违规
- 跨 app 源码引用
- 跨 package 非公开入口引用
- 测试目录被生产代码误引用

它的定位是架构边界检查器，不是单纯依赖图工具。

### 4.5 Husky

Husky 负责把本地 Git hooks 落地到开发流程中。

### 4.6 lint-staged

lint-staged 负责在提交前只检查暂存区文件，降低本地等待成本。

## 5. 工程治理边界

本规范覆盖以下内容：

- 脚本与静态检查统一入口
- 样式规范检查
- 提交规范检查
- 架构与依赖边界检查
- 本地 / 提交前 / CI 检查策略

本规范不直接覆盖以下内容：

- 业务埋点规范
- OpenAPI 规范
- 测试设计规范
- 文档编写规范
- AI 协作规范

这些内容应由各自专题规范负责。

## 6. 执行层级

### 6.1 本地开发层

开发者在编码过程中应主动执行：

- `dev`
- `check`
- `test`
- 必要时执行样式检查和依赖边界检查

目标是尽量在本地发现问题，而不是提交后再返工。

### 6.2 提交前层

提交前通过 Git hooks 自动执行：

- staged 文件静态检查
- staged 文件样式检查
- commit message 校验

目标是在最短反馈路径拦住低质量提交。

### 6.3 CI 层

CI 执行全量检查：

- 全量静态检查
- 全量样式检查
- 依赖边界检查
- 测试

目标是保证进入主干的代码满足统一标准。

## 7. 检查与测试分层原则

为避免脚本职责重叠，工程检查与测试命令应分层设计，不建议长期维持单一模糊入口。

### 7.1 `check` 的职责

`check` 用于开发过程中的主动检查，重点解决代码与工程的基础静态质量问题。  
它适合作为日常开发默认入口，但不等同于完整发布门禁。

### 7.2 `verify` 的职责

`verify` 用于提交前或 CI 的完整门禁入口。  
其职责是聚合当前项目要求的关键检查，确保变更在进入主干前满足统一质量要求。

### 7.3 `test` 的职责

`test` 用于统一触发项目测试能力，但不应长期只保留单一模糊入口。  
测试命令应按测试层级拆分，至少明确以下边界：

- 单元测试
- 集成测试
- E2E 测试

### 7.4 测试层级边界

- 单元测试用于验证局部模块、函数或组件的独立行为
- 集成测试用于验证模块之间协作、接口拼装与流程组合
- E2E 测试用于验证真实用户链路、关键业务路径与系统级交互行为

### 7.5 细节归属

具体脚本命名、命令组合方式、执行时机、是否进入提交前门禁、是否进入 CI 全量门禁，应由对应专项规范进一步定义，不在本总规范中展开。

## 8. 规则分级

### 8.1 `error`

- 阻断提交或阻断 CI
- 必须处理后才能继续

适用场景：

- 循环依赖
- 严重分层违规
- 提交信息格式错误
- 严重样式错误
- 静态检查失败

### 8.2 `warn`

- 不阻断
- 允许临时通过
- 必须被跟踪和消化

适用场景：

- 历史技术债
- 渐进治理中的遗留问题
- 暂时无法立即修复的低风险问题

## 9. 规范与子规范关系

本规范作为总规范，统领以下子规范：

- `ENG-STD-002-提交规范.md`
- `ENG-STD-003-样式规范.md`
- `ENG-STD-004-依赖边界与架构约束规范.md`
- `ENG-STD-005-检查与发布门禁规范.md`

约束关系如下：

- 本规范定义总目标、工具分工、执行层级
- 子规范定义具体规则、例外机制、命令与配置要求

当总规范与子规范发生冲突时，以更具体的子规范为准；若子规范未覆盖，则以本规范为准。

## 11. 推荐落地顺序

建议按以下顺序落地：

### 第一阶段：最小可用

- 接入统一工程入口
- 接入 Stylelint
- 接入 Commitlint
- 接入 Husky + lint-staged

### 第二阶段：架构治理

- 接入 Dependency Cruiser
- 建立基础依赖规则
- 对历史问题建立 baseline

### 第三阶段：CI 完整门禁

- 在 PR / 主干流程中启用 `verify`
- 建立阻断级别和例外机制

### 第二批建议补充能力

以下能力不属于首批最小治理闭环，但建议在项目稳定后逐步补齐：

- Changesets / 版本治理
- 构建体积预算检查
- Browserslist 与兼容性治理
- 依赖清理与死代码治理
- 测试质量增强
- 发布质量门禁
- 可观测性与埋点联动

这些能力属于第二批再补，目的是让团队先建立最小可用治理闭环，再逐步扩大治理半径。

## 12. 例外机制

当某条规则暂时无法满足时，允许申请例外，但必须满足：

1. 有明确原因
2. 有影响评估
3. 有临时范围
4. 有后续治理计划
5. 不允许长期无期限豁免

所有例外应记录在对应专项规范或技术债清单中，不允许口头跳过。

## 13. 维护原则

- 工具升级时先验证兼容性，再更新规范
- 新增工具必须说明解决什么问题、为什么现有工具不能解决
- 不允许重复引入职责重叠工具
- 规范更新后必须同步更新接入手册和脚本

## 14. 一句话总结

工程治理不是堆工具，而是建立一套：

- 职责清晰
- 入口统一
- 分层执行
- 可阻断
- 可渐进治理

的工程约束系统。

本规范作为总规范，负责统一这套系统的目标、边界与执行方式。
