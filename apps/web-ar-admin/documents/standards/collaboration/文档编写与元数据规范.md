# 文档编写与元数据规范

## 1. 目标

统一项目文档的编写方式、元数据字段、命名规则、编号规则与模板结构，保证文档具备：

- 可读性
- 可检索性
- 可维护性
- 可追溯性
- 可供 AI 与人共同稳定消费

## 2. 适用范围

本规范适用于 `documents/` 下的正式文档，包括但不限于：

- `standards/`
- `guides/`
- `design/`
- `decisions/`
- `evolution/`
- `features/`
- `reference/`
- `known-issues/`

不强制适用于以下内容：

- 临时草稿
- 临时会议速记
- 纯素材文件
- 非正式个人备忘

如临时文档后续需要进入正式体系，应补齐本规范要求的字段与结构。

## 3. 核心原则

### 3.1 稳定优先

文档文件名应长期稳定，避免因小改动频繁改名。

### 3.2 元信息前置

正式文档应在文档顶部声明必要元信息，便于人和 AI 快速理解文档属性。

### 3.3 类型明确

每篇文档应明确自己属于规范、手册、方案、决策、演进、需求还是参考资料。

### 3.4 结构清晰

文档应具备清晰标题层级，避免大段无结构堆叠。

### 3.5 便于维护

文档字段不追求过度复杂，以满足“找得到、看得懂、能更新”为优先。

## 4. 文档类型定义

### 4.1 `standard`

规范类文档。回答“应该怎样做”。

### 4.2 `guide`

手册类文档。回答“具体怎么用”。

### 4.3 `design`

方案类文档。回答“本次准备怎么设计和落地”。

### 4.4 `decision`

决策类文档。回答“关键设计为什么这样定”。

### 4.5 `evolution`

演进类文档。记录架构、能力、流程随时间的变化过程。

### 4.6 `feature`

需求/迭代类文档。记录功能需求、产品背景、阶段实现方案。

### 4.7 `reference`

参考资料类文档。用于承载背景知识、概念说明、外部方案调研等内容。

## 5. 元数据书写规范

正式文档建议在开头使用统一元信息头。推荐使用 YAML frontmatter。

示例：

```yaml
---
title: Bug 修复规范
type: standard
code: FE-QUALITY-002
status: stable
author: cole
owner: cole
created: 2026-03-26
updated: 2026-03-26
---
```

## 6. 通用基础字段

以下字段适用于大多数正式文档。

### 6.1 `title`

文档标题。必填，应与正文一级标题保持一致或基本一致。

### 6.2 `type`

文档类型。可选值：

- `standard`
- `guide`
- `design`
- `decision`
- `evolution`
- `feature`
- `reference`

### 6.3 `status`

文档状态。可选值：

- `draft`
- `review`
- `stable`
- `deprecated`
- `archived`

### 6.4 `author`

文档作者。支持单人或多人，多人使用英文逗号分隔。

示例：

```yaml
author: cole
author: cole, alice
```

### 6.5 `owner`

当前维护负责人。支持单人或多人，多人使用英文逗号分隔。

示例：

```yaml
owner: cole
owner: cole, alice
```

### 6.6 `created`

创建时间。格式：`YYYY-MM-DD`

### 6.7 `updated`

最后更新时间。格式：`YYYY-MM-DD`

## 7. 按类型补充字段

### 7.1 规范类文档补充字段

- `code`：规范编号
- `scope`：适用范围
- `applies_to`：适用对象

示例：

```yaml
---
title: Bug 修复规范
type: standard
code: FE-QUALITY-002
status: stable
author: cole
owner: cole
created: 2026-03-26
updated: 2026-03-26
scope: apps/*, packages/*
applies_to: human, ai-agent
---
```

### 7.2 使用手册类文档补充字段

- `tool`：对应工具名
- `tool_version`：适用工具版本或版本范围
- `audience`：目标读者

示例：

```yaml
---
title: Vitest 使用手册
type: guide
status: stable
author: cole
owner: cole
created: 2026-03-26
updated: 2026-03-26
tool: Vitest
tool_version: 4.x
audience: frontend, ai-agent
---
```

### 7.3 方案类文档补充字段

- `related_standards`：关联规范
- `related_decisions`：关联决策文档

### 7.4 决策类文档补充字段

- `decision_id`：决策编号
- `decision_date`：决策日期
- `supersedes`：替代了哪些旧决策
- `superseded_by`：被哪份新决策替代

## 8. 时间字段规范

### 8.1 时间字段应写在元信息中

不建议把时间直接写入文件名，除非该文档本身是会议纪要、阶段快照或一次性记录。

### 8.2 长期文档必须维护 `updated`

规范、手册、方案、决策类文档更新后，应同步更新 `updated` 字段。

### 8.3 文件名不因更新时间变化而变动

更新时间是元数据，不应成为长期文档文件名的一部分。

## 8.4 正式文档落盘后固定自检

只要修改的是 `documents/` 下的正式文档，执行后必须显式检查以下内容：

- frontmatter 是否完整
- `updated` 是否已同步
- `updated` 是否等于当前环境日期
- 最近一级 `README.md` 是否已挂载
- 文档落点与文档类型是否一致

其中：

- `updated` 不得凭记忆、相对时间或历史值推断
- `updated` 必须以当前任务执行时的实际环境日期为准

## 9. 文件命名规范

### 9.1 通用原则

- 文件名应简洁、清晰、可长期引用
- 尽量避免无意义缩写
- 不应因小改动频繁改名
- 文件名主要表达主题，不表达临时状态

### 9.2 规范类文档命名

推荐格式：

`编号-主题.md`

示例：

- `FE-QUALITY-002-Bug修复规范.md`
- `FE-ARCH-004-前端请求规范.md`

### 9.3 手册类文档命名

推荐格式：

- `工具名-使用手册.md`
- `主题-操作指南.md`

示例：

- `Vitest-使用手册.md`
- `Playwright-使用手册.md`

### 9.4 方案类文档命名

推荐格式：

- `主题-架构设计方案.md`
- `主题-实施方案.md`

示例：

- `request-client-架构设计方案.md`
- `OpenAPI-自动化实施方案.md`

### 9.5 决策类文档命名

推荐格式：

`ADR-编号-标题.md`

示例：

- `ADR-012-统一请求层方案.md`

### 9.6 不推荐写法

不建议：

- `xxx-v2.md`
- `xxx-final-final.md`
- `xxx-2026-03-26.md`

这些信息更适合放在元数据中，而不是文件名中。

### 9.7 显式命名例外

默认情况下，`standard` 文档仍应优先遵守 `编号-主题.md`。

当前项目存在一个显式保留的命名例外：

- `documents/standards/collaboration/文档编写与元数据规范.md`

保留原因是：

- 它同时承担项目文档编写入口的识别作用
- 已被项目入口规则显式引用
- 该例外不应外溢到其他规范类文档

除上述明确例外外，其他 `standard` 文档仍应按推荐编号格式命名。

## 10. 编号规范

### 10.1 规范类编号

建议使用统一前缀：

- `FE-STD`：通用编码规范
- `FE-ARCH`：架构规范
- `FE-QUALITY`：质量规范
- `FE-COLLAB`：协作规范
- `FE-DOMAIN`：领域专项规范

示例：

- `FE-STD-001`
- `FE-ARCH-004`
- `FE-QUALITY-002`
- `FE-COLLAB-001`

### 10.2 决策类编号

统一使用：

- `ADR-001`
- `ADR-002`

### 10.3 手册与方案类文档

可不强制编号；若后期数量过多，可再逐步引入。

## 11. 文档结构规范

正式文档建议至少具备以下结构：

### 11.1 标题区

包含标题与必要元数据。

### 11.2 背景或目标

说明文档为什么存在。

### 11.3 正文主体

按主题拆分层级，避免无结构长段落。

### 11.4 适用范围或边界

说明适用对象与不适用对象。

### 11.5 示例

适当提供正反例或模板示例。

### 11.6 更新说明

重大改版文档可增加“变更记录”或“改动点”。

```yaml
owner: cole
created: 2026-03-26
updated: 2026-03-26

applies_to:
---
```

### 12.2 手册模板

```yaml
---
title: 文档标题
type: guide
status: draft
author: cole
owner: cole
created: 2026-03-26
updated: 2026-03-26
tool:
tool_version:
audience:
---
```

### 12.3 方案模板

```yaml
---

type: design
status: draft
author: cole
owner: cole
created: 2026-03-26
updated: 2026-03-26
```

```yaml
updated: 2026-03-26
decision_date: 2026-03-26
supersedes:
superseded_by:
---
```

## 13. 更新与废弃规则

### 13.1 更新规则

文档内容发生实质变化时，应同步更新 `updated` 字段。

### 13.2 废弃规则

当文档被新文档替代时，应将状态改为 `deprecated`，并在正文中说明替代文档。

### 13.3 归档规则

长期不再维护但仍有历史价值的文档，可标记为 `archived`，必要时移入 `archive/`。

## 14. 与 README 的关系

`documents/README.md` 负责说明文档体系总结构与目录职责。本文档负责说明文档本身该如何书写。

两者分工如下：

- `README.md`：目录地图与体系入口
- 本文档：文档编写、元数据、命名、模板规则

## 15. 最低执行要求

在正式落库时，至少应满足以下要求：

- 有明确标题
- 有 `type`
- 有 `status`
- 有 `author`
- 有 `owner`
- 有 `created`
- 有 `updated`

规范类文档额外建议具备：

- `code`

工具手册类文档额外建议具备：

- `tool`
- `tool_version`

## 16. 一句话原则

**README 解决“文档放哪里”，本规范解决“文档怎么写”。**
