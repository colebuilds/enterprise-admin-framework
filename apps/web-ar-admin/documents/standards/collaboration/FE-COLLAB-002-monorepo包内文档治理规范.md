# monorepo 包内文档治理规范

## 1. 目标

定义 monorepo 场景下根级 `documents/`、包内 `documents/` 与包根 `README.md` 的职责边界，避免文档主源不清、包内副本漂移和目录结构失控。

## 2. 适用范围

本规范适用于以下场景：

- 项目同时存在根级 `documents/` 与多个 `apps/*`、`packages/*`
- 需要判断正式文档应放在根级还是包内
- 需要为单个包建立就近文档目录
- 需要判断包根 `README.md` 与包内 `documents/README.md` 的职责
- 需要判断是否允许将根级文档复制到包内

## 3. 核心原则

### 3.1 根级 `documents/` 仍是项目级正式文档主源

以下内容必须优先放在根级 `documents/`：

- 跨包共享规范
- 项目级协作规则
- 跨包架构设计与长期决策
- 多个 app / package 共用的指南
- 项目级参考资料与长期已知问题

除明确声明外，根级正式文档仍视为权威原文。

### 3.2 包内 `documents/` 只承载当前包专属文档

包内 `documents/` 只用于承载与当前包强相关、主要由当前包维护者消费的文档，例如：

- 当前包的设计方案
- 当前包的接入细则
- 当前包的局部决策
- 当前包的迁移说明
- 当前包的调试与开发补充说明

若文档已经影响多个包，应回到根级 `documents/` 治理，而不是继续留在单包内。

### 3.3 包内文档目录统一使用 `documents/`

在 monorepo 中，单个包若需要正式文档目录，应统一使用：

```text
<package>/
  documents/
```

不建议在包内混用：

- `docs/`
- `doc/`
- `references/`

等不同命名。

### 3.4 包内不复制根级文档目录结构

包内允许建立自己的 `documents/`，但不得复制根级 `documents/` 的完整目录结构。

默认不在包内展开完整镜像，例如：

- `standards/`
- `reference/`
- `known-issues/`

包内目录应按最小需要建立，避免为了“看起来统一”而形成第二套项目文档体系。

## 4. 根级与包内的落点判断

### 4.1 应放根级 `documents/` 的情况

命中以下任一情况时，应放根级：

- 规则会被多个包共同遵守
- 内容表达的是项目级统一约束
- 内容涉及跨包架构、边界或依赖关系
- 内容属于长期共享知识，而非单包局部知识
- 当前文档一旦复制到多个包，容易出现重复维护

### 4.2 应放包内 `documents/` 的情况

命中以下多数情况时，可放包内：

- 内容只服务当前包
- 当前包维护者需要高频就近查看
- 文档与当前包实现细节强相关
- 文档主要用于当前包内部协作，而不是项目统一治理

## 5. 包根 `README.md` 规则

### 5.1 包根 `README.md` 的定位

包根 `README.md` 是包级公共说明页，不是包内正式文档目录页，也不是设计、决策、迁移文档的正文承载位置。

### 5.2 包根 `README.md` 的推荐内容

包根 `README.md` 建议参考 npm 包说明习惯，至少说明：

- 包名称
- 一句话描述
- 包的作用与职责边界
- 适用场景
- 安装或接入方式
- 基本使用示例
- 主要导出能力或公共 API
- 开发、调试或发布入口
- 重要限制或注意事项
- 深入文档入口，例如 `./documents/README.md`

### 5.3 包根 `README.md` 默认弱联动

新增、修改、删除包内 `documents/` 文档时，默认只要求更新包内最近一级 `documents/README.md`。

包根 `README.md` 不默认联动更新。

仅当文档变化影响以下内容时，才应同步更新包根 `README.md`：

- 包的公共职责
- 安装或使用方式
- 对外导出能力
- 开发或发布入口
- 使用者需要优先知道的重要限制

## 6. 包内 `documents/README.md` 规则

### 6.1 包内 `documents/README.md` 是正式目录页

包内 `documents/README.md` 负责：

- 解释当前包文档目录用途
- 挂载当前包内正式文档
- 作为当前包文档治理入口

### 6.2 最近一级挂载优先落在包内目录页

当新增或修改的是包内正式文档时，应优先检查最近一级 `documents/README.md` 是否需要更新挂载，而不是默认要求更新包根 `README.md`。

## 7. 根级文档复制到包内的规则

### 7.1 复制粒度

允许按需复制根级正式文档的单个文件到包内，用于就近查看或包内协作。

复制限制如下：

- 只允许复制单个文件
- 不允许复制根级目录结构
- 不允许为了一个文件顺带复制整组导航结构

### 7.2 默认身份

根级文档复制到包内后，默认仍是包内副本，不构成新的项目级权威主源。

根级 `documents/` 仍是原始主源，除非该文件被明确声明为包内派生版本。

### 7.3 允许复制的条件

只有命中以下条件之一时，才适合复制单文件到包内：

- 文档与当前包高度相关
- 当前包需要高频离线或就近查看该文档
- 当前包未来可能拆仓或独立发布
- 当前文档需要在根级原文基础上形成包内派生版本

### 7.4 默认禁止复制的内容

以下内容默认不应复制到包内：

- 全局规范
- 跨包决策
- 跨包架构文档
- 多包共用的统一指南
- 高频更新的项目级制度文档

### 7.5 复制文件的最小标识要求

若根级文档被复制到包内，建议在 frontmatter 或正文头部至少标注：

- 来源文档路径
- 当前身份：镜像副本或派生版本
- 同步方式
- 最近同步日期

示例：

```yaml
source: documents/standards/architecture/FE-ARCH-004-前端请求规范.md
source_type: root-document
local_role: mirror
sync_mode: manual
updated: 2026-04-01
```

若为派生版本，可补充：

```yaml
local_role: derived
derived_for: packages/request-client
```

## 8. 包内目录建议

### 8.1 最小结构

单个包默认最小结构为：

```text
<package>/
  README.md
  documents/
    README.md
```

### 8.2 按需扩展

当当前包正式文档逐渐增多时，可按需继续扩展：

```text
<package>/
  README.md
  documents/
    README.md
    guides/
    design/
    decisions/
```

是否继续扩展，应以当前包文档数量和职责复杂度为依据，不应机械复制根级结构。

## 9. 执行建议

- 先判断文档是项目级还是包级，再决定落点
- 包根 `README.md` 优先保持公共说明职责
- 包内正式文档优先挂到 `documents/README.md`
- 根级文档复制到包内前，先判断是否真的需要副本
- 若无法明确同步责任与身份边界，则不应复制
