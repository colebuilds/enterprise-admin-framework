# ENG-STD-002 提交规范

## 1. 文档定位

本文档用于定义提交信息的格式、语义边界、特殊场景写法与例外机制，重点回答：

- 提交信息应该写成什么样
- `type`、`scope`、`subject` 分别承担什么职责
- 什么样的提交信息应被阻断
- 如何处理 `merge`、`revert`、`breaking change` 等特殊场景

本文档不负责定义具体工具接入方式，工具接入与门禁执行细则由工程门禁规范负责。

## 2. 目标

本规范希望解决以下问题：

- 提交信息格式混乱
- 同类变更命名不一致
- 提交意图不清晰，后续追踪困难
- 版本治理、变更审计与回溯成本高

提交规范的目标不是让提交信息“好看”，而是让变更能够被稳定追踪、汇总和回放。

## 3. 基本格式

推荐使用：

```text
type(scope): subject
```

约束如下：

- `type` 用于表达变更类别
- `scope` 用于表达影响范围，可选但建议在复杂变更中填写
- `subject` 用于表达变更摘要，要求简洁、明确

示例：

- `docs(engineering): 增加提交规范`
- `feat(user-profile): 支持头像上传`
- `fix(order-flow): 修复支付回跳问题`

## 4. `type` 规则

### 4.1 推荐 `type`

以下 `type` 可作为基础集合：

- `feat`
- `fix`
- `docs`
- `refactor`
- `style`
- `test`
- `build`
- `ci`
- `chore`
- `perf`
- `revert`

### 4.2 `type` 的选择原则

- 新增能力用 `feat`
- 修复缺陷用 `fix`
- 规范或说明文档更新用 `docs`
- 不改变业务语义的重组用 `refactor`
- 仅样式或格式相关变更用 `style`
- 测试相关变更用 `test`
- 构建、工具链、依赖治理用 `build` 或 `chore`
- CI / 门禁相关变更用 `ci`
- 性能优化用 `perf`
- 回滚类提交用 `revert`

## 5. `scope` 规则

`scope` 用于表达影响范围，建议围绕可识别的业务域、模块、包或规范域命名。

### 5.1 适合写入 `scope` 的内容

- 业务模块
- package 名称
- 目录域
- 工具链域
- 规范域

### 5.2 `scope` 使用原则

- 单一变更尽量使用单一 `scope`
- 多范围影响较大时，可拆分提交而不是堆叠过多 `scope`
- 不建议使用过于抽象的 `scope`

## 6. `subject` 规则

### 6.1 基本要求

- 简短
- 明确
- 直接表达结果，不写过程
- 不堆叠多个动作

### 6.2 中文 / 英文约定

建议遵守以下约定：

- `type` 使用英文小写
- `scope` 使用英文小写或项目统一命名
- `subject` 默认使用中文，便于团队快速理解

如果项目已存在统一英文提交习惯，也应保持全仓一致，不应同一仓库内混用多套风格。

## 7. 特殊场景

### 7.1 `revert`

回滚提交应明确回滚目标，避免只写“revert”但不说明范围。

示例：

- `revert: 回滚门禁脚本误改动`

### 7.2 `breaking change`

当提交包含破坏性变更时，应明确标记：

- `feat!: 调整公共组件入参`
- 或在正文中注明 `BREAKING CHANGE: ...`

### 7.3 `merge`

合并提交应优先保持清晰可追踪，不应以“合并分支”掩盖真实变更内容。

### 7.4 `chore`

`chore` 适用于不直接改变产品行为、但影响工程维护的提交，例如：

- 规范更新
- 工具接入
- 脚本调整

## 8. 提交失败处理

当提交信息不符合规范时，应先修正提交信息，再继续提交流程。

不建议：

- 用临时备注代替正式提交信息
- 用含糊不清的泛化描述覆盖真实变更

## 9. 例外机制

在以下场景可申请例外：

- 历史仓库已有固定提交风格
- 外部协作要求必须使用特定前缀
- 紧急修复需要兼容现有流程

例外应满足：

1. 有明确原因
2. 有影响评估
3. 有临时范围
4. 有后续收敛计划

## 10. 一句话总结

提交规范的核心不是固定某一种“好看的格式”，而是让提交信息稳定表达：

- 这次改了什么
- 影响范围是什么
- 是否破坏兼容
- 后续如何追踪
