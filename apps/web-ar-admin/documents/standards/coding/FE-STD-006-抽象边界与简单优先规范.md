# 抽象边界与简单优先规范

## 1. 目标

本规范用于统一团队对“要不要抽一层”的判断口径。

它要解决的核心问题不是“文案怎么抽”，而是：

- 什么时候应该直接实现
- 什么时候值得引入抽象
- 什么样的抽象属于有收益
- 什么样的抽象只是增加中间层和维护成本

本规范希望减少以下问题：

- 为了整齐而整齐的无收益抽象
- 只为避免裸字符串而创建的搬运型常量
- 把显示文案当内部语义或控制流条件
- 为了复用把错误路径、边界行为、条件差异一并包掉
- 页面、组件、工具函数之间出现职责不清的中间层

---

## 2. 总纲领

### 2.1 简单优先，抽象有据

- 能直接实现并保持清晰可读时，优先直接实现
- 只有在收益明确时，才引入抽象
- 不反对抽象，反对无收益抽象、过早抽象、掩盖信息的抽象

一句话原则：

**反对无收益抽象，优先简单实现。**

### 2.2 判断抽象时，先问“为什么”

引入抽象前，应先能回答清楚：

- 这层抽象具体解决了什么问题
- 如果不抽，当前代码到底哪里会变差
- 这层抽象的收益是否大于新增的理解成本和跳转成本

如果这些问题回答不清，默认先不抽。

---

## 3. 适用范围与非目标

### 3.1 适用范围

本规范适用于前端业务代码中的：

- 页面与页面私有子组件
- composable
- 工具函数
- 配置对象
- 常量定义
- 与页面行为强相关的轻量文案收口

### 3.2 非目标

本规范不是：

- i18n 规范
- 命名规范
- 页面分层规范
- 文案撰写规范

文案常量、按钮标题、placeholder 只是本规范覆盖的一个高频场景，不是本规范的全部主题。

---

## 4. 什么时候值得抽象

满足以下任一条件时，可以考虑抽象：

### 4.1 存在稳定复用

- 被多个位置复用，且这种复用不是临时性的
- 未来大概率仍会保持同一语义和同一变更点

### 4.2 承载稳定业务语义

- 它不只是展示文字，而是领域规则的一部分
- 例如状态值映射、流程状态、操作语义、请求参数映射

### 4.3 形成结构化配置

- 多个字段共同组成一组稳定配置
- 例如 `key + label + tooltip + preset` 一体的卡片配置
- 例如列标题、说明文案、tooltip 描述组成的文件内列配置

### 4.4 带来明确的边界收益

- 让错误处理边界更清楚
- 让测试更容易覆盖
- 让页面职责更清晰
- 让重复分支明显减少

---

## 5. 什么情况默认不该抽

出现以下情况时，默认优先保留内联或局部实现：

### 5.1 只是搬运字符串

- 单次使用按钮文案常量
- 单次使用 placeholder 常量
- 单次使用弹窗标题常量
- 只为避免裸字符串而创建的 `COPY`、`TEXT`、`LABELS` 对象

这里的问题不在于名字本身是不是 `COPY`、`TEXT`、`LABELS`，而在于：

- 没有真实复用
- 没有稳定语义
- 没有形成结构化配置
- 只是把本地字符串搬到另一个位置

### 5.2 没有形成稳定语义

- 内容只是当前组件的一次性展示文本
- 没有跨处复用，也没有配置意义
- 抽出来后并不会提升理解效率

### 5.3 抽象掩盖了关键信息

- 把多个不完全相同的流程硬包成一个通用函数
- 把失败路径、差异行为、边界条件一起吞掉
- 让调用者看不出真实行为差异

### 5.4 把显示文案当内部语义

- 用中文 label 参与逻辑判断
- 用展示文案充当动作标识或内部枚举
- 让文案变更影响业务行为

---

## 6. 推荐与不推荐示例

### 6.1 推荐保留的抽象

```ts
const ORDER_FILTER_TABS = [
  { label: '全部', value: 'all' },
  { label: '待派发', value: 'PendingDispatch' },
  { label: '待审核', value: 'PendingAudit' },
];
```

原因：

- 状态值和展示标签形成稳定映射
- 同时服务展示和业务筛选语义

```ts
const orderStatusCards = [
  {
    cardKey: 'pendingDispatchOrderCount',
    label: '待派发订单单数',
    tooltip: '当前队列中等待派发给出款员工的订单总数',
    presetStates: ['PendingDispatch'],
  },
];
```

原因：

- 这是视图配置，不是单独搬运文案
- `cardKey / label / tooltip / presetStates` 是一组语义

```ts
const BATCH_REASSIGN_MISMATCH_MESSAGE =
  '批量转派仅支持相同提现大类且相同币种的订单';
```

原因：

- 这是业务规则提示
- 后续如果规则变化，存在稳定修改点

### 6.2 默认不推荐的抽象

```ts
const ACTION_COPY = {
  reassign: '转派',
};
```

如果只在当前按钮使用一次，更适合直接写：

```ts
{ default: () => '转派' }
```

```ts
const PAGE_COPY = {
  query: '查询',
  reset: '重置',
  merchantPlaceholder: '全部商户',
};
```

如果这些内容只服务单个组件，且尚未进入正式 i18n 改造范围，默认应留在组件本地，而不是先建一层临时文案系统。

如果确实形成了文件内配置或稳定语义分组，应使用更贴近职责的命名，例如：

- `...Config`
- `...Options`
- `...Messages`
- `...Labels`

## 7. 评审口径

代码评审时，优先检查以下问题：

1. 这层抽象解决的是实际问题，还是“看起来更整齐”？
2. 如果回退成内联实现，代码是否真的会变差？
3. 它承载的是稳定语义，还是只是展示文本？
4. 文案是否参与了业务逻辑判断？
5. 抽象是否隐藏了失败路径、边界行为或条件差异？
6. 这层抽象是否增加了不必要的跨文件跳转？

这些问题是评估信号，不是机械打分规则。判断时应结合模块职责、复用稳定性、测试边界和页面复杂度综合考虑。

并且要明确：

- 不是所有跨文件跳转都是问题
- 不是所有常量定义都属于过度抽象
- 关键在于这层抽象是否真的带来了职责、复用、配置或边界收益

---

## 8. 与页面分层和 i18n 的边界

### 8.1 与页面分层的边界

本规范不反对页面按职责拆分为：

- `index.vue`
- `useXxxPage.ts`
- `components/`
- 页面私有 helper / config

只要拆分带来了明确职责边界、测试边界或共享收益，就属于合理抽象。

本规范反对的是：

- 为单次文本或弱语义制造额外跳转
- 为了复用形式而牺牲行为清晰度
- 模块共享文件逐渐演变成大而全的中间层

### 8.2 与 i18n 的边界

本规范不替代正式 i18n 方案。

对于尚未进入 i18n 改造范围的历史页面：

- 优先先修复“文案驱动逻辑”和“文案充当内部语义”的问题
- 不要为了“以后可能要接 i18n”提前制造临时 copy 层
- 纯展示文本是否内联，按本规范判断

一旦页面正式进入 i18n 改造范围，应按 i18n 规范迁入正式翻译资源，而不是继续扩张本地文案常量层。

---

## 9. 执行要求

- 新增抽象时，应能说明抽象收益
- 重构时，优先删除搬运型常量，再考虑保留配置型抽象
- 如果后续团队对边界判断有调整，统一更新本规范，不在会话中长期依赖口头规则
- 本规范当前状态为 `trial`，后续可在更多页面和模块中验证后再升级
