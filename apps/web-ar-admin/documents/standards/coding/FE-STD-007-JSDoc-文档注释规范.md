# JSDoc 文档注释规范

## 1. 目标

统一项目中的 JSDoc 文档注释写法，确保共享能力、复杂 helper、组件边界和业务约定能够被调用者与维护者快速理解。

本规范要解决的核心问题不是“让每个函数都有注释”，而是：

- 让调用者在不深入阅读实现细节的前提下，快速理解某项能力的职责
- 让调用者能够判断某项能力的适用边界、非适用边界和关键约定
- 让相近 helper / 组件之间的区别和选择标准更清楚
- 避免只有阅读实现后才能判断其用法、限制和业务语义

一句话目标：

**JSDoc 主要服务调用者和维护者，用来说明能力的职责、边界、约定和选择标准，从而降低误用、误解和维护成本。**

---

## 2. 适用范围

本规范中的“JSDoc”包含两类写法：

- **声明附着型 JSDoc**：直接附着在函数、组件、类型、导出符号上的文档注释，供调用者阅读，也可被工具识别
- **JSDoc 风格职责块注释**：写在复杂组件、复杂配置对象或映射对象附近，用于说明职责、边界和业务语义的块注释

后文若未特别区分，默认同时覆盖这两类写法。

本规范适用于：

- 共享 helper
- composable
- 公共组件与业务组件
- 复杂配置对象、映射对象附近的说明块
- 对外暴露的函数、类型工厂、格式化能力、转换能力

本规范不要求：

- 给所有局部小函数补齐注释
- 给 TypeScript 已经表达清楚的类型信息再重复写一遍
- 把实现过程翻译成注释

---

## 3. 在 TypeScript 项目中的角色

### 3.1 JSDoc 是补充，不是替代

本项目已经使用 TypeScript，因此：

- 类型系统负责表达输入输出形状
- JSDoc 负责补充职责、语义、边界、使用方式和设计意图

不要把 JSDoc 写成另一份重复的类型定义。

### 3.2 优先解释“为什么”和“什么时候”

对调用者最重要的不是作者如何实现，而是：

- 这个能力解决什么问题
- 适用边界和非适用边界分别是什么
- 有什么关键约定或隐含前提
- 与相近能力相比应如何选择

---

## 4. 核心规则

### 4.1 注释写给使用者，不写给作者自我提醒

好的 JSDoc 至少应优先回答：

- 这个函数 / 组件承担什么职责
- 调用前需要了解哪些约定和边界

当能力存在误用风险、职责交叉、使用边界不直观或相近能力易混淆时，还应进一步回答：

- 谁更适合调用它，谁不应优先使用它
- 调用者应如何判断自己是否适合使用它

不应主要写：

- 我是怎么实现的
- 每一行代码做了什么
- 代码本身已经显而易见的内容

### 4.2 优先写职责和边界，不重复实现细节

应优先写：

- 职责
- 业务语义
- 边界
- 约束
- 例外

不应优先写：

- 重复函数名
- 重复参数类型
- 重复返回值类型
- 把代码逻辑逐句翻译成中文

### 4.3 函数或组件存在交叉能力时，必须写清选择标准

当两个或多个能力容易被混用时，JSDoc 不能只写“做什么”，还必须写：

- 与相近能力的区别
- 推荐使用场景
- 非推荐使用场景
- 选择标准

例如：

- `renderGrid`
- `renderDetailRows`
- `createDetailRowStyleSet`

这类能力都在处理“多行 label/value 对齐”，但适用场景不同。注释应明确告诉调用者：

- 何时用紧凑型网格
- 何时用固定标签宽度的详情块
- 何时只复用样式而不是直接复用渲染器

### 4.4 共享能力优先补，局部简单函数克制补

以下场景优先写 JSDoc：

- 跨文件复用的 helper
- 页面私有但职责复杂的核心函数
- 共享组件
- composable
- 格式化 / 解析 / 映射 / 转换类工具
- 隐含业务约定的请求参数构造器
- 容易被误用的能力

以下场景默认不必写：

- 单文件内一次性的小函数
- 名称和行为都非常直接的局部函数
- 纯粹的薄包装且没有额外约定

### 4.5 判断粒度以“符号职责”为准，不以文件路径机械判定

- 是否需要补注释，应优先按函数、组件、composable、类型工厂、配置对象等“具体符号”判断
- `shared.ts`、`utils/*.ts`、页面目录下 helper 这类文件路径只用于提示高概率场景，不表示文件内所有导出都必须机械补齐
- 同一文件内可以同时存在：
  - 需要写清边界的共享能力
  - 不需要额外注释的简单局部函数
- 当文件路径和实际复杂度冲突时，以职责复杂度、误用风险和复用稳定性为准

### 4.6 “简单逻辑可不写”的判断边界

仅当某个函数或局部逻辑同时满足以下条件时，才可默认不写 JSDoc 或职责块注释：

- 作用域局部，仅服务当前文件或当前组件
- 命名已经足够直接，阅读函数签名和实现即可理解用途
- 没有隐含业务协议、单位约定、时区口径、状态映射或参数构造规则
- 没有与相近能力发生职责交叉，也不存在明显误用风险
- 没有不直观的副作用、异常路径或调用前提
- 注释若写出来，大概率只是在重复代码字面含义

如果以上条件有任意一项不满足，就不应再按“简单逻辑”处理，而应进一步判断是否需要补注释。

可直接不写的典型例子：

- 单文件内的一次性事件转发
- 明显的 `loading` 开关切换
- 命名直白的局部 `computed`
- 不含业务约定的简单格式化拼接

不应因为逻辑短小就省略注释的典型例子：

- 请求参数构造器
- 时间 / 时区转换函数
- 状态值与显示值映射函数
- 共享 helper
- 与其他 helper 职责接近、容易混用的函数

## 5. 什么时候必须写

满足以下任一条件时，默认应补 JSDoc：

### 5.1 对外暴露的共享 helper

例如：

- `shared.ts`
- `utils/*.ts`
- 页面目录下供多个组件使用的 helper

这里的文件路径只是高频场景举例。真正的判断对象应是其中具体暴露给他人使用的 helper 或工厂函数，而不是文件内所有函数一律补齐。

### 5.2 composable 或页面行为工厂

例如：

- `useXxxPage.ts`
- `useTenantOptions.ts`
- `useReportPage.ts`

应说明：

- 返回什么能力
- 状态由谁维护
- 有哪些关键约定

### 5.3 共享组件或业务复杂组件

例如：

- 组件顶部的职责说明块
- 核心 props 的业务语义
- 弹窗类组件的使用边界

### 5.4 存在隐含协议或业务映射

例如：

- 时间戳协议
- 时区口径
- 状态值映射
- 请求参数构造规则

### 5.5 容易混淆的相近能力

例如：

- `renderGrid` vs `renderDetailRows`
- `formatXxx` vs `resolveXxx`
- `parseXxx` vs `normalizeXxx`

这类函数必须把边界写清。

## 6. 什么时候不该写

### 6.1 不要给显而易见的局部函数机械加注释

反例：

```ts
/** 设置 loading 为 true */
function startLoading() {
  loading.value = true;
}
```

这类注释没有新增信息。

### 6.1.1 不要为了“覆盖率”给简单逻辑补一行废话注释

反例：

```ts
/** 点击时关闭弹窗 */
function handleClose() {
  emit('close');
}
```

如果函数名、作用域和行为都已经非常直白，这类注释只会增加噪音，不会提升理解效率。

### 6.2 不要重复 TypeScript 已经表达的内容

反例：

```ts
/**
 * @param tenantId 商户 ID，类型 number
 * @returns 返回 string
 */
function getTenantName(tenantId: number): string {}
```

如果只有这点信息，注释价值很低。

### 6.3 不要把实现过程翻译成自然语言

反例：

```ts
/**
 * 循环数组，然后生成一个新的 div。
 */
```

应改成解释职责和使用场景，而不是重复实现步骤。

### 6.4 不要为注释而注释

当函数足够简单、作用域足够局部、命名足够直接时，不写比写废话注释更好。

## 7. 推荐标签与使用方式

本项目推荐使用一小组高价值标签，不追求堆满所有 JSDoc tag。

### 7.1 推荐标签

- 描述段落：默认优先，说明职责、边界、约定
- `@param`：参数有业务语义、隐含约定或易误用时使用
- `@returns`：返回值语义不直观时使用
- `@example`：存在典型用法或易错用法时使用
- `@throws`：确实会抛出异常且调用方需要感知时使用
- `@deprecated`：旧能力即将被替换时使用

### 7.2 谨慎使用

- 不要为了“看起来专业”堆很多 tag
- 没有新增信息时，不要机械写 `@param` / `@returns`
- 简单函数优先用自然语言描述块，不必强行标签化

## 8. 推荐写法

### 8.1 共享 helper

```ts
/**
 * 渲染紧凑型两列键值网格。
 *
 * 适用于统计型字段或紧凑型 label/value 信息块。
 * label 列宽由最长标签自动决定，适合内容结构稳定、无需固定标签宽度的场景。
 *
 * 不适用于详情型字段块；若需要固定标签宽度并让 value 列从统一起点开始，
 * 应优先使用 `renderDetailRows`。
 *
 * @param items 要渲染的键值对列表
 * @param options 可选对齐配置；默认保持 label 右对齐、value 左对齐
 */
```

### 8.2 存在交叉能力的 helper

```ts
/**
 * 渲染固定标签宽度的详情行。
 *
 * 适用于订单时间、时长总计、USDT 明细等详情型字段。
 * 与 `renderGrid` 相比，本函数更强调固定 label 宽度和统一 value 起点，
 * 不追求紧凑统计布局。
 *
 * 若行内结构不再是纯 label/value，而是还包含链接、按钮或复合内容，
 * 应改为复用 `createDetailRowStyleSet`，而不是继续硬套本函数。
 */
```

### 8.3 composable

```ts
/**
 * 管理报表页的查询状态、日期区间协议和弹窗联动行为。
 *
 * 本 composable 负责页面级状态和副作用，不负责具体表格渲染。
 * 返回值供 `index.vue` 直接绑定；子组件不应自行复制其中状态。
 */
```

### 8.4 组件顶部职责说明

```vue
/** * 提现订单第三方详情单元格。 * *
负责展示三方名称、三方单号、三方状态、消息和提交次数， *
并在具备订单上下文时提供“详情”弹窗入口。 * * 本组件负责 UI
呈现和弹窗打开逻辑，调用方负责从原始 row 中抽取字段。 */
```

## 9. 推荐结构

优先使用以下信息顺序：

1. 先写一句职责概述
2. 再写适用场景
3. 再写非适用场景或边界
4. 最后再补 `@param` / `@returns` / `@example`

也就是说，优先级应是：

**职责 > 场景 > 边界 > 参数细节**

而不是：

**参数列表 > 参数列表 > 参数列表**

---

## 10. 评审口径

代码评审时，可优先检查以下问题：

1. 这段 JSDoc 是否真的增加了使用信息？
2. 它解释的是职责和边界，还是只在重复代码？
3. 相近函数之间的区别是否写清楚了？
4. 调用者是否能仅通过注释判断“什么时候该用它”？
5. 注释是否和当前实现、当前规则一致？
6. 是否出现了为了完整而堆砌低价值标签的情况？
7. 是否把本来足够直白的简单逻辑也机械补成了噪音注释？

如果注释无法帮助调用者做选择，通常说明它还不够好。

---

## 11. 执行建议

- 先给共享 helper、composable、复杂组件补 JSDoc
- 先补最容易误用或最依赖业务约定的能力
- 对有交叉职责的能力，先写边界，再写参数
- 不要一次性给所有局部函数机械补注释
- 若某个模块准备系统性补注释，应先有统一规范，再批量落地

一句话原则：

**JSDoc 的价值不在“有注释”，而在“调用者能更快判断某项能力是否适用、应如何使用，以及哪些场景不应使用”。**
