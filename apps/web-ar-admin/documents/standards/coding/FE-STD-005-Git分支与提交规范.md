# Git 分支与提交规范

## 1. 目标

统一项目中的分支创建、分支命名、提交拆分、提交消息与合并前整理方式，避免“分支随意命名、提交过大、提交信息无意义、历史难以追溯、回滚成本过高”的问题。

本规范聚焦的是 Git 层面的行为约束，不替代协作治理规则。协作边界、执行授权、AI 参与方式仍由 `documents/standards/collaboration/` 与 `documents/standards/ai-collaboration/` 负责。

说明：本文档当前按本次任务边界落在 `documents/standards/coding/`；若后续 `collaboration/` 目录形成更完整的协作治理体系，可再评估其长期归属。

## 2. 适用范围

本规范适用于：

- 功能开发分支
- 修复分支
- 重构分支
- 迁移分支
- 文档分支
- AI 与人工共同参与的变更提交场景

本规范主要覆盖：

- 分支命名
- 提交粒度
- 提交消息
- commit 拆分
- 合并前整理
- 禁止事项

## 3. 核心规则

### 3.1 分支命名要能表达意图

分支名应至少表达以下信息中的一项或多项：

- 功能主题
- 修复主题
- 迁移主题
- 文档主题
- 阶段性目标

推荐采用简洁、稳定、可检索的命名方式，避免无意义编号、随手缩写、临时口语化命名。

示例：

- `feature/checkout-form`
- `fix/login-redirect`
- `docs/ai-collaboration`
- `refactor/utils-split`

### 3.2 一个逻辑动作对应一个合理提交

提交应尽量围绕一个可描述的逻辑动作展开，例如：

- 新增一个独立能力
- 修复一个明确问题
- 调整一个共享约定
- 补充一个测试闭环
- 更新一组关联文档

不建议把互不相关的改动混在一个提交里。

### 3.3 提交消息要能说明“为什么改”

提交消息应尽量说明：

- 改了什么
- 为什么改
- 是否影响行为

提交信息格式统一为：

- `type(scope): subject`

其中：

- `type` 使用英文规范值，如 `feat`、`fix`、`docs`
- `scope` 使用英文领域标识，如 `finance`、`system`、`api`
- `subject` 默认使用简体中文，简洁说明本次改动；除非有明确约定，否则不要使用全英文主题

执行说明：

- 当前仓库已通过 `husky + commitlint` 接入 `commit-msg` 校验
- 不符合 `type(scope): subject` 格式或 `subject` 整句全英文的提交，默认会在本地提交阶段被拒绝
- 如确有英文 subject 的特殊需求，应先有明确约定，再按约定放行

推荐风格：

- `feat(system): 添加角色权限管理`
- `fix(finance): 修复出款组别配置校验提示`
- `docs(coding): 补充分支与提交规范`

### 3.4 大改动必须拆 commit

当一次变更同时包含多个独立逻辑点时，应拆成多个 commit，避免一个 commit 里混入过多主题。

典型需要拆分的场景：

- 代码改动与文档改动混在一起
- 重构与行为修复混在一起
- 迁移与格式化混在一起
- 主功能与测试补充混在一起

### 3.5 合并前要先整理历史

在合并前，应优先保证：

- 提交主题清晰
- 提交顺序合理
- 不包含明显噪音提交
- 不把临时修复永久遗留在历史里

必要时应在本地先整理提交，再进入合并流程。

## 4. 推荐示例

### 4.1 推荐的分支拆法

如果任务是“补一组文档规范”，可以拆成：

- `docs/coding-naming`
- `docs/coding-ts`
- `docs/coding-vue-sfc`
- `docs/coding-composables`
- `docs/coding-git-workflow`

如果任务是“修一个页面问题”，可以拆成：

- `fix/page-render`
- `fix/page-test`
- `fix/page-doc`

### 4.2 推荐的提交拆法

一个合理的提交序列可以是：

1. 先补规范或测试骨架
2. 再实现核心逻辑
3. 再修引用或入口
4. 最后补文档与验证

这样每个提交都更容易被理解和回滚。

### 4.3 推荐的提交消息

- `docs(coding): 补充分支与提交规范`
- `fix(finance): 修复工具栏布局与刷新交互`
- `test(finance): 补充分组配置页面回归用例`

## 5. 禁止项与反例

以下行为原则上禁止：

- 使用 `final-final`、`tmp`、`test2` 这类无意义分支名
- 把多个完全不同主题硬塞进一个 commit
- 用“顺手修一下”掩盖无边界改动
- 把格式化、重构、修复、文档混成一个不可追溯提交
- 让提交消息只剩“update”“fix”“misc”

反例：

- 一个 commit 同时包含重构、修复、测试和文档
- 一个分支名完全看不出要做什么
- 一个 PR 里所有改动都堆成一坨，无法单独审查

## 6. 例外说明

以下情况可以适度放宽拆分要求：

- 非常小的单点修复
- 仅用于验证的临时分支
- 一次性迁移且明确会在后续统一整理的场景

但即使放宽，也不应牺牲可读性与可回滚性。

## 7. 执行建议

建议在实际工作中遵守以下顺序：

1. 先想清楚这次变更的主题
2. 再创建能表达主题的分支
3. 每完成一个独立逻辑点就考虑是否应提交
4. 提交前检查是否混入无关改动
5. 合并前统一整理历史

如果任务同时涉及 AI 协作与变更控制，应先遵守协作规范与执行控制，再执行本规范中的分支和提交要求。
