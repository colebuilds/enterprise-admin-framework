# UI结构与交互规范

## 1. 目标

统一项目中页面结构、信息层级、基础交互状态与常见 UI 结构的写法，保证不同页面与模块在布局、反馈和状态表达上保持一致。

本规范关注“UI 怎么组织、交互怎么表达”，不负责设计令牌、主题变量、色板数值等底层视觉系统定义。

## 2. 适用范围

本规范适用于 `apps/` 与 `packages/` 中所有直接影响页面呈现的前端代码，包括但不限于：

- 页面级布局
- 列表页、详情页、表单页、弹窗页、抽屉页
- 组件的结构拆分与组合方式
- 加载态、空态、错误态、禁用态、反馈态
- 需要面向用户输出的交互提示与状态文案

本规范不直接约束以下内容：

- 设计令牌与主题变量定义
- 颜色、字号、圆角等视觉 token 的数值系统
- 组件内部具体业务逻辑实现
- 状态管理与请求分层的技术实现细节

如果某项规则同时涉及结构、状态与实现细节，应优先按 `FE-ARCH-003`、`FE-ARCH-004`、`FE-ARCH-005` 与 `FE-STD-003` 的边界拆分理解。

## 3. 核心规则

### 3.1 页面信息层级必须清晰

页面应先表达“谁是主信息”，再表达“谁是辅助信息”，最后才是“补充信息”。

推荐顺序通常是：

1. 页面标题或核心标题
2. 关键摘要或主要操作
3. 主内容区
4. 次要说明或辅助操作
5. 附加信息、调试信息或低优先级提示

禁止把多个同等权重的信息并排堆叠，导致用户无法判断主要任务。

### 3.2 间距与分组必须一致

页面中的区块间距、卡片内边距、列表项间距、表单项间距应遵循统一节奏。

建议遵守以下原则：

- 同级元素使用同级间距
- 同类容器采用相近的内外边距节奏
- 相关内容使用视觉分组，减少用户扫描成本
- 不为局部“看起来更满”而破坏整体节奏

### 3.3 常见 UI 结构应稳定复用

以下结构应尽量形成一致模式：

- 按钮组
- 表单布局
- 列表结构
- 弹窗结构
- 抽屉结构
- 分页与筛选区

每一类结构都应保持相似的：

- 标题位置
- 操作区位置
- 内容区位置
- 反馈区位置

### 3.4 按钮区要有明确主次

同一操作区域中，按钮必须有主次顺序。

推荐规则：

- 只保留一个主按钮
- 次按钮数量尽量少
- 危险操作应与普通操作分组
- 仅在必要时才使用多主按钮并列

按钮文案应简洁明确，避免同一组里出现语义重复的按钮，例如“提交”“确认”“保存并提交”同时出现。

### 3.5 表单结构要服务于任务完成

表单页不应只追求字段完整，而应优先保证用户能顺利完成任务。

推荐规则：

- 先放最关键字段，再放次要字段
- 相关字段分组展示
- 长表单要有清晰分段
- 默认值、校验提示、提交结果反馈要统一
- 不必要的字段说明不要占据主视觉

### 3.6 列表结构要适合浏览和定位

列表页的结构应服务于“快速扫描、快速定位、快速操作”。

推荐规则：

- 先给筛选与搜索，再给列表，再给批量操作
- 列表项主信息要足够突出
- 次要信息可折叠或弱化
- 行内操作数量不宜过多
- 批量操作应独立于单项操作

### 3.6.1 列表中的多行详情字段应保持统一对齐

当列表单元格内需要展示多行详情信息时，布局规则应服务于“快速扫视同类字段”，不要让同一列里的标签和值起点漂移。

推荐规则：

- 统计型字段可使用两列网格布局，例如“标签 + 值”的紧凑统计块
- 详情型字段应使用固定标签宽度，标签左对齐，值列左对齐
- 同一列内的多行值应从同一纵向起点开始
- 共享详情组件必须遵守同一套对齐规则，不能每个页面各写一套
- 这条规则只约束“多行详情字段”的内部对齐，不等同于所有数值列的整体对齐规则

### 3.7 弹窗与抽屉必须只承载一个明确任务

弹窗和抽屉适合承载短任务，不适合塞入过多页面逻辑。

推荐规则：

- 一个弹窗只负责一类明确动作
- 标题要说明动作目标
- 内容长度过长时优先考虑页面化或抽屉化
- 提交、取消、关闭行为要明确

### 3.8 状态表达必须完整

以下状态都应考虑到：

- 加载态
- 空态
- 错误态
- 禁用态
- 成功反馈态
- 操作中反馈态

状态表达应满足：

- 用户能看懂当前发生了什么
- 用户知道下一步该做什么
- 用户不会误以为页面失效

### 3.9 状态文案要短且明确

状态文案应避免过长解释。

推荐原则：

- 加载态说明当前正在发生什么
- 空态说明为何为空以及下一步建议
- 错误态说明失败原因或兜底处理
- 禁用态说明何时可恢复

### 3.10 交互反馈必须及时

用户触发操作后，应尽快给出反馈。

推荐反馈包括：

- 按钮 loading
- 表单校验提示
- toast / message
- 页面局部刷新提示
- 操作结果回显

禁止用户连续触发但界面没有任何变化。

## 4. 推荐示例

### 4.1 页面结构示例

```vue
<template>
  <section class="page">
    <header class="page__header">
      <h1 class="page__title">订单管理</h1>
      <p class="page__desc">查看、筛选和处理订单数据</p>
      <div class="page__actions">
        <button class="btn btn-secondary">导出</button>
        <button class="btn btn-primary">新建订单</button>
      </div>
    </header>

    <section class="page__filters">
      <!-- 筛选区 -->
    </section>

    <section class="page__content">
      <!-- 列表区 -->
    </section>
  </section>
</template>
```

这个结构把主标题、辅助说明、主操作、筛选区和内容区分开，用户更容易理解页面任务。

### 4.2 状态示例

```vue
<template>
  <div>
    <div v-if="loading">加载中，请稍候…</div>
    <div v-else-if="error">加载失败，请重试</div>
    <div v-else-if="items.length === 0">暂无数据，试试调整筛选条件</div>
    <ul v-else>
      <li v-for="item in items" :key="item.id">{{ item.name }}</li>
    </ul>
  </div>
</template>
```

这个结构把加载、错误、空、正常四种状态拆开，避免用户看到空白页面却不知道发生了什么。

## 5. 禁止项 / 反例

- 把所有信息堆在一行里，主次不分
- 一个页面同时塞入多个并列主按钮，没有明确主次
- 表单字段顺序混乱，相关字段被拆散
- 列表项里塞入过多同级操作，导致用户无法判断主要动作
- 弹窗里塞入长篇说明、复杂表单和多个流程
- 页面只做成功态，不做加载态、空态和错误态
- 交互后没有任何反馈，用户无法判断操作是否成功
- 为了局部对齐，破坏全局间距节奏

## 6. 例外说明

- 复杂业务页面可以在局部打破简单结构，但必须有明确原因
- 调试页、运营后台、配置页等特殊页面可以增加更多信息密度，但仍应保留清晰层级
- 设计系统未覆盖的特殊组件可以先按页面实际需求处理，但后续应尽量纳入统一结构
- 若某页面必须承载长流程，优先考虑分步、分区或页面拆分，而不是直接加大单页复杂度

## 7. 执行建议

- 新页面先定义信息层级，再写具体组件
- 先确定主操作，再补次操作和补充信息
- 先考虑加载、空、错、禁用四类状态，再补正常态细节
- 复用同类页面时，优先复用结构，不要每页重新发明布局
- 当 UI 结构开始与组件实现、请求分层或状态归属纠缠时，分别回到 `FE-STD-003`、`FE-ARCH-003`、`FE-ARCH-004`、`FE-ARCH-005` 处理
