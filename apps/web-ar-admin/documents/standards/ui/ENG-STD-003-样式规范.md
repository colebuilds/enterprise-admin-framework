# ENG-STD-003 样式规范

## 1. 文档定位

本文档用于定义样式代码的书写原则、命名方式、边界限制与治理方式，重点回答：

- 样式应该怎么写
- 哪些写法应视为高风险
- 样式检查关注什么
- 历史样式问题如何渐进治理

本文档不负责定义业务交互设计，也不负责前端视觉方案本身。

## 2. 目标

本规范希望解决以下问题：

- 样式风格不统一
- 层级和选择器失控
- 变量、主题、token 使用混乱
- Vue SFC 中 style 写法缺少一致性
- 历史样式债长期堆积

样式治理的目标不是限制表现力，而是让样式代码可维护、可复用、可检查。

## 3. 适用范围

本规范适用于：

- CSS
- SCSS
- Less
- Vue SFC 中的 `style` 块

## 4. 样式书写原则

### 4.1 语义优先

样式类名和结构表达应优先服务语义，而不是纯视觉描述。

### 4.2 层级克制

选择器嵌套不应无限加深，避免形成难以维护的强耦合样式。

### 4.3 复用优先

相似样式应优先抽取变量、token、公共类或复用结构，而不是复制堆叠。

### 4.4 结果导向

样式书写应优先服务最终渲染结果、响应式稳定性与跨页面一致性。

## 5. 命名规则

建议遵守一致的命名体系，例如：

- 组件块名
- 状态名
- 变体名
- 布局辅助类

命名应满足：

- 可读
- 可定位
- 可扩展

## 6. 嵌套与选择器限制

### 6.1 嵌套原则

- 避免过深嵌套
- 避免依赖极长选择器链
- 避免因局部方便牺牲全局可维护性

### 6.2 高风险写法

以下写法应视为高风险：

- 过深嵌套
- 依赖父子结构强绑定
- 过度使用 `!important`
- 同类样式在多个位置重复定义

## 7. 变量、token 与主题

样式治理应优先建立统一的变量或 token 使用方式，用于表达：

- 颜色
- 间距
- 圆角
- 字体
- 层级

原则如下：

- 先使用统一 token，再考虑局部特例
- 不应把临时值直接扩散为全局规则
- 不应让页面自行发明一套互不兼容的颜色和间距体系

## 8. 禁止项

以下情况应优先阻断或收敛：

- 大量重复样式
- 选择器过度耦合
- 规则和变量散落各处
- 样式中夹杂明显不属于样式层的逻辑
- 通过视觉补丁掩盖结构问题

## 9. Stylelint 职责边界

Stylelint 负责对样式层进行静态检查，例如：

- 命名一致性
- 嵌套层级
- 重复规则
- 语法与格式问题

Stylelint 不应承担：

- 业务架构治理
- 提交规范治理
- 依赖边界治理

## 10. 历史问题渐进治理

对于已有样式债，不要求一次性清零。

建议做法：

- 先建立新代码基线
- 再逐步收敛旧代码
- 对高风险页面优先治理
- 对低风险遗留样式采用渐进重构

## 11. 例外机制

当某类样式限制暂时无法满足时，可申请例外，但必须满足：

1. 有明确原因
2. 有影响评估
3. 有临时范围
4. 有后续治理计划

例外不应长期保持不收敛。

## 12. 一句话总结

样式规范的核心不是追求“写法统一得很漂亮”，而是让样式层具备：

- 可维护性
- 可检查性
- 可复用性
- 可渐进治理
