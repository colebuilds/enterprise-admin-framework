# Mock规范

## 1. 目标

统一项目中的 Mock 使用方式、适用边界、目录约定与维护要求，避免“到处随手 mock、mock 掩盖真实问题、mock 与真实接口脱节、mock 无法复用”的低质量模拟行为，建立可控、可维护、可复用、尽量贴近真实链路的 Mock 体系。

本规范主要定义验证层与测试层的 Mock 策略；接口联调过程中的 Mock 回收与协作边界，继续参考 `FE-API-005-接口Mock与联调协作规范`。

## 2. 适用范围

本规范适用于：

- `apps/*` 下所有前端业务代码
- `packages/*` 下所有共享包、工具包、基础设施包
- 日常开发、联调、测试、回归、迁移中的模拟数据与模拟服务
- AI 与人工共同参与的 Mock 生成与维护场景

本规范主要覆盖：

- 开发态 Mock
- 测试态 Mock
- 浏览器链路 Mock
- 请求层 Mock
- fixtures / fake data 管理
- MSW / Orval 生成 Mock 的使用规则

## 3. 核心原则

### 3.1 Mock 是手段，不是默认答案

Mock 的目标是：

- 提高开发效率
- 支撑测试复现
- 隔离不稳定依赖
- 构建可重复验证场景

Mock 不应成为掩盖真实问题的默认做法。

### 3.2 优先贴近真实链路

在前端开发与页面测试中，应优先选择更贴近真实请求链路的 Mock 方式，例如：

- MSW
- Orval 生成的 MSW handlers

避免大量直接 stub 掉请求函数，导致浏览器层请求行为、调试体验与真实链路脱节。

### 3.3 Mock 不能篡改系统边界

Mock 应模拟“外部输入”，而不应随意绕过：

- 请求层
- 参数构造层
- header 注入层
- 错误处理层
- DTO 解析层
- 状态流转层

### 3.4 Mock 要服务于验证，而不是制造假安全感

Mock 过度、Mock 错位、Mock 脱离真实接口，都会导致：

- 测试通过但真实系统仍有问题
- 页面联调与实际运行不一致
- Bug 被临时掩盖
- 错误边界长期无人发现

### 3.5 Mock 必须可维护

Mock 数据、handler、fixtures 必须：

- 有明确来源
- 命名清晰
- 可复用
- 可定位
- 随真实协议演进及时更新

---

## 4. Mock 分层模型

## 4.1 浏览器链路 Mock

目标：

- 模拟前端与服务端之间的真实请求交互
- 保持浏览器层调试体验接近真实请求

推荐工具：

- MSW
- Orval 生成的 MSW handlers

适用于：

- 页面开发
- 联调前置
- 页面级测试
- E2E / smoke 中对服务端的可控模拟

特点：

- 更接近真实前端请求行为
- 可在浏览器调试工具中观察请求过程
- 更适合与 Chrome DevTools / DevTools MCP 配合

## 4.2 模块测试 Mock

目标：

- 隔离模块依赖
- 验证模块行为
- 构建稳定、可重复的测试场景

推荐工具：

- Vitest mock 能力
- 局部 stub / spy
- 测试专用 fixtures

适用于：

- utils
- composable
- service
- store
- request 处理链中的局部依赖隔离

特点：

- 运行快
- 更聚焦
- 适合单元与集成测试
- 不能替代浏览器链路模拟

## 4.3 数据层 Mock

目标：

- 提供稳定的输入数据
- 覆盖正常、空态、异常、边界条件

形式：

- fixtures
- fake data builder
- sample payload

适用于：

- 单元测试
- 集成测试
- 页面联调
- Story / Demo 场景（如后续有）

---

## 5. Mock 工具职责映射

### 5.1 MSW

适合：

- 浏览器链路 Mock
- 页面联调 Mock
- 请求行为验证
- 页面级测试中的接口模拟

定位：

- 前端主 Mock 基础设施

### 5.2 Orval 生成的 MSW handlers

适合：

- 基于 OpenAPI 文档生成 Mock handlers
- 保持 Mock 与接口协议更一致
- 降低手写 handler 成本

定位：

- OpenAPI 驱动的 Mock 生成能力

### 5.3 Vitest mock / spy / stub

适合：

- 单元测试与模块级集成测试
- 隔离局部依赖
- 验证调用次数、入参、返回值

定位：

- 逻辑测试中的局部模拟工具

### 5.4 fixtures / fake data builder

适合：

- 统一管理测试输入数据
- 构建边界场景
- 复用测试样本

定位：

- 测试数据资产层

---

## 6. 何时应该使用 Mock

以下场景适合使用 Mock：

- 后端接口尚未就绪
- 外部服务不稳定
- 希望构造稳定可复现测试场景
- 需要覆盖边界数据与异常数据
- 需要脱离真实环境快速开发或回归
- 需要在页面层验证请求链路但不依赖真实后端

---

## 7. 何时不应优先使用 Mock

以下场景不应默认首先使用 Mock：

- 需要确认真实后端行为
- 需要确认真实 header / cookie / token 注入效果
- 需要验证真实环境配置
- 需要定位真实联调问题
- 需要验证跨系统集成问题

此类场景应优先：

- 真实后端联调
- 真实环境验证
- 最小 Mock 辅助，而不是完全替代真实链路

---

## 8. Mock 层级选择原则

### 8.1 优先在最贴近验证目标的一层 Mock

原则：

**只 mock 必须 mock 的部分。**

例如：

- 验证 util 逻辑，不需要 MSW
- 验证页面请求链路，不应直接 mock 掉 request-client 最终调用
- 验证 service 行为，可以局部 stub 下游依赖
- 验证浏览器请求行为，应优先使用 MSW

### 8.2 不以低层 stub 取代高层链路验证

直接 stub 请求函数虽然方便，但不能替代：

- 页面层请求行为验证
- 浏览器调试体验
- 参数/请求头/状态流转链路验证

### 8.3 不以高层 Mock 替代低层逻辑测试

页面层 Mock 不应替代单元测试和集成测试。

---

## 9. Mock 数据与目录组织建议

## 9.1 fixtures 应集中管理

建议：

- 通用 fixtures 统一放置
- 不同模块按领域拆分
- 命名体现业务含义，不使用无意义编号

示例：

- `fixtures/payment-methods.ts`
- `fixtures/user-profile.ts`
- `fixtures/game-list.ts`

## 9.2 页面级 handlers 应按模块组织

建议：

- 按业务域拆分
- 避免单个大而乱的 handlers 文件
- 让人和 AI 能快速定位具体 mock 来源

## 9.3 不允许到处内联匿名 mock 数据

禁止大量在测试文件中直接塞入不可复用的长 JSON 数据，而不做抽取和命名。

---

## 10. Mock 质量要求

Mock 应满足以下要求：

- 与真实接口结构尽量一致
- 能区分成功、空态、异常、边界场景
- 不遗漏关键字段
- 不随意省略会影响业务判断的字段
- 不伪造不合理数据导致错误结论
- 修改真实协议后，相关 Mock 应同步更新

---

## 11. 与真实接口一致性要求

### 11.1 接口协议变更后必须评估 Mock

当 OpenAPI、接口字段、错误结构、状态码规则发生变化时，应同步检查：

- handlers
- fixtures
- fake data builder
- 测试断言

### 11.2 Mock 不能长期脱离真实协议演进

若发现：

- Mock 通过但真实联调失败
- Mock 中字段与真实接口长期不一致
- 页面逻辑只在 Mock 场景成立

应优先修复 Mock 体系，而不是继续堆补丁。

---

## 12. 开发态 Mock 规范

### 12.1 开发态主 Mock 方案

优先推荐：

- MSW
- Orval + MSW handlers

### 12.2 开发态 Mock 应可开关

Mock 能力应明确区分：

- 开启 Mock
- 关闭 Mock
- 部分域启用 Mock

避免开发者不清楚当前正在使用真实后端还是模拟后端。

### 12.3 开发态 Mock 应支持调试

应尽量保留：

- 可观察请求
- 可观察响应
- 可观察错误分支
- 可切换不同 mock 场景

---

## 13. 测试态 Mock 规范

### 13.1 单元测试

允许：

- stub
- spy
- 局部函数 mock
- 模块 mock

但应避免：

- 为了测试方便彻底改写真实行为边界
- 对核心断言点进行自我证明式 mock

### 13.2 集成测试

允许：

- Mock 外部依赖
- 保留被测模块内部真实协作
- 使用 fixtures 构造真实输入

### 13.3 页面级测试

优先：

- MSW
- 浏览器链路级 mock

不推荐：

- 在页面测试中大面积直接绕过请求层

---

## 14. AI 协作 Mock 要求

### 14.1 AI 生成 Mock 时应说明 Mock 层级

AI 输出中应说明：

- 当前 mock 的目标是什么
- mock 发生在哪一层
- 为什么选择这一层
- 是否会偏离真实链路

### 14.2 AI 不应过度 mock

禁止 AI 为了“让测试通过”而：

- mock 掉所有下游依赖
- 绕过真实请求构造
- 绕过真实状态流转
- 绕过真实错误处理路径

### 14.3 AI 生成的 fixtures 应可读、可复用

要求：

- 命名有业务含义
- 结构清晰
- 避免大段匿名硬编码 JSON
- 能区分场景用途

---

## 15. 禁止事项

以下行为原则上禁止：

- 用 mock 长期替代真实联调
- mock 掩盖真实请求链问题
- mock 中遗漏关键字段导致错误判断
- 测试里大量复制匿名 JSON
- 页面级测试直接绕过请求层
- 因为接口未好就把 mock 逻辑混进生产实现
- AI 为了让测试通过而过度 mock

---

## 16. 与其他规范的关系

本规范与以下规范协同工作：

- `FE-QUALITY-003 测试规范`：定义测试分层与测试职责
- `FE-QUALITY-001 调试规范`：定义问题定位方法
- `FE-QUALITY-002 Bug 修复规范`：定义问题修复闭环
- `FE-ARCH-004 前端请求规范`：定义请求链路边界
- OpenAPI 自动化规范：定义 Orval / 生成链路规则

关系可概括为：

- 请求规范：规定真实请求怎么走
- 本规范：规定哪些地方可以、应该如何模拟
- 测试规范：规定 mock 服务于哪类测试
- 调试规范：规定如何借助 mock 排查问题

---

## 17. 最低执行要求

正式引入 Mock 时，至少应满足以下要求：

- 明确 mock 的目标
- 明确 mock 所在层级
- 不绕过关键系统边界
- 数据可复用、可维护
- 能区分正常 / 空态 / 异常场景
- 与真实协议保持基本一致

对于页面开发与浏览器测试，额外建议：

- 优先使用 MSW 或等价的浏览器链路级 mock

---

## 18. 一句话原则

**只 mock 必须 mock 的部分；优先选择贴近真实链路的 Mock 方式；Mock 应服务于验证，而不是掩盖真实问题。**
