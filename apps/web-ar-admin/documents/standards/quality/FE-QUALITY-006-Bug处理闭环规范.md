# Bug处理闭环规范

## 1. 目标

统一项目中的 Bug 处理闭环，避免“直接改代码、不留清单、不做验证、无法对账、无法回填”的低质量处理方式，建立可清单化、可验证、可交接、可回填的缺陷处理流程。

本文档负责定义：

- Bug 如何进入处理链路
- Bug 如何形成处理前、处理中、处理后的可追溯清单
- Bug 如何在进入实施前形成可审核的修复方案与实施计划
- Bug 在什么条件下可以进入修复、验证、提交与交付
- 哪些条件不满足时必须挂起或阻塞

本文档不替代具体修复动作规范，也不替代 AI 协作边界规范。Bug 修复动作参考 `FE-QUALITY-002-Bug 修复规范`，回归验证层级参考 `FE-QUALITY-005-回归验证规范`，AI 与人工分工参考 `FE-AI-008-AI-Bug处理协作规范`。

## 2. 适用范围

本规范适用于：

- 禅道、Issue、测试清单等来源的 Bug 修复任务
- 日常缺陷修复、回归问题、批量 Bug 处理场景
- AI 与人工共同参与的缺陷处理场景

本规范不完全适用于：

- 无明确 `bug_id` 的临时探索性问题
- 纯需求开发
- 无法稳定复现的紧急止血场景
- 外部系统偶发异常导致的非代码逻辑问题

对于不完全适用场景，应先止血、隔离影响，再补充复现、验证与回填链路。

## 3. 核心原则

### 3.1 全流程以 bug_id 为唯一主键

所有处理记录、清单、交付物都必须以 `bug_id` 作为唯一主键，并尽量保持与来源列表一致的顺序。

### 3.2 先清单化，再进入修复

任何一轮 Bug 处理都应先形成处理前清单，再进入分类、批次、修复与验证。禁止直接从列表口头描述进入代码修改。

### 3.3 信息不足判断必须晚于详情深化分析

若某条 Bug 在列表标题和简述层面信息不足，不应立即判定为“待补充”。应先完成一轮详情深化分析，再决定是否挂起。

详情深化分析至少包括：

- bug 详情页正文
- 截图、附件、录屏等证据
- 历史记录中的补充线索
- 当前项目页面、字段、接口、代码结构的实际映射分析
- 当当前项目证据仍不足时，进一步查看原型项目 `/Users/cola/Documents/gs/ar-saas` 的对应功能实现；查看前应先同步最新代码，优先使用 `git pull --ff-only`

只有在完成上述分析后，仍无法明确复现入口、预期行为、影响范围或实现映射时，才应将其标记为“待补充”。

一旦确认某条 Bug 仍信息不足，应将其单独挂起并进入待测试补充清单；其余信息充分的 Bug 继续推进。

### 3.4 修复完成不等于闭环完成

代码已修改、AI 本地验证通过，均不等于闭环完成。闭环至少还应包含人工本地验证、交付清单和后续 SIT / 禅道回填链路。

### 3.5 产品口径待确认不等于信息不足

若 Bug 的改动意见已经明确，但当前实现本身也合理，且争议点落在“产品最终想要哪种实现口径”而不是“信息够不够”，则不应标记为信息不足。

这类问题应标记为 `待产品确认`，并在清单中明确：

- 当前实现为什么合理
- 测试提出的改动口径是什么
- 预计修改会改变哪类实现策略
- 需要产品确认的决策点是什么

在产品确认前，不应把此类问题当作普通可直接实施的 Bug 进入正式修复。

### 3.6 处理结果必须可对账

处理前、处理中、处理后所有输出都必须能与来源列表逐条对账。未处理、阻塞、待补充的 Bug 也必须保留在清单中，不得消失。

## 4. 标准闭环总览

一轮标准 Bug 处理闭环至少包括以下阶段：

1. 获取 Bug 列表
2. 生成处理前清单
3. 详情深化分析
4. 审查信息完整度
5. 分类与批次决策
6. 需求理解与拟修复方案确认
7. 生成 Bug 修复方案卡
8. 进入 PIR 规划阶段
9. 进入 PIR 实施阶段
10. 进入 PIR 评审阶段
11. 人工本地验证
12. 处理后交付清单
13. SIT 验证与禅道回填

## 5. 输入基线与处理前清单

每一轮 Bug 处理都必须先生成处理前清单，作为本轮输入基线。

处理前清单至少应包含：

- `bug_id`
- `bug_title`
- 来源状态
- 模块/页面
- bug 类型
- 信息完整度
- 本轮是否处理
- 建议批次
- 备注

规则要求：

- 未进入处理前清单的 Bug，不得默认纳入本轮
- 处理前清单应尽量保持与禅道列表一致的顺序
- 处理前清单生成后，方可进入正式处理阶段

## 6. 详情深化分析、信息完整度与挂起规则

每条 Bug 在进入修复前，不应只凭标题和列表摘要直接判断信息完整度。

应先完成一轮详情深化分析。

### 6.1 详情深化分析

详情深化分析至少包括：

- 阅读 bug 详情页正文、期望描述和补充说明
- 检查截图、附件、录屏中能直接提供的字段、文案、交互或数值口径
- 检查历史记录里是否已有补充说明
- 结合当前项目实际页面、字段命名、接口注释、代码结构做一轮映射分析
- 当当前项目证据仍不足时，继续查看原型项目 `/Users/cola/Documents/gs/ar-saas` 的对应功能实现；查看前应先同步最新代码，优先使用 `git pull --ff-only`

目标不是立即修复，而是先判断：

- 当前项目中是否存在可对应的页面或字段
- 截图或附件是否已经足够说明预期行为
- 标题与详情是否能映射到实际改动范围
- 所谓“缺信息”是否只是列表摘要太短，而不是问题真的不可判断

### 6.2 信息完整度判断

在完成详情深化分析后，再判断信息完整度。

信息完整度分为：

- `完整`
- `基本完整`
- `不足`

处理规则如下：

- `完整`：可直接进入后续处理
- `基本完整`：允许先尝试复现和处理
- `不足`：进入待测试补充清单，本轮先不处理

以下情况在完成详情深化分析后仍然存在时，通常应判定为信息不足：

- 缺少复现步骤
- 缺少页面入口
- 缺少预期行为说明
- 缺少环境说明
- 缺少账号、权限或角色条件
- 缺少截图、录屏或明确表现描述
- 描述存在歧义，无法判断具体问题

### 6.3 挂起规则

信息不足的 Bug 不应阻塞整轮处理，但必须在本轮结束时统一输出待补充项。

若截图、附件和项目现状已足以支撑映射分析，即使标题较短或字段名称与当前项目不完全一致，也不应过早进入待测试补充清单。

### 6.4 产品口径待确认规则

若完成详情深化分析后，已经能够明确：

- 当前实现的行为和来源
- 测试提出的目标改动
- 两种实现都具有合理性

但无法仅凭测试单、截图和现有代码判断“产品最终想要哪种口径”，则应将该问题标记为 `待产品确认`，而不是 `待测试补充`。

这类问题通常具有以下特征：

- 当前实现并非明显错误，而是另一种合理实现
- 修改后会改变数据来源策略、固定集合范围、业务口径或交互策略
- 测试给出了改动建议，但该建议不足以单独替代产品决策

对于 `待产品确认` 的问题，AI 应输出：

- 当前实现合理性说明
- 测试建议口径
- 拟修改实现策略
- 待确认问题点

在确认前，该问题不进入正式修复。

## 7. 分类与批次决策

进入修复前，应先对可处理 Bug 做分类与批次决策。

分类建议至少考虑以下维度：

- 模块
- 页面
- Bug 类型
- 影响范围
- 验证链路
- 是否可并单处理

批次规则如下：

- 同模块、同链路、同验证上下文的 Bug 可优先纳入同一批次
- 跨模块、风险差异大、验证路径不同的 Bug 应考虑拆批次
- 批次决策完成后，方可进入分支准备与正式修复

## 8. 分支准备要求

分支准备应在清单审查和批次决策之后进行。

规则要求：

- 新建 Bug 修复分支必须基于最新 `main`
- 不得默认从旧工作分支直接派生新的 Bug 分支
- 同一批次 Bug 默认在同一修复分支内处理
- 分支命名、同步主线与提交方式按项目协作规范执行

若当前工作区不干净、分支用途不明确或共享分支风险未确认，则不得直接进入分支准备。

## 9. 需求理解与拟修复方案确认

在进入正式修复前，应先对每条可处理 Bug 形成两项内容：

- 需求理解结论
- 拟修复方案

其中：

- 需求理解结论用于说明“这条 Bug 实际要求改什么、不要求改什么”
- 拟修复方案用于说明“准备改哪些现有字段、交互、校验或渲染点，以及明确不做哪些扩改”

若存在以下任一情况，应优先进入人工审查，而不是直接开修：

- 测试标题与详情描述可能不一致
- 截图和当前实现映射存在歧义
- 业务口径可能被误读
- 预计修复方案会改变页面结构、字段范围或交互方式
- 当前实现本身也合理，改动实质上是在两种产品口径之间做选择

## 10. Bug 修复方案卡与实施计划

在进入正式实施前，应先把分析结果沉淀为可审核的 `Bug 修复方案卡`，再进入实施计划阶段。

`Bug 修复方案卡` 至少应包含：

- `bug_id`
- 证据来源
- 当前实现说明
- 测试 / 设计 / 原型口径
- 需求理解结论
- 具体修复方案
- 代码落点
- 明确不做什么
- 风险点
- 待确认项

若 Bug 已具备明确边界和可实施方案，则应基于方案卡进入 PIR 规划阶段，产出实施计划；未经方案卡审查或未经实施计划批准，不应直接进入代码修改。

## 11. 修复进入条件

只有满足以下条件，Bug 才允许进入正式修复：

- 已纳入处理前清单
- 已完成详情深化分析
- 已完成信息完整度判断
- 已完成批次归属
- 已形成需求理解结论与拟修复方案
- 已形成 Bug 修复方案卡
- 已完成实施计划
- 已明确当前行为与预期行为差异

未满足以上条件时，不得直接开始修改代码。

## 12. PIR 阶段路由要求

在本规范中，各阶段默认按以下路由执行：

- Bug intake、证据归集、清单化、方案卡产出：由 `bug-sop` 负责
- 方案审查通过后的实施计划：进入 `pir-planner`
- 已批准计划的实际修复：进入 `pir-implementer`
- 实现完成后的独立评审：进入 `pir-reviewer`

若某条 Bug 尚未形成可审查方案卡，或尚未完成实施计划，则不应提前进入 `pir-implementer`。

## 13. 验证闭环要求

Bug 修复后的验证闭环至少分为三层：

### 10.1 AI 本地验证

至少包括：

- 原问题路径验证
- 同页或同模块最小必要冒烟验证
- 必要检查命令执行结果

### 10.2 人工本地验证

由人实际操作本地页面，确认业务表现符合预期，并决定是否允许进入正式提交阶段。

### 10.3 SIT 验证

由人在 SIT 环境完成最终验证，并根据验证结果决定禅道如何回填。

规则要求：

- AI 本地验证通过，不等于允许直接提交
- 至少经过人工本地验证通过后，才允许进入正式提交
- 若 SIT 验证失败，应回到处理链路重新分析，不得直接关闭问题

## 14. 提交与交付要求

默认情况下，同一批次、同一模块、同一分支内的 Bug 应统一提交。

进入正式提交前，至少应具备：

- 处理前清单
- 处理中状态记录
- 处理后交付清单
- 人工本地验证结论

如果清单与来源列表无法逐条对账，则不得进入正式交付。

## 15. 禅道回填前交付物

在进入 SIT 与禅道回填前，必须输出处理后交付清单。

处理后交付清单至少应包含：

- `bug_id`
- `bug_title`
- 是否已修复代码
- 修复内容摘要
- AI 本地验证结果
- 人工本地验证结果
- 是否需要 SIT 验证
- SIT 关注点
- 建议禅道动作
- 备注

## 16. 阻塞与异常处理

以下情况应明确标记为阻塞或挂起状态：

- 信息不足
- 待产品确认
- 无法复现
- 根因不明确
- 业务预期不一致
- 分支准备失败
- AI 本地验证失败
- 人工本地验证失败
- SIT 验证失败

所有阻塞项都必须保留在清单中，并明确当前状态和下一步责任方。

## 17. 最小闭环完成标准

一条 Bug 只有同时满足以下条件，才可视为完成闭环：

- 已纳入清单并可对账
- 已完成代码修复
- 已完成 AI 本地验证
- 已完成人工本地验证
- 已生成处理后交付清单
- 已进入 SIT / 禅道回填链路

对外部系统视角的最终闭环，以人工完成禅道回填为准。

## 18. 项目适配项

当前项目默认适配值如下：

- Bug 来源：禅道 `我的 Bug`
- 基础分支：`main`
- Bug 修复分支：`fix/<task_slug>`
- 主线同步策略：默认 `merge`
- 禅道状态回填：人工执行
- SIT 验证：人工执行
- 默认提交策略：按批次统一 commit
