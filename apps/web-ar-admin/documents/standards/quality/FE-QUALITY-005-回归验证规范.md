# 回归验证规范

## 1. 目标

统一项目中的回归验证方式、验证范围、触发条件与执行要求，避免“改完当前问题就结束、不验证相邻影响面、不沉淀关键回归链路”的低质量交付行为，建立可重复、可分层、可控范围的回归验证体系。

本规范负责“回归范围、回归层级与验证要求”；Bug 修复动作本身的处理流程继续参考 `FE-QUALITY-002-Bug 修复规范`。

## 2. 适用范围

本规范适用于：

- `apps/*` 下所有前端业务代码
- `packages/*` 下所有共享包、工具包、基础设施包
- Bug 修复后的验证
- 新功能完成后的验证
- 重构后的验证
- 迁移后的行为一致性验证
- 发布前 smoke 检查
- AI 与人工共同参与的改动验证场景

本规范主要覆盖：

- 单模块回归
- 关联模块回归
- 页面链路回归
- 关键业务流程回归
- 发布前验证
- 回归范围控制

## 3. 核心原则

### 3.1 修复当前问题，不等于完成交付

任何改动都不应只验证“当前问题没了”，还必须评估：

- 相邻功能是否受影响
- 相同模式问题是否仍存在
- 关键链路是否被破坏
- 共享能力是否引入回归

### 3.2 回归范围由改动影响面决定

回归验证应根据以下因素确定范围：

- 改动位置
- 改动层级
- 影响模块
- 是否共享逻辑
- 是否关键链路
- 是否历史高风险区域

### 3.3 优先最小必要回归，但不遗漏关键面

不做无边界大范围乱测，也不允许只测当前页面就结束。

原则：

**以最小必要范围覆盖主要风险。**

### 3.4 自动化优先，手工补充

能通过自动化稳定验证的，应优先自动化；无法自动化或短期不具备条件的，再使用手工回归补充。

### 3.5 回归验证结果必须可说明

回归完成后，应至少能说明：

- 验证了哪些范围
- 使用了哪些方式
- 哪些风险已覆盖
- 哪些风险暂未覆盖

---

## 4. 回归验证分层模型

## 4.1 单元 / 模块回归

目标：

- 验证本模块逻辑是否仍然正确

适用于：

- utils
- mapper
- service
- composable
- store
- request 处理链
- 基础设施方法

推荐工具：

- Vitest

## 4.2 模块协作回归

目标：

- 验证相关模块之间的交互是否仍然成立

适用于：

- composable ↔ store
- service ↔ request
- DTO ↔ mapper ↔ domain
- package 之间协作

推荐工具：

- Vitest
- 必要时使用 mock / fixtures

## 4.3 页面级回归

目标：

- 验证页面层行为与交互是否仍然成立

适用于：

- 页面展示
- 表单联动
- 条件展示
- route 切换
- 弹窗流程
- 页面级异常处理

推荐工具：

- Playwright
- 手工验证
- Chrome DevTools / Vue Devtools 辅助确认

## 4.4 关键链路回归

目标：

- 验证关键业务路径没有被破坏

适用于：

- 登录
- 支付
- 鉴权
- 页面主路径
- 配置注入
- 宿主链
- 租户初始化
- 路由守卫
- 时间与配置相关关键逻辑

推荐工具：

- Playwright
- smoke 测试
- 发布前检查清单

---

## 5. 回归范围判定原则

## 5.1 按改动层级判定

### 改动在 util / mapper / 时间处理层

至少回归：

- 该逻辑的单元测试
- 使用该逻辑的关键模块测试
- 高风险调用点

### 改动在 composable / service / store 层

至少回归：

- 该模块自身测试
- 直接依赖它的页面或模块
- 相关状态流与交互路径

### 改动在 request / API / 基础设施层

至少回归：

- 相关请求链测试
- 主要消费模块
- 关键接口交互页面
- 关键业务流程

### 改动在页面层

至少回归：

- 当前页面核心场景
- 相邻状态场景
- 与该页面强耦合的跳转和交互流程

## 5.2 按共享程度判定

若改动发生在共享包、基础设施包、通用 composable、通用 request 层，则必须扩大回归范围，不能只测一个页面。

## 5.3 按风险等级判定

以下场景应扩大回归：

- 曾多次出问题
- 历史上回归频繁
- 直接影响支付 / 鉴权 / 路由 / 配置
- 多租户差异逻辑
- 时间 / 时区 / 金额处理
- AI 大范围修改

---

## 6. 不同改动类型的回归要求

## 6.1 Bug 修复

至少要求：

- 原问题验证通过
- 同类路径验证通过
- 相邻模块未受影响
- 新增回归测试通过

## 6.2 新功能开发

至少要求：

- 新功能主路径通过
- 与已有功能交界处验证通过
- 关键边界条件验证通过

## 6.3 重构

至少要求：

- 原有核心行为不变
- 保护性测试通过
- 关键页面或主链路无回归

## 6.4 迁移

至少要求：

- 新旧行为一致性验证
- 高风险页面 smoke
- 核心路径对齐验证
- 必要时保留 golden case 或对比样例

若旧测试无法直接支撑本次大改动，不得直接放弃旧行为基线。至少应选择一种方式承接旧行为保护：

- 调整后仍可复用的保护性测试
- golden case
- 对比样例
- 关键路径 smoke
- 主要消费方验证
- 新旧行为一致性验证

## 6.5 基础设施改动

至少要求：

- 相关包测试通过
- 主要消费方验证通过
- 关键业务链路 smoke 通过

---

## 7. 回归方式选择原则

### 7.1 能自动化则优先自动化

优先顺序建议：

1. 现有自动化测试
2. 新增最小必要自动化测试
3. 手工验证补充

### 7.2 手工回归必须有清单

手工验证不应模糊进行。至少应记录：

- 验证页面 / 模块
- 验证步骤
- 预期结果
- 实际结果

### 7.3 页面级问题不等于只能手工点

若问题具有稳定链路、重复价值高，应尽量沉淀为 Playwright 用例，而不是长期依赖手工点测。

### 7.4 旧测试不适配新改动时的处理原则

若本次任务会有意改变部分旧行为，应明确区分：

- 哪些旧测试代表仍应成立的行为边界，必须继续通过
- 哪些旧测试仅对应被明确调整的旧预期，应更新为新预期

不应因为旧测试不适配新改动，就整体删除旧行为验证而不提供替代保护。

---

## 8. 最低回归清单

任何正式改动在合入前，至少应完成以下之一：

- 相关自动化测试通过
- 必要的手工回归完成并可说明

以下内容原则上应至少检查其一：

- 改动点本身
- 改动点相邻路径
- 关键交互入口
- 关键异常分支
- 历史高风险区域

---

## 9. 发布前回归要求

发布前应根据改动范围，选择适当级别的 smoke / 回归：

### 小改动发布

至少验证：

- 改动点主路径
- 关键页面可正常打开
- 基础错误无新增

### 中等改动发布

至少验证：

- 改动点主路径
- 相邻路径
- 关键业务流 smoke
- 核心页面打开与交互

### 高风险发布

至少验证：

- 核心流程 smoke
- 关键业务路径
- 关键配置 / 鉴权 / 路由
- 关键租户或关键站点场景
- 必要时多环境验证

---

## 10. 回归结果记录要求

回归完成后，建议至少记录：

- 本次改动范围
- 本次回归范围
- 使用的验证方式
- 是否通过
- 未覆盖风险点
- 是否需要后续补充自动化测试

可记录位置包括：

- PR 描述
- 任务记录
- 发布说明
- 测试报告
- known-issues

---

## 11. AI 协作回归要求

### 11.1 AI 修改后不得只验证当前文件

AI 对代码进行修改后，不应只验证“当前修改的地方编译通过”，还应评估：

- 相邻模块是否受影响
- 同模式调用点是否需要验证
- 是否需要补自动化测试
- 是否需要 smoke

### 11.2 AI 输出应说明回归建议

AI 应至少说明：

- 需要回归哪些范围
- 推荐使用哪些测试或手工验证方式
- 已执行了哪些验证
- 哪些部分仍有风险

### 11.3 AI 不得以“未报错”代替“已回归”

编译通过、不报错、不抛异常，不等于完成回归验证。

---

## 12. 禁止事项

以下行为原则上禁止：

- 只验证当前问题，不验证影响面
- 共享逻辑改动后只测单页面
- 高风险改动无 smoke
- 把“本地能跑”当成“完成回归”
- 手工验证无记录
- AI 修改后不补充回归说明
- 因赶时间跳过关键路径验证

---

## 13. 与其他规范的关系

本规范与以下规范协同工作：

- `FE-QUALITY-001 调试规范`：负责问题定位
- `FE-QUALITY-002 Bug 修复规范`：负责修复闭环
- `FE-QUALITY-003 测试规范`：负责测试分层与工具职责
- `FE-QUALITY-004 Mock规范`：负责模拟边界与方式
- Git / 变更控制规范：负责合入前门禁

关系可概括为：

- 调试规范：找到问题
- Bug 修复规范：修复问题
- 测试规范：建立验证能力
- 本规范：决定本次改动要回归哪些范围
- 变更控制规范：确保必要回归不被跳过

---

## 14. 最低执行要求

正式改动在合入前，至少应满足：

- 明确改动影响面
- 明确回归范围
- 至少完成一种有效验证
- 对高风险改动完成关键路径检查
- 能说明哪些风险已覆盖、哪些未覆盖

对于共享逻辑、基础设施、高风险改动，额外要求：

- 不能只做单点验证
- 应扩展到主要消费方或关键路径

---

## 15. 一句话原则

**改完当前问题，不代表完成交付；回归验证必须覆盖改动影响面，并优先用最小必要范围兜住主要风险。**
