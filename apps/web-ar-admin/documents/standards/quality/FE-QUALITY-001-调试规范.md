# 调试规范

## 1. 目标

统一项目中的调试方式，避免“直接猜问题、只看现象、不留证据、不沉淀经验”的低质量排查行为，建立可定位、可验证、可复用、可沉淀的调试闭环。

## 2. 适用范围

本规范适用于：

- `apps/*` 下所有前端业务代码
- `packages/*` 下所有共享包、工具包、基础设施包
- 日常开发、联调、回归、迁移中的问题排查
- AI 与人工共同参与的问题定位场景

本规范主要覆盖：

- 页面显示异常
- 组件渲染异常
- 状态流异常
- 请求链路异常
- 逻辑行为异常
- 路由与交互异常
- Monorepo 协作问题
- AI 编程中的调试与验证过程

## 3. 核心原则

### 3.1 分层调试，不依赖单一工具

调试应按问题层次选择合适工具，不依赖单一工具覆盖所有场景。

### 3.2 先定位问题类型，再选择工具

调试前应先判断问题属于：

- 浏览器呈现问题
- Vue 运行时问题
- 请求链路问题
- 纯逻辑问题
- 构建 / 工程问题
- 多包协作问题

### 3.3 以证据为基础，不凭感觉修改

正式进入修复前，应尽量取得明确证据，例如：

- 复现步骤
- console 报错
- Network 请求记录
- DOM / 样式快照
- 组件树状态
- 关键变量值
- 相关改动历史

### 3.4 先缩小范围，再深入根因

调试时应优先回答：

- 是页面问题还是逻辑问题
- 是组件问题还是全局状态问题
- 是请求没发出，还是请求发错了
- 是输入错误，还是转换错误，还是展示错误

### 3.5 调试不是终点，必须进入后续动作

调试结束后，问题应进入以下出口之一：

- 进入 Bug 修复流程
- 进入测试补充流程
- 进入 known-issues 记录
- 进入规范补充 / 文档补充
- 进入环境治理 / 监控治理

具体出口规则继续参考：

- `FE-QUALITY-002-Bug 修复规范`
- `FE-QUALITY-003-测试规范`
- `FE-QUALITY-005-回归验证规范`

## 4. 调试工具职责映射

### 4.1 Vue Devtools

适合查看：

- 组件树
- props
- emits
- Pinia / store
- route
- 组件命中是否正确
- 运行时状态变化

定位：

- Vue 运行时状态调试工具

### 4.2 Chrome DevTools / DevTools MCP

适合查看：

- DOM 结构
- class 覆盖关系
- 样式优先级
- console error
- network 请求
- request / response 数据
- 页面真实浏览器行为

定位：

- 浏览器层调试工具

### 4.3 vite-plugin-vue-inspector

适合查看：

- 页面元素对应源码位置
- 大型组件树快速定位
- 多目录 / monorepo 组件来源追踪

定位：

- 源码定位增强工具

### 4.4 Vitest

适合查看：

- 工具函数
- mapper
- composable
- service
- store
- 纯逻辑模块
- 模块级协作问题

定位：

- 逻辑与模块调试验证工具

### 4.5 Playwright

适合查看：

- 页面交互链路
- 关键流程复现
- 宿主链 smoke
- 独立 app smoke
- 页面级回归问题

定位：

- 浏览器级自动化复现与验证工具

## 5. 问题分型与首选入口

### 5.1 DOM / 样式 / 控制台 / 请求问题

首选：

- Chrome DevTools
- DevTools MCP

### 5.2 组件树 / props / store / route 问题

首选：

- Vue Devtools

### 5.3 不知道页面元素来自哪个组件

首选：

- vite-plugin-vue-inspector

### 5.4 util / mapper / composable / service / store 逻辑问题

首选：

- Vitest

### 5.5 页面交互链路、回归问题

首选：

- Playwright

## 6. 标准调试流程

### 6.1 先确认问题现象

至少明确以下内容：

- 现象是什么
- 预期是什么
- 影响范围是什么
- 是否稳定复现
- 最近是否有相关变更

### 6.2 对问题进行分型

将问题归入以下一类或多类：

- 页面结构 / 样式问题
- 组件运行时问题
- 请求链路问题
- 状态管理问题
- 纯逻辑问题
- 工程构建问题
- 多包依赖或 Monorepo 问题

### 6.3 选择首个排查入口

选择与问题最接近的一层开始排查，而不是从所有工具一起乱试。

### 6.4 收集证据并缩小范围

优先收集：

- 报错信息
- request / response
- DOM 变化
- 组件状态
- 输入输出差异
- 最近变更点

### 6.5 确认根因候选

在进入修改前，应至少形成一个可说明的问题判断，例如：

- 页面没有命中正确组件
- props 传递不正确
- store 状态来源冲突
- 请求参数构造错误
- DTO 转换错误
- 响应式链路中断
- 样式优先级被覆盖

### 6.6 再进入修复或测试补充

调试阶段的目标是找到问题，不是顺手直接修掉一切。定位完成后应转入：

- Bug 修复流程
- 测试补充流程
- 文档补充流程

## 7. 常见问题分类与排查路径

### 7.1 页面显示不对

优先检查：

- DOM 结构是否正确
- class 是否命中
- 样式优先级是否被覆盖
- 是否命中错误组件
- 页面数据是否为空

推荐工具：

- Chrome DevTools
- Vue Devtools
- vue-inspector

### 7.2 组件有渲染但行为不对

优先检查：

- props 是否正确
- emits 是否触发
- composable 返回值是否正确
- store / route 是否符合预期

推荐工具：

- Vue Devtools
- Vitest
- vue-inspector

### 7.3 接口没按预期工作

优先检查：

- 请求是否真的发出
- URL / method / params / body 是否正确
- header 是否注入
- response 是否符合预期
- 错误处理是否吞掉信息

推荐工具：

- Chrome DevTools / Network
- DevTools MCP
- Vitest
- Mock 能力

### 7.4 纯逻辑结果不对

优先检查：

- 输入输出
- 边界条件
- 时间 / 时区
- DTO → Domain 转换
- mapper / util / service 链路

推荐工具：

- Vitest

### 7.5 页面流程链路偶发异常

优先检查：

- 是否存在竞态
- 状态是否同步
- route 是否正确切换
- 多步骤流程中断点在哪
- 是否需要页面级复现

推荐工具：

- Playwright
- Vue Devtools
- Chrome DevTools

### 7.6 Monorepo / 多包协作异常

优先检查：

- 包是否正确链接
- 导出是否正确
- 类型是否一致
- 构建产物是否过期
- 引用路径是否有误

推荐工具：

- Vite / TS 报错信息
- 本地构建命令
- Vitest
- package 级调试日志

---

## 8. 最低证据清单

正式进入修复前，至少应具备以下证据中的两项：

- 明确复现步骤
- console 报错或错误栈
- Network 请求证据
- DOM / 样式快照
- 关键状态快照
- 相关代码定位结果
- 最近变更记录

高价值或复杂问题，建议保留三项及以上证据。

---

## 9. Vue3 / Vite / Monorepo 专项要求

### 9.1 Vue3 项目

重点检查：

- 组件树是否命中
- props / emits 是否匹配
- store 状态来源是否唯一
- composable 是否成为双真相来源
- watch / computed / ref 是否正确联动

### 9.2 TypeScript 项目

重点检查：

- 类型推断是否掩盖真实问题
- any / unknown 是否绕过了约束
- DTO / Domain 类型是否混用
- 类型正确但业务语义错误的情况

### 9.3 Monorepo 项目

重点检查：

- package 导出是否正确
- workspace 链接是否生效
- 本地依赖是否使用到旧产物
- 不同 app 是否消费了错误版本的共享包

---

## 10. AI 协作调试要求

### 10.1 AI 不应直接跳到修复

AI 辅助调试时，应先输出：

- 问题分型
- 初步根因候选
- 需要查看的证据
- 推荐排查工具

### 10.2 AI 修改前应先说明定位依据

若 AI 准备修改代码，应至少说明：

- 为什么判断问题在这里
- 证据是什么
- 修改影响范围是什么
- 是否已补充或建议补充测试

### 10.3 AI 调试结果应可追溯

AI 输出中建议包含：

- 复现方式
- 定位路径
- 根因判断
- 修改建议
- 验证建议

---

## 11. 禁止事项

以下行为原则上禁止：

- 未分型即盲目排查
- 无证据直接改代码
- 一上来就全局搜索式乱改
- 用打印和猜测替代真正定位
- 发现问题后不补记录
- AI 未验证即宣称问题解决

---

## 12. 与其他规范的关系

本规范主要负责：

- 如何定位问题
- 如何分型问题
- 如何选择调试工具
- 如何收集调试证据

相关规范分工如下：

- `FE-QUALITY-002 Bug 修复规范`：负责定位后的修复闭环
- `FE-QUALITY-003 测试规范`：负责测试分层、补测试与验证要求
- `FE-QUALITY-005 回归验证规范`：负责修复后如何做防回归验证

---

## 13. 最低执行要求

正式调试活动，至少应满足以下要求：

- 明确问题现象
- 给出初步问题分型
- 选择首个排查入口
- 收集至少两项有效证据
- 形成可说明的根因候选
- 明确进入后续哪个出口动作

---

## 14. 一句话原则

**先分型，再选工具；先拿证据，再做判断；先定位根因，再进入修复。**
