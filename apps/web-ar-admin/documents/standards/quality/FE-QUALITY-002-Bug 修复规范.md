# Bug 修复规范

## 1. 目标

统一项目中的 Bug 修复方式，避免“直接改现象、不固化问题、不做回归”的低质量修复行为，建立可复现、可验证、可追溯的 Bug 修复闭环。

## 2. 适用范围

本规范适用于：

- `apps/*` 下所有前端业务代码
- `packages/*` 下所有共享包、工具包、基础设施包
- 日常开发中的缺陷修复、回归修复、迁移修复
- AI 与人工共同参与的代码修复场景

不完全适用于：

- 紧急线上止血且暂时无法稳定复现的问题
- 第三方服务偶发异常导致的临时绕过处理
- 纯环境故障、网络抖动、外部依赖失效等非代码逻辑问题

对于不适用场景，应先止血、先隔离影响，再补充定位、测试与规范化修复。

## 3. 核心原则

### 3.1 先复现，后修复，再回归

Bug 修复必须遵循：

**先复现 → 再修复 → 再回归**

禁止直接凭感觉修改代码后声称“问题已修复”。

其中，本规范负责“Bug 修复流程与修复动作”，回归范围控制与回归层级继续参考 `FE-QUALITY-005-回归验证规范`。

### 3.2 可复现 Bug 优先先写失败测试

凡是能够稳定复现的 Bug，优先通过测试固化失败场景，再修改实现。

目标是先证明：

- 该 Bug 当前确实存在
- 当前代码无法通过该场景
- 修复后可用同一测试验证结果

### 3.3 修复必须可验证

修复完成后，必须通过以下方式之一验证：

- 原失败测试转为通过
- 相关测试集通过
- 必要的手工回归通过
- 关键链路 smoke 通过

### 3.4 修复不只消除现象，还要关注根因

修复时应优先定位根因，避免：

- 仅修改表层判断
- 仅屏蔽异常输出
- 仅增加临时兜底而不解决真实问题

### 3.5 高价值 Bug 要沉淀资产

对高价值、高频、易复发或影响架构边界的 Bug，修复完成后应沉淀至少一种资产：

- 测试用例
- 文档说明
- 规范补充
- known-issues 记录
- 决策或演进记录

## 4. 标准修复流程

### 4.1 确认 Bug

修复前应先明确以下内容：

- 现象是什么
- 预期行为是什么
- 复现步骤是什么
- 影响范围是什么
- 是否稳定复现
- 是否已有相关测试或历史问题记录

建议至少在任务描述、Issue、PR 或修复说明中写清以上内容。

### 4.2 选择复现方式

根据 Bug 类型，选择最合适的复现方式。

#### 4.2.1 单元测试复现

适用于：

- 工具函数
- mapper / parser
- 纯计算逻辑
- 时间处理
- 条件分支
- 数据转换

推荐工具：

- Vitest

#### 4.2.2 集成测试复现

适用于：

- composable
- service
- store
- request 处理链
- 模块协作逻辑

推荐工具：

- Vitest

#### 4.2.3 浏览器链路复现

适用于：

- 页面交互问题
- UI 状态错误
- 路由跳转错误
- 网络交互链路问题
- 多步骤业务流程问题

推荐工具：

- Playwright
- Vue Devtools
- Chrome DevTools / DevTools MCP

### 4.3 先写失败测试

对于可稳定复现的 Bug，应先新增一个能稳定失败的测试。

该测试应满足：

- 能准确命中问题
- 尽量最小复现
- 不引入无关上下文
- 失败原因清晰
- 可在修复后直接复用为回归测试

### 4.4 再修改实现

只有在失败测试明确存在后，才开始修改实现。

修复时要求：

- 以最小必要改动为优先
- 优先解决根因
- 避免顺手引入无关重构
- 避免在同一次修复中混入 unrelated feature

### 4.5 跑回归验证

修复完成后，至少执行：

- 新增的失败测试
- 与该模块相关的测试集

视影响范围，进一步执行：

- 相关页面手工验证
- 关键业务链路 smoke
- Playwright 关键流程
- 多环境验证

### 4.6 更新说明与沉淀

根据问题价值，决定是否补充：

- 文档
- known-issues
- 测试策略
- 规范条款
- 演进记录

## 5. Bug 分类与推荐修复方式

### 5.1 纯逻辑 Bug

示例：

- 条件判断错误
- 计算错误
- 映射错误
- 时间格式错误

修复建议：

- 优先写 Vitest 单元测试
- 修复后保留测试作为长期回归用例

### 5.2 模块协作 Bug

示例：

- composable 与 store 协作异常
- service 调用链行为异常
- request 包装链逻辑异常

修复建议：

- 优先用 Vitest 集成测试复现
- 必要时补充 mock 或测试桩

### 5.3 页面交互 Bug

示例：

- 按钮状态错误
- modal 打开关闭异常
- 表单联动异常
- route 跳转错误

修复建议：

- 先使用 Vue Devtools / Chrome DevTools 排查
- 能沉淀为 Playwright 的优先沉淀
- 难以自动化的至少补手工回归说明

### 5.4 样式与结构 Bug

示例：

- DOM 结构错位
- class 覆盖异常
- 样式优先级错误
- 响应式显示异常

修复建议：

- 使用 Chrome DevTools / DevTools MCP
- 必要时补页面级验证
- 若根因是组件职责错误，应回到架构层处理

### 5.5 请求与接口 Bug

示例：

- header 丢失
- 参数序列化错误
- response 解析错误
- 错误处理不一致

修复建议：

- 优先补 Vitest 测试
- 涉及请求链路时结合 mock / network 调试
- 如属基础设施问题，应补 request 层或 API 层回归测试

## 6. 必须补测试的场景

以下场景，原则上必须补测试：

- Bug 可稳定复现
- Bug 属于纯逻辑或可程序化验证问题
- Bug 曾反复出现
- Bug 位于核心路径
- Bug 位于共享包或基础设施层
- Bug 属于迁移中的行为对齐问题
- Bug 由 AI 修改产生或由 AI 修复

## 7. 可以暂不先写测试的场景

以下场景可先止血、后补测试：

- 线上高优先级事故
- 问题暂时无法稳定复现
- 强依赖外部第三方不稳定服务
- 主要问题在环境、配置或网络层
- 需要先加日志/观测手段才能定位

但后续应尽量补齐以下内容之一：

- 复现方式
- 自动化测试
- known-issues 记录
- 根因说明

## 8. 禁止事项

以下行为原则上禁止：

- 未复现即直接修改代码
- 未验证即宣称修复完成
- 将大量无关改动混入 Bug 修复
- 用注释、屏蔽、吞异常替代真正修复
- 只做页面层兜底而不查根因
- 修复共享逻辑却不补测试
- AI 修改后不跑测试直接提交

## 9. 提交与合并要求

### 9.1 开发过程允许多次过程性提交

在开发分支中，允许存在多次调试型、试探型提交。

### 9.2 合入前应整理历史

在提 PR 或合入主干前，应整理提交历史，避免将以下内容直接带入主干：

- 无语义提交
- 重复修正提交
- 临时调试提交
- 噪声 commit

### 9.3 Bug 修复建议保留 1~2 个清晰提交

推荐形式：

#### 形式 A：单提交

- `fix: resolve xxx bug and add regression test`

#### 形式 B：双提交

- `test: add regression test for xxx bug`
- `fix: resolve xxx bug`

对于高价值 Bug，更推荐保留“测试复现 + 修复实现”两个清晰提交。

## 10. AI 修复补充要求

当 Bug 由 AI 辅助修复时，额外要求：

- AI 应优先补失败测试，而不是直接改实现
- AI 修复后必须运行相关测试
- AI 输出中应说明：
  - Bug 复现方式
  - 修改范围
  - 验证结果
  - 是否补充测试
- AI 不应在一次 Bug 修复任务中顺手做大范围无关重构

## 11. 与其他规范的关系

本规范与以下规范协同工作：

- 调试规范：负责定位问题
- 测试规范：负责定义测试分层与工具职责
- 回归验证规范：负责定义修复后的验证范围
- Git 提交规范：负责修复提交的表达方式
- 变更控制规范：负责修复类改动的门禁与约束

关系可概括为：

- 调试规范：找到问题
- 本规范：修问题
- 测试规范：验证问题
- 回归规范：防止问题再次出现

## 12. 最低执行要求

对正式 Bug 修复，最低要求如下：

- 明确 Bug 现象
- 明确预期行为
- 完成至少一种有效复现
- 完成修复验证
- 更新相关测试或回归说明

对可稳定复现的 Bug，额外要求如下：

- 优先新增失败测试
- 修复后确保该测试转为通过

## 13. 一句话原则

**凡是可稳定复现的 Bug，先写失败测试固化问题，再修复代码，最后跑测试验证修复成立并防止回归。**
