# ENG-STD-004 依赖边界与架构约束规范

## 1. 文档定位

本文档用于定义仓库内依赖关系、架构边界和引用约束，重点回答：

- 哪些层可以互相引用
- 哪些引用应被视为违规
- 如何控制跨 app / 跨 package 依赖
- 如何防止循环依赖与测试污染生产代码

本文档不负责定义具体工具安装方式，工具接入细节由接入手册或执行规范承载。

## 2. 目标

本规范希望解决以下问题：

- 页面可以随意直连底层实现
- package 的内部实现被外部随意穿透
- 依赖图失控，后续重构代价高
- 测试代码污染生产代码依赖
- 架构边界没有明确治理手段

架构约束的目标不是让代码绝对不能引用，而是让引用关系可解释、可治理、可逐步收敛。

## 3. 适用范围

本规范适用于仓库内所有源码、共享层、包层、测试层与工具层之间的依赖关系。

## 4. 分层依赖原则

建议将仓库依赖抽象为若干稳定层级，例如：

- app 层
- feature 层
- shared 层
- package 层
- 基础设施层

原则如下：

- 高层可以调用低层提供的公开能力
- 低层不应反向依赖高层实现细节
- 共享层不应被业务层任意污染
- 跨层引用应优先走公开入口

## 5. 引用规则

### 5.1 app / package / shared

建议遵守：

- app 层只组合能力，不沉淀基础设施逻辑
- package 层提供可复用能力，不直接依赖具体页面实现
- shared 层提供稳定公共能力，不应包含强业务耦合逻辑

### 5.2 非公开入口限制

不应直接引用包的内部实现细节，而应优先使用公开入口。

典型违规场景包括：

- 直接穿透到 package 内部深层路径
- 依赖测试目录里的辅助实现
- 以“临时可用”为由绕过公开 API

### 5.3 测试与生产隔离

测试代码应与生产代码保持清晰边界：

- 生产代码不应依赖测试目录
- 测试辅助能力不应默认暴露给生产使用
- E2E / 集成测试的工具代码应保持独立定位

## 6. 循环依赖规则

循环依赖应视为高风险问题，优先阻断或收敛。

处理原则：

- 先识别循环来源
- 再判断是否为结构性设计问题
- 能拆分就拆分
- 不能立即拆分时，必须记录为技术债并设后续治理计划

## 7. Dependency Cruiser 职责边界

Dependency Cruiser 的职责是辅助发现依赖边界问题，例如：

- 循环依赖
- 分层依赖违规
- 跨包违规引用
- 测试/生产隔离违规

它不是架构决策本身，架构决策应由规范和评审确定。

## 8. Baseline 与渐进治理

历史仓库可能已存在违规依赖，不要求一次性清零。

建议做法：

- 先建立 baseline
- 再限制新代码继续扩大问题
- 对高风险路径优先治理
- 对旧问题按优先级消化

## 9. 例外机制

当某条依赖约束暂时无法满足时，可申请例外，但必须满足：

1. 有明确原因
2. 有影响评估
3. 有临时范围
4. 有后续治理计划
5. 不允许长期无期限豁免

## 10. 一句话总结

依赖边界与架构约束规范的核心，是让仓库里的引用关系保持：

- 可解释
- 可检查
- 可收敛
- 可逐步演进
