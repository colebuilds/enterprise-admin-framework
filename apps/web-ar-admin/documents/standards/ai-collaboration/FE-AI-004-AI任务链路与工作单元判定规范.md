# AI 任务链路与工作单元判定规范

## 1. 目标

定义正式任务的标准链路、前置判断、验证出口、提交门槛与完整工作单元判定方式，避免任务在协作过程中出现阶段跳跃、判断重复、验证失焦和提交范围失控。

## 2. 适用范围

本规范适用于以下场景：

- 进入正式任务卡、边界、计划、执行、验证与提交流程的任务
- 需要判断是否适合 TDD、任务编排、多 agent 或自动执行的任务
- 需要把每个环节的输入、输出与出口条件定义清楚的任务
- 需要判断一个任务是否可视为完整工作单元的任务

本规范不直接实现自动执行、状态机代码或 runtime 存储，只定义协作判断规则。

## 3. 核心原则

### 3.1 每个环节都必须有输入和输出

任务链路中的每个环节都必须定义：

- 输入
- 输出
- 输出去向
- 出口条件

当前环节的输出应能直接被后续环节消费，不应依赖模糊的聊天记忆继续推进。

### 3.2 前置判断必须发生在 `生成边界` 前

正式任务在进入 `生成边界` 前，必须先完成以下判断：

- 是否支持 TDD 开发
- 是否支持任务编排
- 是否支持多 agent
- 是否需要同步生成或更新文档
- 是否支持自动执行

这些判断只负责策略收敛，不直接启动实现、子任务或自动执行。

### 3.2A 自动执行判断必须承担执行入口分流职责

`是否支持自动执行` 不能只停留在分析层。

当该判断完成后，必须进一步作用于执行入口分流：

- 命中自动执行条件：进入自动执行链路
- 未命中自动执行条件：回到人工 `开始执行` 链路

若自动执行判断结果不会改变执行入口，则该判断不应视为已闭环完成。

### 3.2B 简单任务可进入默认自动闭环

对于低风险、边界清晰、验证方式明确的简单任务，系统应允许进入默认自动闭环，而不是要求用户逐步驱动每一个阶段。

简单任务默认自动闭环的目标是：

- 减少重复确认
- 缩短低风险任务从判断到收口的路径
- 让已具备闭环条件的任务自动推进到收口

但该规则不覆盖复杂需求的拆分前置与拆分确认机制。

### 3.2C `开始执行` 后应默认自动推进后续链路

`开始执行` 属于总授权点。

当任务已经完成前置判断、边界、计划与必要确认，并收到 `开始执行` 后，若当前任务未命中人工接入条件，则系统应默认自动推进后续链路，而不是继续逐阶段等待人工驱动。

默认自动推进的后续链路至少包括：

- 执行
- 验证
- 文档落盘
- code review
- 提交结果
- commit / push
- 必要时版本号与迭代日志联动
- 总收口

以下场景仍必须停止自动推进并等待人工接入：

- 复杂需求尚未完成拆分确认
- 命中高风险或 `L3 / L4` 架构改动
- 反复验证不通过
- 依赖人工验收
- 需要人工做关键决策
- 命中受保护路径或其他显式禁止条件

该规则不替代复杂需求拆分确认机制，也不放宽高风险任务的人工作为优先级。

### 3.2D 复杂任务在 `开始执行` 前必须明确拆分与批次结构

对于大任务或复杂需求，系统在 `开始执行` 前必须已经明确任务拆分与批次结构，不得在进入执行后才临时决定任务结构。

至少必须明确：

- 当前是否已完成任务拆分
- 已拆成哪些中任务
- 当前属于：
  - 单任务
  - 单批次
  - 多批次
- 当前批次是什么
- 下一批次或下一中任务是什么
- 当前可提交单元是什么
- 总任务最终收口路径是什么

若上述结构信息未在 `生成计划` 阶段形成，则当前任务不得进入 `开始执行`。

该规则用于避免：

- 执行中临时拆任务
- 执行中临时改批次
- 当前 / 下一任务关系不清
- 总任务在执行后变成黑盒推进

### 3.2E Harness 本体实现任务默认必须先经过边界与计划

对于直接修改 Harness 本体系统能力的实现任务，系统不得因为上下文连续、任务较小或前序已存在设计文档，就跳过正式任务流中的边界与计划阶段。

此类任务至少包括：

- 生命周期、phase、hook、plugin、config、runtime、diagnostics、blocked / resume、版本治理等 Harness 系统能力实现
- `skills/harness-*`、`skills/ai-subtask-orchestration/*`、`scripts/harness-*`、`.harness-runtime/*` 等直接影响 Harness 主链路或控制面的修改

对这类任务，进入实现前默认必须具备：

- 任务卡
- 边界
- 计划
- `开始执行`

仅在“同一已激活正式任务内的极小连续修正”场景下，才允许使用短版边界 / 短版计划，但仍不得完全跳过这两个阶段。

该规则不否定：

- `开始执行` 后的默认自动推进
- 简单任务默认自动闭环

它只收紧“开始执行前的正式前置阶段”，避免 Harness 本体实现退回到“标题即实现”的旧模式。

### 3.3 验证不只等于测试

任务验证不仅包括测试结果，还必须覆盖：

- 功能结果验证
- 测试策略验证
- 边界一致性验证
- 计划一致性验证
- 提交文件范围验证

### 3.4 提交必须建立在验证完成之后

任务进入提交前，必须先确认：

- 实现已完成
- 验证已完成
- 实际文件范围符合边界约定
- 无关文件已排除
- 不依赖尚未完成的人工验证

### 3.5 需要人工验证的任务不可直接收口提交

若当前任务依赖人工验证，而人工验证尚未完成，则任务应停留在：

- `待人工验证`
- 或 `待验收`

此时不进入提交阶段。

### 3.6 自动执行规则需要文档层与 skill 层同时承接

自动执行属于执行链路规则，不能只停留在规范文档中。

至少需要两层同时承接：

- 文档规范层：定义自动执行何时成立、何时禁止、何时退回人工授权
- skill 执行层：在实际任务流中命中判断、消费判断结果并执行入口分流

若只更新文档而不更新对应 skill，则自动执行规则仍不具备真实执行约束力。

## 4. 标准任务链路

正式任务链路 V1 推荐顺序如下：

1. 任务卡
2. 分析
3. 是否支持 TDD 开发
4. 是否支持任务编排
5. 是否支持多 agent
6. 是否需要同步生成或更新文档
7. 是否支持自动执行
8. 生成边界
9. 生成计划
10. 执行入口判断
11. 执行
12. 验证
13. 提交结果
14. 提交
15. 开启下一链路

## 5. 前置判断规则

### 5.1 是否支持 TDD 开发

用于判断当前任务是否适合测试先行。

至少应判断：

- 需求边界是否清晰
- 行为是否可稳定断言
- 是否适合先写失败测试再实现
- 若不支持 TDD，替代验证方式是什么

输出应至少包括：

- `tdd_supported`
- 判断依据
- 测试策略建议
- 替代验证方式

### 5.2 是否支持任务编排

用于判断当前任务是否应从单任务升级为任务批次或任务集合。

至少应判断：

- 是否适合拆分成多个任务对象
- 是否存在强依赖
- 是否值得进入任务编排
- 建议继续单任务还是进入任务批次

输出应至少包括：

- `task_orchestration_supported`
- 拆分建议
- 依赖关系摘要
- 单任务 / 多任务建议

### 5.3 是否支持多 agent

用于判断当前任务在执行层是否适合多代理协作。

注意：

- 不支持任务编排，不等于不判断多 agent
- 单任务内部仍可能适合多 agent 协作

输出应至少包括：

- `multi_agent_supported`
- `agent_mode`
- 拆分原因
- 文件边界建议

### 5.4 是否需要同步生成或更新文档

用于判断开发任务是否需要同步沉淀文档结果。

至少应判断：

- 是否形成长期规则
- 是否形成新接入方式
- 是否形成新架构边界
- 是否需要沉淀给后续人和 AI 复用

输出应至少包括：

- `doc_update_required`
- 文档动作：`create / update / none`
- 文档落点
- 是否需要挂载最近一级 `README.md`

### 5.5 是否支持自动执行

用于判断当前任务是否具备自动推进条件。

至少应判断：

- 边界是否足够清晰
- 风险是否可控
- 验证方式是否明确
- 是否依赖人工验收
- 是否涉及高风险路径

输出应至少包括：

- `auto_execute_supported`
- 判断依据
- 建议走自动链路还是人工 `开始执行` 链路

当 `auto_execute_supported = yes` 时，应视为当前任务命中自动执行入口，不再要求额外的人工 `开始执行` 口令。

当 `auto_execute_supported = no` 时，必须回到人工 `开始执行` 链路。

自动执行不得适用于以下场景：

- 高风险改动
- 边界不清
- 验证不可闭环
- 依赖人工验收
- 命中受保护路径
- 需要人工做关键决策或关键确认

### 5.6 是否属于简单任务自动闭环场景

用于判断当前任务是否可默认自动推进到收口。

至少应判断：

- 风险是否较低
- 边界是否清晰
- 涉及文件范围是否较小
- 是否不需要复杂拆分
- 验证方式是否明确
- 是否不依赖人工验收

可作为简单任务的典型场景包括：

- 清晰文档任务
- 小范围规则修正
- 小型 bugfix
- 边界明确的单点配置或脚本补强

不应作为简单任务自动闭环的场景包括：

- 复杂需求
- 多阶段改造
- 跨多个模块或目录的大范围修改
- 需要拆分确认的任务
- 依赖人工验收或关键决策的任务

输出应至少包括：

- `simple_task_supported`
- 判断依据
- 默认自动收口止点
- 需要人工确认的回退条件

当 `simple_task_supported = yes` 时，默认自动收口止点可为：

- 执行
- 验证
- 提交
- push

其中当前阶段默认推荐的简单任务自动闭环止点为：

- `push`

前提是：

- 文件范围符合边界
- 验证已完成
- 文档落盘要求已满足
- 不依赖人工验证

当 `simple_task_supported = no` 时，应继续按普通正式任务链路推进，不得强行自动闭环。

## 6. 环节输入与输出

### 6.1 任务卡

输入：

- 用户需求
- 当前上下文
- 项目规则

输出：

- 任务标题
- 任务类型
- 风险等级
- 本次目标
- 本次不包含
- 当前阶段

输出去向：

- 分析

出口条件：

- 已形成唯一任务对象

### 6.2 分析

输入：

- 任务卡
- 代码或文档现状
- 项目规则

输出：

- 现状摘要
- 关键约束
- 风险点
- 影响范围初判

输出去向：

- 所有前置判断节点

出口条件：

- 已具备进入策略判断的信息基础

### 6.3 前置判断组

输入：

- 任务卡
- 分析结果

输出：

- TDD 判断结果
- 任务编排判断结果
- 多 agent 判断结果
- 文档判断结果
- 自动执行判断结果

输出去向：

- 生成边界
- 生成计划
- 后续验证与执行组织

出口条件：

- 所有前置策略已完成收敛

### 6.4 生成边界

输入：

- 任务卡
- 分析结果
- 所有前置判断结果

输出：

- 本次包含
- 本次不包含
- 影响范围
- 预期文件范围
- 风险限制
- 验证口径
- 预期产物

输出去向：

- 生成计划
- 执行
- 验证
- 提交结果

出口条件：

- 已能约束执行和后续提交范围

### 6.5 生成计划

输入：

- 边界
- 所有前置判断结果

输出：

- 执行步骤
- 依赖关系
- 风险缓解方式
- 验证方案
- 提交策略
- 拆分结果
- 批次结构
- 当前 / 下一任务关系
- 总收口路径

输出去向：

- 执行
- 验证
- 提交结果

出口条件：

- 已具备可执行顺序与收口方式
- 若属于复杂任务，还必须已具备稳定拆分与批次结构

### 6.6 执行入口判断

输入：

- 自动执行判断结果
- 当前授权状态

输出：

- `execution_entry = auto_start / manual_start`
- 当前任务进入自动执行链路或人工 `开始执行` 链路的结论

输出去向：

- 执行

出口条件：

- 已明确当前任务的唯一执行入口

执行规则如下：

- 若 `execution_entry = auto_start`，则当前任务可直接进入执行阶段
- 若 `execution_entry = manual_start`，则当前任务必须等待人工 `开始执行`
- 若当前任务已收到人工 `开始执行`，则应将后续链路视为默认自动推进链路，除非中途命中人工接入条件

对于复杂任务：

- 若拆分结构、批次结构、当前 / 下一任务关系或总收口路径未明确，则不得进入 `开始执行`

在自动执行规则正式生效前，若项目入口规则仍声明 `开始执行` 为唯一授权口令，则应将当前状态视为“规则未闭环”，而不是默认已支持自动执行

### 6.7 执行

输入：

- 边界
- 计划
- 协作方式
- 文档要求

输出：

- 实际改动
- 实际命令
- 实际产物
- 未完成项
- 计划外改动说明

输出去向：

- 验证
- 提交结果

出口条件：

- 已有可验证的实际结果

### 6.8 验证

输入：

- 边界
- 计划
- 执行结果
- TDD 判断结果
- 文档判断结果
- 当前工作区状态

输出：

- 功能验证结果
- 测试策略验证结果
- 边界验证结果
- 文件范围验证结果
- 计划一致性验证结果
- 是否需要人工验证
- 是否允许提交

输出去向：

- 提交结果
- 提交
- 或 `待人工验证 / 待验收`

出口条件：

- 已明确是否具备提交条件

### 6.9 提交结果

输入：

- 执行结果
- 验证结果
- 当前文件状态

输出：

- 新增文件
- 修改文件
- 删除文件
- 明确不提交的文件
- 是否建议提交

输出去向：

- 提交

出口条件：

- 提交范围已清晰
- 已完成提交前文件清单展示

执行要求：

- 当任务已进入默认自动推进链路时，`提交结果` 属于内部必经展示节点
- 系统在验证完成后应自动进入该节点
- 展示文件清单后应继续进入提交动作
- `提交结果` 不应再被视为需要人工额外触发的停止点

### 6.10 提交

输入：

- 提交结果
- 提交指令
- 提交权限状态

输出：

- 提交记录
- 推送状态
- 当前任务最终状态

输出去向：

- 下一链路

出口条件：

- 当前任务已正式收口

## 7. 验证分层

### 7.1 功能结果验证

验证功能是否真的满足任务目标。

可采用：

- 单元测试
- 集成测试
- E2E / 冒烟测试
- 人工操作验证
- 联调结果
- 日志、截图或结果证明

### 7.2 测试策略验证

验证本次是否兑现了既定开发策略。

- 若支持 TDD，应检查是否按测试先行推进
- 若不支持 TDD，应检查是否补了可补测试
- 若不能补自动化测试，应检查替代验证是否已完成

### 7.3 边界一致性验证

验证实际结果是否仍在边界内。

至少包括：

- 修改文件是否符合边界约定
- 是否碰了未授权路径
- 是否引入计划外范围扩散
- 是否出现顺手修复

### 7.4 计划一致性验证

验证执行是否按计划推进。

至少包括：

- 计划步骤是否完成
- 是否有未完成项
- 是否有计划外改动
- 风险是否已显式说明

### 7.5 提交文件范围验证

验证当前提交集是否符合边界承诺。

至少包括：

- 应提交文件是否齐全
- 不应提交文件是否已排除
- 当前文件集是否表达单一任务意图
- 当前文件集是否符合 `生成边界` 时约定的文件范围

## 8. 提交门槛

任务进入提交前，必须同时满足：

- 实现已完成
- 最小验证已完成
- 文件范围验证通过
- 无关文件已明确排除
- 不依赖尚未完成的人工验证

若上述任一条件不满足，则任务不得进入提交。

## 9A. 自动执行闭环要求

一个任务若要被认定为“支持自动执行”，至少必须同时满足：

- 自动执行判断结果明确
- 执行入口分流结果明确
- 自动执行禁止场景已被排除
- 验证方式可闭环
- 对应 skill 已具备消费该判断并执行分流的能力

若缺少上述任一项，则只能视为“具备自动执行分析结论”，不能视为“具备自动执行能力”。

## 9B. 人工授权后的默认自动收口要求

当任务未命中自动执行入口，但已收到人工 `开始执行` 时，应适用“人工授权后的默认自动收口”规则。

该规则至少要求：

- `开始执行` 后不再逐阶段等待人工 `继续`
- 系统默认继续完成执行、验证、文档落盘、提交结果与提交动作
- 若任务属于 Harness 系统能力变化，还应继续完成版本号与迭代日志联动检查
- 中途若命中人工接入条件，必须显式停止自动推进并回报当前状态

若系统仍要求用户在 `开始执行` 后逐阶段再次发出：

- `继续`
- `提交结果`
- `add commit push`

才肯继续推进，则不应视为已满足本条规则。

## 9. 完整工作单元定义

满足以下条件的任务，可视为完整工作单元：

- 能完成任务卡、分析、前置判断、边界、计划、执行、验证、提交与收口
- 无需人工补关键决策
- 无需人工补核心验证
- 无需人工补提交判断
- 能在既定规则下形成可独立验证、可独立提交的结果

若任务仍依赖以下任一内容，则不应视为完整工作单元：

- 人工补边界
- 人工补验证
- 人工决定是否提交
- 人工决定是否切换下一链路

## 10. 执行建议

- 不要在没有前置判断结果时直接生成边界
- 不要把验证简化为“测试是否通过”
- 不要在未核对文件范围时直接进入提交
- 需要人工验证的任务，应显式停留在待验证状态
- 一个可独立收口的任务，优先形成一个可独立提交的工作单元
