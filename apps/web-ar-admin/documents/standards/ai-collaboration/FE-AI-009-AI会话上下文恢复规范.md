# AI会话上下文恢复规范

## 1. 目标

定义项目内跨 session、跨 `/new`、跨天续做时的会话上下文恢复机制，避免出现“需求重讲一遍、当前进度丢失、分支状态不清、下一步不明确”的问题。

本规范的目标是：

- 为所有任务提供统一的恢复底座
- 降低对聊天历史的依赖
- 让 AI 和人都能快速恢复到可继续工作的状态
- 为 `bug-sop`、PIR、调试流等场景提供稳定挂接点

本文档定义的是“如何恢复任务现场”，而不是某一种具体业务流程。

## 2. 适用范围

本规范适用于：

- 普通需求开发的跨 session 恢复
- Bug 处理流程的跨 session 恢复
- PIR 规划、实施、评审阶段的跨 session 恢复
- 调试、验证、交付等需要中断后恢复的任务

本规范不主要解决：

- 远端多人协作同步
- 仓库正式知识沉淀
- 需求本身的分析、设计或实现规范

这些内容应分别由正式项目文档、场景规范和协作规范承担。

## 3. 核心原则

### 3.1 会话恢复是通用底座，不绑定单一场景

恢复机制应先服务“所有任务”，再按需挂接 Bug、PIR、调试等场景层。

### 3.2 优先恢复任务现场，而不是依赖聊天历史

新 session 恢复时，默认先读交接快照，而不是回溯整段旧聊天。

### 3.3 快照只在稳定检查点生成

恢复快照不追求实时自动记录，而应在阶段稳定、信息可信时手动生成，避免噪音和半成品状态污染恢复入口。

### 3.4 统一入口优先于分散入口

恢复入口应固定且可预测。新 session 不应依赖“记得住上次文件名”，而应先从统一入口定位当前有效快照。

### 3.5 通用层与场景层分离

通用层只负责恢复任务现场；场景层负责恢复 Bug、PIR、调试等业务上下文。两者应明确分层，不互相替代。

## 4. 分层模型

本规范采用四层恢复模型。

### 4.1 L0：工作区恢复

负责恢复代码现场，包括：

- 当前仓库
- 当前分支
- 工作区是否干净
- 最近提交状态

L0 只回答“代码现场是什么”，不回答“任务做到哪了”。

### 4.2 L1：任务恢复

负责恢复任务的通用现场，包括：

- 当前任务目标
- 已完成事项
- 待办事项
- 改动文件
- 已做验证
- 下一步动作

L1 是所有任务都必须具备的恢复层。

### 4.3 L2：场景恢复

负责恢复某个任务类型特有的信息，例如：

- Bug 处理中的 bug list、batch、待补充项
- 需求开发中的设计结论、非目标、已确认方案
- 调试任务中的复现路径、根因假设、已排除项

L2 不替代 L1，而是在 L1 之上补充场景上下文。

### 4.4 L3：阶段恢复

负责恢复任务在执行链条中的具体阶段，例如：

- `pir-planner`
- `pir-implementer`
- `pir-reviewer`

L3 回答“现在卡在哪一步”，用于精确衔接执行流程。

## 5. 目录与文件约定

### 5.1 本地恢复目录

恢复快照统一放在：

- `/.local-work/session-context/`

该目录仅用于本地恢复，不替代正式规范文档，不作为项目共享真源。

### 5.2 统一恢复入口

恢复入口统一为：

- `/.local-work/session-context/LATEST_PATH.txt`

`LATEST_PATH.txt` 负责指向当前最值得恢复的一份交接快照。

### 5.3 快照文件命名

每个任务应对应一份独立快照，命名建议：

- `<task-slug>-YYYY-MM-DD.md`

示例：

- `withdraw-dashboard-batch-2-2026-04-23.md`
- `merchant-user-import-2026-04-23.md`

命名应表达任务语义，而不是聊天轮次。

## 6. 快照内容最小要求

每份交接快照至少应包含以下内容：

1. 会话目标
2. 当前工作区状态
3. 本轮已完成事项
4. 本轮改动文件清单
5. 校验与结果
6. 待办与下一步
7. 关联文档
8. `/new` 后恢复步骤

以上 8 项构成通用恢复底座，不因任务类型变化而省略。

## 7. 场景附加块

在满足通用快照最小要求后，可按场景追加附加块。

### 7.1 Bug 场景

可追加：

- bug list 范围
- batch 划分
- 待测试补充项
- 待产品确认项
- 方案卡状态

### 7.2 需求开发场景

可追加：

- 需求结论
- 非目标
- 已确认设计决策
- 实施计划状态

### 7.3 PIR 场景

可追加：

- 当前阶段：`planner / implementer / reviewer`
- 已批准计划
- 当前阻塞点

### 7.4 调试场景

可追加：

- 复现路径
- 根因假设
- 已排除项
- 下一轮验证点

## 8. 快照生成时机

恢复快照应只在稳定检查点手动生成。

推荐生成时机：

- 需求结论已确认后
- 方案已审过后
- 实施计划已批准后
- 一个 batch 完成后
- 准备切 session 前

不建议在“实现进行到一半”或“信息仍大幅不确定”时生成快照。

## 9. 标准恢复流程

新 session 恢复时，统一按以下流程执行：

1. 读取 `/.local-work/session-context/LATEST_PATH.txt`
2. 读取其指向的交接快照
3. 执行 `git status --short`
4. 执行 `git rev-parse --abbrev-ref HEAD`
5. 根据交接快照中的“待办与下一步”，决定是否进入：
   - 普通需求流
   - `bug-sop`
   - `pir-planner`
   - `pir-implementer`
   - `pir-reviewer`
   - 调试流

恢复入口始终先走通用层，再进入场景层。

## 10. 与场景层的挂接方式

### 10.1 与 bug-sop 的关系

`bug-sop` 不是恢复底座，而是恢复后的 Bug 场景处理流程。

恢复顺序应为：

- 先恢复任务快照
- 再进入 `bug-sop`

### 10.2 与 PIR 的关系

PIR 不负责恢复现场，只负责执行特定阶段。

恢复顺序应为：

- 先恢复任务快照
- 再判断当前处于 `planner / implementer / reviewer` 哪一阶段
- 再进入对应 PIR skill

### 10.3 与本地私有 skills 的关系

若项目存在本地私有 skills，它们可读取恢复快照，但不应替代恢复底座本身。

## 11. 完成标准

一次恢复只有满足以下条件，才可视为成功：

- 新 session 无需回读完整旧聊天
- 能明确当前任务目标
- 能明确当前分支与工作区状态
- 能明确上次停在什么阶段
- 能明确下一步该做什么
- 能明确应进入哪个场景流程或执行阶段

## 12. 禁止事项

- 不要把聊天历史当作唯一恢复依据
- 不要把恢复快照当作正式规范真源
- 不要在非稳定阶段频繁生成低质量快照
- 不要让多个并行任务共享同一份任务快照正文
- 不要跳过通用恢复层，直接假设进入某个场景 skill
- 不要把只在某个场景里成立的信息误写为通用恢复信息

## 13. 项目适配项

当前项目默认约定如下：

- 本地恢复目录：`/.local-work/session-context/`
- 统一入口：`LATEST_PATH.txt`
- 场景化恢复可挂接：
  - `bug-sop`
  - `pir-planner`
  - `pir-implementer`
  - `pir-reviewer`
  - `commit-local`
- 本地恢复机制默认只在当前本地仓库与当前工作区生效
