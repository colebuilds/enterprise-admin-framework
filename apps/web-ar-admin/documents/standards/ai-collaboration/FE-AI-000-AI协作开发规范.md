# AI-协作开发规范

## 元数据

- 目标：建立项目级 AI 协作规范，让 AI 从“临时对话助手”升级为“受控开发协作者”

## 语言约定

本项目当前主要面向中文协作环境。

规则要求：

- 面向项目协作的说明性文本默认使用中文
- 不应输出全英文的任务理解、边界说明、执行计划、风险说明、结果回报、提交说明、文档说明、测试说明或评审说明
- 如需中英混用，应以中文叙述为主体，英文仅用于命令、路径、代码标识符、协议字段、技术术语或精确引用
- 面向内部中国开发团队使用的开发者工具文案、测试说明、测试用例说明、调试提示、内部工程能力说明与面向开发团队的交互提示，默认也应中文优先，不应因其属于“技术工具”或“开发工具”而默认使用英文
- 若当前无法明确判断某项开发者工具、工程化能力或交互提示是否需要国际化，应先确认，不默认使用英文
- 业务多语言文案、i18n 内容、产品页面中的英文、日志原文、报错原文、第三方协议或外部资料原文，不纳入本条中文约束

## 1. 文档定位

本文档用于定义项目中的 AI 协作方式，帮助团队成员理解：

- 为什么不能只靠对话临时约束 AI
- 为什么规则需要沉淀为项目文件
- 为什么需要区分项目规则、技能、文档与工程护栏
- 如何在不同工具中正确落地规则

本文档属于 AI 协作总纲。其下位文档包括但不限于：

- `FE-AI-001-AI变更控制协议.md`
- `FE-AI-002-AI任务边界模板.md`
- `AGENTS.md`
- `skills/*/SKILL.md`

---

## 2. 为什么需要 AI 协作规范

仅通过对话约束 AI，存在以下问题：

1. 规则不持久  
   新会话开始后，临时口头约束可能丢失。

2. 优先级不清楚  
   AI 不一定知道哪条规则是项目级硬规则，哪条只是本次补充说明。

3. 缺少默认行为  
   如果规则没有沉淀为项目文件，AI 不会天然把它当成进入项目后的默认约束。

4. 难以审计与复用  
   规则只存在于聊天记录中，不利于团队协作、项目传承与新成员接手。

## 3. AI 协作的四层体系

### 3.1 全局偏好层

用于定义跨项目稳定不变的协作偏好，例如：

- 默认中文输出
- 默认先计划后执行
- 默认最小必要改动

### 3.2 项目入口规则层

用于定义本项目的硬规则与优先级入口。

典型载体：

- `AGENTS.md`
- 不同工具的项目级规则文件

项目入口规则除了声明必读文件，还应负责：

- 路由文档总纲
- 路由 AI 协作规范
- 路由 Skill
- 声明分支动态规则
- 声明执行授权规则

### 3.3 专项 Skill 层

用于封装高频、可复用的专项任务规则，例如：

- 变更控制
- 1:1 迁移
- 文档治理
- Bugfix 流程
- 标准执行口令驱动的修改流程

当前阶段推荐先将“固定化口令协议”并入：

- `skills/ai-change-control/SKILL.md`

后续若规则进一步复杂，再考虑拆出独立 skill。

### 3.4 文档沉淀层

用于定义完整制度、目录结构、规范分类、决策记录与演进信息。

典型载体：

- `documents/README.md`
- `documents/standards/*`
- `documents/decisions/*`
- `documents/evolution/*`

> 说明：工程护栏（如 CI、Git hook、只读目录）同样重要，但当前阶段先以规则与文档治理为主，后续再补强。

## 4. 工具入口差异（概念层）

不同工具对规则文件的入口机制不同，落地前必须确认：

- 识别的文件名是否正确
- 规则的作用域是全局还是项目级
- skills 是否支持自动发现
- 是否需要显式引用其他规则文件

本规范先统一项目内部抽象，不绑定某一个具体工具。实际落地时，应根据所使用的工具补充对应入口配置说明。

### 4.1 OpenAI / Codex 类工具

重点关注：

- `AGENTS.md`
- 官方支持的 repo-local skills 目录
- 项目级规则与会话内对话指令的区别

### 4.2 Claude 类工具

重点关注：

- `CLAUDE.md`
- skills
- 项目记忆与自动记忆能力的边界

当前项目的推荐落地方式：

- 使用根目录 `CLAUDE.md` 作为 Claude Code 原生入口
- 由 `CLAUDE.md` 轻量承接并路由到 `AGENTS.md`、`documents/` 与正式 `skills/`
- 不在 `CLAUDE.md` 中重复复制整套制度正文，避免双份维护

### 4.3 Cursor 类工具

重点关注：

- Rules
- Skills
- 文件格式、目录位置与当前版本兼容性

### 4.4 项目私有目录与默认自动发现的区别

项目中如果存在自定义目录，例如：

- `.repo-local-skills/`

需要明确区分：

- 工具官方支持的默认目录
- 项目内部私有约定目录

项目私有目录不能默认假设会被所有工具自动加载。  
若要让 AI 稳定命中，必须在 `AGENTS.md` 中显式声明：

- 该目录的用途
- 何时需要检查
- 命中条件
- 读取优先级
- 是否先读取 metadata，再按需读取正文

## 5. 项目内标准落地结构

推荐结构如下：

```text
project-root/
  AGENTS.md

  documents/
    README.md
    standards/
      ai-collaboration/
        FE-AI-000-AI协作开发规范.md
        FE-AI-001-AI变更控制协议.md
        FE-AI-002-AI任务边界模板.md

  skills/
    ai-change-control/
      SKILL.md
```

其中推荐由 `AGENTS.md` 统一路由：

- 文档相关任务 → `documents/README.md`
- AI 协作制度相关任务 → `documents/standards/ai-collaboration/*`
- 修改类任务 → `skills/ai-change-control/SKILL.md`

若项目还存在私有 skill 目录，例如：

```text
.repo-local-skills/
  <skill-name>/
    SKILL.md
```

应注意：

- 该目录属于项目私有约定，不应默认假设所有工具都会自动加载
- 必须在 `AGENTS.md` 中显式声明其作用、触发条件、读取方式与优先级
- 只有工具官方支持的默认目录，才能依赖自动发现

## 6. 每种文件分别负责什么

### 6.1 `AGENTS.md`

负责项目级硬规则、优先级、必读文件、规则路由。

建议至少负责：

- 项目入口规则
- 文档规范路由
- Skill 路由
- 分支动态规则
- 执行授权规则
- 复杂任务计划规则

### 6.2 `documents/README.md`

负责项目文档体系总纲，说明文档怎么分层、怎么归档、怎么更新。

### 6.3 `FE-AI-001-AI变更控制协议.md`

负责 AI 在修改文件、执行命令时的过程控制规则。

### 6.4 `FE-AI-002-AI任务边界模板.md`

负责单次任务的边界确认。

### 6.5 `skills/*/SKILL.md`

负责将高频任务沉淀为可复用执行能力。

### 6.6 私有 skill 目录

例如：

- `.repo-local-skills/*/SKILL.md`

用于承载项目个性化、工具未必原生识别的 skill。  
这类目录不是默认入口，必须通过 `AGENTS.md` 做声明和路由。

推荐规则：

- 若目录存在，可先轻量枚举 `./.repo-local-skills/*/SKILL.md` 的 metadata，建立可用私有 skill 列表
- metadata 至少建议包含 `name`、`description`，可按需补充 `keywords`
- 不默认全文读取全部私有 skill
- 项目标准协作口令属于流程控制信号，不参与私有 skill 自动匹配
- 当前任务明显命中某个私有 skill 的名称 / 描述 / 关键词范围时，应优先读取对应 `SKILL.md`
- 用户点名某个私有 skill 时，作为补充兜底方式读取对应 `SKILL.md`
- 私有 skill 只作补充，优先级低于 `AGENTS.md`、`documents/README.md`、`documents/standards/ai-collaboration/*` 与正式 `skills/*/SKILL.md`

最小模板示例：

```text
.repo-local-skills/
  my-private-skill/
    SKILL.md
```

`SKILL.md` 最小示例：

```md
---
name: my-private-skill
description: Briefly describe when this private skill should be used.
keywords: keyword-a, keyword-b, keyword-c
---

# My Private Skill

## 目标

- 说明这个私有 skill 解决什么问题

## 适用场景

- 任务命中哪些关键词、目录或工作类型时使用

## 规则

- 需要额外遵守的补充细则
- 不得覆盖项目硬规则和正式规范
```

使用示例：

- 若任务描述中出现“任务规划”“执行方案”“步骤拆解”“风险评估”等自然语言需求，可优先匹配并读取相关私有 skill
- 若任务描述与某个私有 skill 的名称、描述或关键词明显匹配，可读取对应 `SKILL.md`
- 用户明确说“使用 `my-private-skill`”时，作为补充兜底方式读取 `.repo-local-skills/my-private-skill/SKILL.md`
- 若未点名且没有明显命中，不默认全文读取所有私有 skill

说明：

- `先分析`、`生成边界`、`生成计划`、`开始执行`、`停止执行`、`提交结果` 等项目标准口令，应先按流程控制规则处理
- 这些标准口令不应直接触发私有 skill 自动匹配

## 6.7 标准执行口令协议

为了降低歧义，项目内建议固定使用以下口令：

- `先分析`
- `生成边界`
- `生成计划`
- `开始执行`
- `停止执行`
- `提交结果`

其中：

- `开始执行`：唯一允许进入代码修改阶段的口令
- `停止执行`：立即停止当前执行
- `继续`、`继续执行`、`按方案做`、`处理一下` 等表达，不构成代码修改授权

推荐硬规则：

- 未收到 `开始执行` 前，不得修改代码
- 收到 `停止执行` 后，必须立刻停止并汇报当前状态

## 6.8 分支动态规则

AI 在分析阶段应先读取当前分支名，再决定是否启用项目分支规则。

推荐使用只读方式获取当前分支，例如：

- `git rev-parse --abbrev-ref HEAD`

当前项目已确认的规则：

### nova 分支规则

当当前分支名匹配 `nova-*`，或分支路径段中包含 `*/nova-*` 时，以下路径视为受保护路径，默认禁止修改：

- `/src/**`
- `/entrance/**`
- `/index.html`
- `/.env`
- `/.env.*`

说明：

- 这些路径属于 AI 层的“软禁止”范围
- 应同时通过 Git hook / CI 做工程层硬拦截
- 若计划或执行中命中受保护路径，必须停止并重新确认

### 非 nova 分支

若当前分支既不匹配 `nova-*`，且分支路径段中也不包含 `*/nova-*`，则不自动启用上述受保护路径规则。  
但仍需遵守本次任务边界、文档规范与执行 Skill。

## 7. 一次完整任务的标准流程

### 步骤 1：输入任务目标

由人提供最小必要输入，例如：

- 任务目标
- 本次不包含
- 允许修改
- 禁止修改
- 允许命令
- 禁止命令

### 步骤 2：AI 生成任务边界草案

AI 根据项目规则与模板生成本次任务边界。

### 步骤 3：人工确认边界

由人确认：

- 文件范围是否正确
- 命令范围是否正确
- 是否需要新增文件
- 是否存在高风险区域

### 步骤 4：AI 输出执行计划

AI 列出：

- 计划修改文件
- 计划执行命令
- 风险与影响面

### 步骤 5：人工批准执行

未获许可前，不得落盘修改。  
建议以固定口令 `开始执行` 作为唯一代码修改授权。

### 步骤 6：AI 执行与回报

执行后回报：

- 实际改动
- 实际命令
- 偏差审计
- 验证建议

### 步骤 7：文档沉淀与规则回写

必要时更新：

- `documents/README.md`
- `documents/standards/*`
- `AGENTS.md`
- `skills/*/SKILL.md`

## 7.1 复杂任务的计划规则

当任务满足以下任一条件时，建议视为复杂任务：

- 涉及多个模块
- 涉及跨目录修改
- 涉及迁移或重构
- 预计改动文件较多
- 涉及共享层 / 配置层 / 基础设施层
- 需要多人协作
- 需要分阶段推进

复杂任务建议遵循：

1. 先生成开发计划
2. 计划至少包含：
   - 目标
   - 本次不包含
   - 影响范围
   - 分步骤执行计划
   - 风险与影响面
   - 验证方式
3. 计划未确认前，不得改代码
4. 执行中计划变化时，必须重新确认

当前阶段可先直接在对话中输出计划；后续复杂度升高，再考虑独立计划文档。

## 8. 规则为什么会失效

常见原因包括：

1. 文件名不符合工具要求
2. 文件位置不在工具识别范围内
3. 项目级规则没有在入口文件中声明
4. 规则只存在于对话中，没有沉淀为项目文件
5. 规则过长、过散，缺少入口摘要
6. skill 与项目硬规则冲突，但未定义优先级
7. 自定义目录被误当成默认自动加载目录
8. 工具版本变化导致配置方式变化
9. 分支动态规则未声明，AI 未先读取当前分支
10. 执行授权口令不固定，导致“分析”和“改代码”边界失控

## 9. 全局规则与项目规则如何分层

### 9.1 全局规则

只放跨项目长期稳定偏好，例如：

- 默认中文
- 默认先计划后执行
- 默认最小改动
- 默认不跑高风险命令

### 9.2 项目规则

只放本项目特有约束，例如：

- 目录结构
- 文档归档方式
- 老项目只读
- 迁移必须 1:1 一致
- 项目私有 skill 目录约定
- 分支动态受保护路径规则
- 执行授权口令协议

### 9.3 单次任务规则

只放当前任务边界，例如：

- 本次允许修改哪些文件
- 本次禁止执行哪些命令
- 本次是否允许新增文件

## 10. 新手上手清单

第一次在项目中接入 AI 协作，至少完成以下事项：

1. 确认当前使用的 AI 工具与其规则入口
2. 创建项目根入口规则文件
3. 创建 `documents/README.md` 并明确文档总纲
4. 创建 `FE-AI-001-AI变更控制协议.md`
5. 创建 `FE-AI-002-AI任务边界模板.md`
6. 创建至少一个基础 skill：
   - `skills/ai-change-control/SKILL.md`
7. 若存在私有 skill 目录，在 `AGENTS.md` 中明确声明
8. 在 `AGENTS.md` 中声明规则优先级、分支动态规则与必读文件
9. 固定执行授权口令，明确只有 `开始执行` 才能改代码
10. 对 `nova-*` 分支配置受保护路径规则
11. 用一次真实任务验证规则是否生效
12. 发现失效点后及时补文档，不要只在对话中口头修补

## 12. 最终原则

AI 协作的目标，不是让 AI 更能聊天，而是让 AI 成为项目中的受控执行者。

正确的协作方式不是：

- 只在对话里临时补规则
- 每次重复提醒
- 靠记忆维持约束
- 默认假设所有自定义目录都会自动加载

而是：

- 用入口规则定义项目默认行为
- 用 skill 固化高频工作流
- 用文档沉淀详细制度
- 用项目文件而不是聊天记录来保存规则
- 对私有目录、自定义入口和工具差异做显式声明
- 对分支动态规则、受保护路径与执行授权做明确约定
