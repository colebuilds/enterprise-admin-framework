# Harness 版本号与迭代日志规范

## 1. 目标

定义 Harness 系统能力演进时的版本号与迭代日志规则，保证系统能力变化可以被：

- 稳定识别
- 稳定追踪
- 稳定回溯
- 稳定由后续 skill 承接

## 2. 适用范围

本规范适用于 Harness 系统本身的能力变化，包括但不限于：

- 任务链路能力
- 任务对象模型
- runtime 控制面与模板
- skill 协议与执行约束
- 自动执行、验证、收口相关能力

本规范不直接适用于以下变化：

- 纯测试补充
- 纯文档润色
- 纯说明性调整
- 不改变 Harness 对外能力的内部表述收敛

## 3. 核心原则

### 3.1 只有系统能力变化才触发版本号判断

Harness 版本号用于表达系统能力变化，而不是所有仓库改动。

应触发版本号判断的变化通常包括：

- 新增系统能力
- 扩展现有系统能力
- 调整已有能力的默认行为
- 引入兼容性变化或破坏性变化

### 3.2 纯测试、纯文档、纯说明性变化不默认 bump 版本

以下变化默认不应单独触发版本号更新：

- 仅新增或调整测试
- 仅补充文档说明
- 仅修正文案、措辞或注释
- 仅让既有规则表述更清晰，但未改变系统能力

### 3.3 版本号与迭代日志必须联动

当 Harness 系统能力发生版本级变化，并决定 bump 版本时，必须同步补充迭代日志。

不得出现：

- 版本号已变化，但没有迭代日志
- 迭代日志记录了版本级变化，但版本号未更新

### 3.4 规则不能只停留在文档层

版本号与迭代日志规则属于执行约束，后续必须由对应 skill 承接。

文档层负责定义制度；skill 层负责在任务执行、提交前检查与收口阶段命中这些制度。

## 4. 何时应 bump Harness 版本

以下情况应触发版本号更新判断：

### 4.1 新增 Harness 功能能力

例如：

- 新增正式任务链路能力
- 新增自动执行能力
- 新增 runtime 控制面能力
- 新增任务对象层级
- 新增自动验证或自动收口能力

### 4.2 扩展已有 Harness 功能能力

例如：

- 已有任务流增加新的前置判断
- runtime 从记录器升级到控制面
- skill 从单任务口径升级到三层任务对象口径
- 自动执行适用范围或默认行为发生实际变化

### 4.3 改变已有 Harness 能力行为

例如：

- 执行入口分流规则改变
- 提交门槛改变
- 写入闭环规则改变
- 总任务收口规则改变

### 4.4 发生兼容性变化或破坏性变化

例如：

- runtime 模板字段变化导致旧写法失效
- skill 协议变化导致旧流程不再成立
- 任务对象模型变化导致旧运行时记录无法直接复用

## 5. 何时不应 bump Harness 版本

以下情况默认不应单独 bump 版本：

- 只新增测试
- 只补文档
- 只补样例
- 只修复说明错误
- 只整理目录而不改变系统能力

若一次任务同时包含：

- 系统能力变化
- 以及纯测试 / 纯文档补充

则应以系统能力变化为准进行版本号判断。

## 6. 版本级别建议

当前阶段先采用最小三档语义：

### 6.1 Patch

适用于：

- 不改变现有能力边界的缺陷修复
- 不改变默认行为的小型补强
- 兼容旧流程的最小能力完善

### 6.2 Minor

适用于：

- 新增功能能力
- 扩展已有能力
- 在保持总体兼容前提下增加新的系统行为

### 6.3 Major

适用于：

- 破坏性变化
- 核心对象模型或主流程重写
- 旧 skill / runtime / 模板约定大面积失效

当前阶段不强制定义更细的预发布语义，也不在本规范中绑定发布流水线。

## 7. 迭代日志规则

### 7.1 何时必须补迭代日志

当发生以下任一情况时，必须补迭代日志：

- bump 了 Harness 版本
- 新增了系统能力
- 改变了核心流程或核心对象模型
- 发生了兼容性变化或破坏性变化

### 7.2 迭代日志至少包含什么

迭代日志至少应包含：

- 版本号
- 日期
- 本次变化摘要
- `Added / Changed / Removed` 分类
- 影响范围
- 是否存在兼容性影响
- 是否需要迁移或注意事项
- 关联规范、skill 或 runtime 变更

### 7.3 迭代日志的目标

迭代日志不是简单的提交记录摘要，而是用于回答：

- Harness 这次到底新增了什么能力
- 哪些能力行为发生了变化
- 后续执行时需要注意什么
- 旧规则是否仍可用

## 8. 版本号与迭代日志联动要求

推荐联动顺序如下：

1. 先判断本次是否属于系统能力变化
2. 若属于，再判断是否需要 bump 版本
3. 若决定 bump，则同步补迭代日志
4. 提交前检查版本号与迭代日志是否一致

不得出现只改其中一项的半闭环状态。

## 9. 后续 skill 承接要求

后续至少需要有对应 skill 承接以下判断：

- 当前任务是否属于 Harness 系统能力变化
- 当前任务是否应 bump 版本
- 当前任务是否必须补迭代日志
- 提交前是否已满足版本号与迭代日志联动条件

也就是说，后续 skill 至少要能在执行层拦住以下错误：

- 应 bump 但未 bump
- 已 bump 但未写迭代日志
- 纯文档任务错误 bump 版本
- 版本号与日志记录不一致

## 10. 最小结论

当前阶段可收敛为以下默认规则：

- 只有 Harness 系统能力变化才默认进入版本号判断
- 纯测试、纯文档、纯说明性调整不默认 bump 版本
- 发生版本级变化时，必须同步补迭代日志
- 这套规则后续必须进入对应 skill，不能只停留在文档层
