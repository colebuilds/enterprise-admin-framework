# 1. 目标

定义 AI 在复杂开发任务中的标准执行方式：可拆解、可连续执行、可中断恢复、可审计。

# 2. 适用范围

- 所有复杂任务（3+步骤，跨文件/跨模块/跨阶段）
- 需求开发、文档治理、重构迁移、联调治理

# 3. 核心模型

- 阶段（Phase）：里程碑级目标
- 执行单元（EU）：最小可执行单元
- 验证（Verification）：每个 EU 必须有验证证据
- 阶段提交（Phase Commit）：每阶段一次提交

# 4. 执行纪律（强制）

1. 任务按“阶段 -> EU”推进。
2. 同一时刻仅允许一个 EU 为 in_progress。
3. EU 验证未通过，不得进入下一 EU。
4. 阶段验收未通过，不得进入下一阶段。
5. 每阶段完成后进行一次阶段提交。
6. 每步都要可恢复（记录当前阶段/当前EU/验证结果/下一步）。

# 5. 与长任务看板（todo）映射

- 一个 EU 对应一个 todo 项。
- todo.id 建议：`p<phase>-eu<index>`（例：`p2-eu3`）。
- 状态映射：
  - pending：未开始
  - in_progress：进行中
  - completed：验证通过
  - cancelled：终止或回滚弃用
- 阶段切换规则：
  - 本阶段 EU 全部 completed 才能执行阶段验收与阶段提交。
  - 验收失败时新增修复 EU，继续本阶段。

# 6. 多-Agent并行规则（可选）

1. 仅允许同阶段内并行独立子任务。
2. 同文件/同核心状态任务必须串行。
3. 并行上限 3 个子任务。
4. 并行后必须统一验证再合并结论。
5. 阶段提交仍保持“一阶段一次”。

# 7. 质量门禁（每个 EU 必填）

- 代码检查：lint/typecheck/test
- 功能检查：需求口径一致
- 数据一致性：筛选/统计/列表一致
- 可观测检查：关键指标可见
- 文件清单：本次变更文件列表

# 8. 验证证据模板（每个 EU 必填）

- 执行命令：
- 结果摘要：
- 证据路径（日志/截图）：
- 结论：pass/fail
- 失败处置：回滚/修复后二次验证（附证据）

# 9. 风险与阻塞登记（每阶段必填）

- 已知风险：
- 当前阻塞：
- 升级触发条件：
- 风险等级：high/medium/low
- 影响范围：当前EU/当前阶段/全计划

# 10. 提交规范（阶段提交）

- 格式：`<type>(<scope>): phase Px - <summary>`
- 提交前必须：
  - 记录阶段验证证据
  - 更新执行状态（阶段/EU/验证/下一步）
  - 输出文件清单

# 11. 开发启动前检查

1. 方案评审通过
2. 依赖策略确认
3. 统计与验收口径冻结
4. 阶段边界确认
5. Git 前置完成（同步 main 后切分支）
