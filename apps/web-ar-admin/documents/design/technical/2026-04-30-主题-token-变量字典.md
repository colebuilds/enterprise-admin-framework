# 2. 命名分层总览

## 2.1 最终目标前缀

| 前缀              | 层级       | 用途                         |
| ----------------- | ---------- | ---------------------------- |
| `--color-*`       | 原子 token | 原始颜色值或颜色阶梯         |
| `--space-*`       | 原子 token | 间距刻度                     |
| `--font-size-*`   | 原子 token | 字号刻度                     |
| `--line-height-*` | 原子 token | 行高刻度                     |
| `--radius-*`      | 原子 token | 圆角基础值                   |
| `--shadow-*`      | 原子 token | 阴影基础值                   |
| `--z-*`           | 原子 token | 层级刻度                     |
| `--motion-*`      | 原子 token | 动效时长与 easing            |
| `--ui-*`          | 语义 token | 页面、组件、布局、交互语义层 |

## 2.2 迁移期前缀

| 前缀          | 类型         | 说明                                    |
| ------------- | ------------ | --------------------------------------- |
| `--primary-*` | legacy alias | 旧变量名，迁移期保留                    |
| `--n-*`       | 第三方变量   | Naive UI 内部变量，不属于本字典定义范围 |

## 2.3 非目标前缀

| 前缀      | 处理策略                                                     |
| --------- | ------------------------------------------------------------ |
| `--app-*` | 不纳入最终目标体系；若后续实施时无消费面，可直接由新体系替换 |

# 3. 原子 Token 字典

## 3.1 品牌色阶（动态）

> 这组 token 由 `themeColor` 派生生成。  
> 其中 `500` 为用户选择的品牌主色基准。

| Token               | 来源规则                   | 用途                  |
| ------------------- | -------------------------- | --------------------- |
| `--color-brand-50`  | `themeColor` 的最浅柔和阶  | 轻背景、浅强调底      |
| `--color-brand-100` | `themeColor` 的浅阶        | soft bg、弱强调底     |
| `--color-brand-200` | `themeColor` 的次浅阶      | hover light surface   |
| `--color-brand-300` | `themeColor` 的中浅阶      | 辅助描边、弱 hover    |
| `--color-brand-400` | `themeColor` 的 hover 阶   | hover、强调次级文字   |
| `--color-brand-500` | `themeColor` 本身          | 品牌主色              |
| `--color-brand-600` | `themeColor` 的 pressed 阶 | pressed、激活深色边框 |
| `--color-brand-700` | `themeColor` 的深阶        | 深层品牌表达          |
| `--color-brand-800` | `themeColor` 的更深阶      | 特殊深背景或图形      |
| `--color-brand-900` | `themeColor` 的最深阶      | 极深品牌表达          |

# 7. Tailwind Adapter 口径

## 7.1 冻结原则

- Tailwind 4 不单独持有主题状态。
- Tailwind 只消费最终导出的 CSS vars。
- 布局、间距、排版仍优先走 utility class。
- 颜色、圆角、阴影、层级要落回 token。

## 7.2 推荐消费范围

| 类别    | 推荐来源                       |
| ------- | ------------------------------ |
| 颜色    | `--ui-*`                       |
| 间距    | `--space-*`                    |
| 圆角    | `--radius-*` / `--ui-radius-*` |
| 阴影    | `--shadow-*` / `--ui-shadow-*` |
| z-index | `--z-*`                        |
| motion  | `--motion-*`                   |

# 8. 页面与组件消费规则

## 8.1 新代码规则

- 页面与组件优先消费 `--ui-*`
- 基础视觉工具层可在必要时消费 `--color-*`、`--space-*`、`--radius-*`
- 新代码禁止新增 `--primary-*`
- 新代码禁止新增 `--app-*`

## 8.2 旧代码迁移规则

- 旧代码短期继续跑在 `--primary-*` alias 上
- 在公共组件迁移完成前，不强行一次性迁业务页面
- 直接依赖 `--n-*` 的 Naive 结构样式可以保留，但不应继续扩散

# 9. 变量字典的维护规则

## 9.1 新增 token 的门禁

新增 token 前必须回答：

1. 这是原子 token 还是语义 token？
2. 是否已有现有 token 可复用？
3. 是否会影响 light / dark 双主题映射？
4. 是否需要增加 legacy alias？

## 9.2 禁止项

- 不允许为单个页面发明私有全局 token
- 不允许把 `--primary-*` 当新体系命名继续扩散
- 不允许新增 `--app-*` 作为长期命名
- 不允许业务代码直接依赖 Naive token 来替代全局语义 token

# 10. V1 收敛建议

## 10.1 当前规模判断

- 当前文档的全局 CSS token 总量已经接近完整设计系统规模：
  - 原子 token：`87`
  - 语义 token：`75`
  - legacy alias：`7`
- 对当前后台项目而言，原子 token 数量基本合理；偏重的部分主要在语义层。
- 当前语义层同时承载了三类职责：
  - 全局主题语义
  - 组件级语义
  - Naive adapter 细粒度语义
- 因此，v1 不建议直接把当前 `75` 个语义 token 全量冻结为“长期全局公开接口”。

## 10.2 v1 必须保留的全局 Token

### 原子层

- 原子层默认整体保留：
  - `--color-brand-*`
  - `--color-neutral-*`
  - `--color-success-*`
  - `--color-warning-*`
  - `--color-error-*`
  - `--space-*`
  - `--font-size-*`
  - `--line-height-*`
  - `--radius-*`
  - `--shadow-*`
  - `--z-*`
  - `--motion-*`
- 说明：
  - 原子层是底座，不是当前过度设计的主要来源。
  - 若实施阶段确认 `info` 在视觉上完全等同品牌色，可将 `--color-info-*` 转为内部映射，不作为 v1 必保原子 token。

### 语义层

- v1 建议优先冻结以下全局语义 token，控制在 `40-45` 个左右：
  - 强调与状态： `--ui-accent-solid`、`--ui-accent-hover`、`--ui-accent-pressed`、`--ui-accent-soft`、`--ui-accent-soft-hover`、`--ui-focus-ring`、`--ui-success-solid`、`--ui-success-soft`、`--ui-warning-solid`、`--ui-warning-soft`、`--ui-danger-solid`、`--ui-danger-soft`
  - 背景与表面： `--ui-bg-page`、`--ui-bg-page-elevated`、`--ui-bg-card`、`--ui-bg-card-elevated`、`--ui-bg-modal`、`--ui-bg-popover`、`--ui-bg-overlay`
  - 文本： `--ui-text-primary`、`--ui-text-secondary`、`--ui-text-tertiary`、`--ui-text-disabled`、`--ui-text-inverse`、`--ui-text-link`、`--ui-text-on-accent`
  - 边框： `--ui-border-default`、`--ui-border-strong`、`--ui-border-focus`、`--ui-divider`
  - 填充与交互： `--ui-fill-hover`、`--ui-fill-active`、`--ui-fill-selected`、`--ui-fill-disabled`
  - 控件基础： `--ui-control-bg`、`--ui-control-border`、`--ui-control-border-focus`、`--ui-control-placeholder`、`--ui-control-text`
  - 语义圆角与阴影： `--ui-radius-control`、`--ui-radius-card`、`--ui-radius-modal`、`--ui-shadow-card`、`--ui-shadow-popover`、`--ui-shadow-modal`

## 10.3 建议下沉到 Adapter / 组件层的 Token

- 以下 token 更适合作为 Naive adapter、布局 adapter 或组件私有派生层，不建议在 v1 就冻结成长期全局公开 token：
  - 导航与菜单： `--ui-nav-sider-bg`、`--ui-nav-header-bg`、`--ui-nav-border`、`--ui-menu-item-text`、`--ui-menu-item-text-active`、`--ui-menu-item-icon`、`--ui-menu-item-icon-active`、`--ui-menu-item-bg-hover`、`--ui-menu-item-bg-active`、`--ui-menu-arrow`
  - 表格： `--ui-table-header-bg`、`--ui-table-row-bg`、`--ui-table-row-hover-bg`、`--ui-table-header-text`、`--ui-table-body-text`
  - 控件细项： `--ui-control-bg-focus`、`--ui-control-border-hover`、`--ui-control-caret`
  - 布局表面： `--ui-bg-table`、`--ui-bg-table-header`、`--ui-bg-footer`
- 原则：
  - 这些 token 不是“不需要”，而是更适合作为第二层派生接口。
  - 应由 adapter 从全局语义 token 推导，而不是先占用全局命名预算。

## 10.4 建议合并或延后定义的 Token

- 建议合并重复语义：
  - `--ui-bg-sider` 与 `--ui-nav-sider-bg`
  - `--ui-bg-header` 与 `--ui-nav-header-bg`
- 建议延后定义，待真实消费场景稳定后再决定：
  - `--ui-bg-action`
  - `--ui-accent-text`
  - `--ui-text-quaternary`
  - `--ui-radius-pill`
- 建议转为内部映射或 v2 再明确：
  - `--color-info-*`
  - `--ui-info-solid`
  - `--ui-info-soft`
  - `--ui-border-accent`
- 说明：
  - 如果 `info` 在本项目里长期只是“品牌色的另一种说法”，则不必在 v1 为其保留一整组全局 token。
  - 如果 `border-accent` 当前仅等价于品牌边框，也可先由 `--ui-accent-solid` 或 `--ui-border-focus` 派生。

## 10.5 v1 收敛后的目标形态

- v1 建议把“长期全局公开语义 token”控制在 `40-45` 个左右。
- 菜单、表格、Layout、Naive 专属细粒度映射优先下沉到 adapter。
- 全局 token 先服务主题一致性和页面通用消费，不追求一步到位覆盖所有组件细节。
