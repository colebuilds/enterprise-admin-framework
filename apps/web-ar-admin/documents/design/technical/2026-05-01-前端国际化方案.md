# 前端国际化方案

## 1. 文档定位

本文档用于从当前仓库真实实现反推并固化“这套前端国际化方案到底是什么”。

它回答的是：

- 当前系统支持哪些语言，资源怎么组织
- 为什么采用 `locale + 后端字典 + fallback` 三层模型
- 为什么采用 namespace 级懒加载，而不是全量加载
- key 命名、文档归档、模块接入 SOP 分别处于什么层级
- 当前方案的能力边界与已知缺口是什么

它不是模块接入执行手册，也不是逐条硬规则清单。执行细节看：

- `documents/guides/development/国际化接入与运行时机制说明.md`
- `documents/standards/coding/FE-STD-008-国际化与字典边界规范.md`
- `documents/guides/development/模块内多语言接入AI提示词模板.md`
- `/.repo-local-skills/module-i18n-onboarding-sop/SKILL.md`

---

## 2. 方案目标与非目标

### 2.1 目标

当前方案的目标是：

- 让商户后台在不刷新页面的前提下切换前端语言
- 让不同业务域按 namespace 拆分 locale 资源，避免首屏全量加载
- 让前端静态文案、后端业务字典、兜底展示各归其位
- 让新增模块能够沿用一套可重复执行的接入路径

### 2.2 非目标

当前方案不负责：

- 设计一套新的全局翻译平台
- 自动治理所有后端返回文本
- 自动刷新所有已缓存字典和已查询列表数据
- 取代后端字典平台对业务枚举值的统一维护

一句话定义：

**这套方案不是“多语言治理平台”，而是“当前前端项目在既有后端配合下的国际化运行与接入方案”。**

---

## 3. 当前支持范围

### 3.1 语言现状

当前运行时支持语言以 `src/i18n/index.ts` 中的 `SUPPORT_LOCALES` 为准，现状为：

- `en`
- `zh`
- `vi`
- `id`

默认语言为：

- `zh`

当前 `pnpm i18n:check` 的静态校验基线仍是：

- `zh`
- `en`

因此，当前语种策略应表述为：

- `en/zh` 是当前基线维护对与脚本校验范围
- `vi/id` 是当前运行时已启用语种
- 若目标 namespace 已存在 `vi/id` 资源，新增 key 时应同步补齐，不能只改 `zh/en`

### 3.2 namespace 现状

截至 `2026-05-01`，`src/i18n/locales/` 下共有 4 套 locale、17 个 namespace，四套目录文件集合对齐。

当前 namespace 包括：

- `common`
- `menu`
- `components`
- `dashboard`
- `account`
- `agent`
- `finance`
- `game`
- `operations`
- `erde`
- `member`
- `tenant`
- `report`
- `analytics`
- `hubConfig`
- `productNav`
- `system`

资源组织约定是：

- 一套语言一个目录：`src/i18n/locales/<locale>/`
- 一个 JSON 文件一个 namespace：如 `finance.json`
- 文件内根对象名与文件名保持一致：如 `finance.json` 的根对象必须是 `finance`

## 4. 三层文案来源模型

当前系统的国际化不是单一 `t()` 模式，而是三层模型。

| 层级 | 真实来源 | 适用内容 | 说明 |
| --- | --- | --- | --- |
| 本地 locale | `src/i18n/locales/<locale>/*.json` | 页面标题、按钮、表头、提示语、校验文案、说明文字 | 前端静态可控文案 |
| 后端字典 | `common / group / platform / v1 / dynamic` | 状态 label、下拉选项、国家/语言/币种、中心化业务值 | 由后端统一维护和下发 |
| fallback | 本地兜底字符串或原始值 | 过渡展示、兼容旧数据、字典缺失兜底 | 不是正式文案来源 |

这套分层的核心价值是：

- 静态 UI 文案可随前端版本稳定演进
- 中心化业务枚举值不在前端重复维护第二份“正式来源”
- 页面能区分“该本地化的 UI 文案”和“该走后端配置的业务值”

---

## 5. namespace 懒加载是方案特征，不是表面写法

### 5.1 当前不是全量加载

当前实现不是启动时一次性加载所有 locale 文件，而是：

1. 启动阶段只预热 `common`、`menu`、`components`
2. 路由进入前按页面所需 namespace 懒加载业务文案
3. 切语言时只回放“当前已经使用过的 namespace”

### 5.2 实际收益

按当前资源体量统计，单个语种全量 locale 资源大约在 `675KB` 到 `768KB`，而启动阶段预热的 `common + menu + components` 只有 `47KB` 到 `56KB`，约占全量资源的 `7%`。

这意味着：

- 首屏不会被 `finance`、`operations` 这类大业务域文案拖重
- 首次进入某业务域前，不需要为未访问页面提前加载整包文案
- 语言切换时只需要补目标语种下“已经使用过的 namespace”

### 5.3 方案含义

因此，namespace 懒加载应被视为当前方案的明确技术特征：

- 不是所有 locale 全量内存常驻
- 不是每个页面都能假定所有业务 key 已经就位
- 路由与 namespace 声明是运行时正确性的组成部分

## 6. 运行时装载策略

当前方案的运行时装载链路可以概括为：

**首屏预热 + 路由懒加载 + 已使用集合回放**

### 6.1 首屏预热

`src/main.ts` 在 `createApp` 前调用 `setupI18n()`。

`setupI18n()` 会：

- 读取当前语言
- 预加载 `BOOT_NAMESPACES`
- 注入 Naive UI locale 消息

### 6.2 路由懒加载

`src/router/guards.ts` 进入页面前会：

- 优先读取 `meta.i18nNamespace`
- 若未声明，则按路径前缀推导 namespace
- `await loadNamespaces(locale, namespaces)` 后再继续路由

### 6.3 切语言回放

`switchLocale(locale)` 不做整页刷新，而是：

- 收集当前已加载 namespace
- 至少补齐 `BOOT_NAMESPACES`
- 在目标语种下重新加载这些 namespace
- 切换 `i18n locale`
- 更新 `<html lang>`

它解决的是：

- 前端本地 locale 文案的无刷新切换

它不自动解决的是：

- 后端字典重拉
- 页面已请求数据重查
- 后端直接返回展示文案的同步切换

## 7. 请求语言与后端协作

当前方案不是前端局部自嗨，它和请求层联动：

- 请求层会把当前语言写入请求体 `language`
- 切语言后，后续请求天然会带上新语言
- 已登录时切语言还会调用后端的 `ChangeLanguage` 同步偏好

这意味着当前方案依赖一条关键前提：

- **前端 locale 负责静态 UI 文案**
- **后端按 `language` 返回的字典和业务文本，负责后端侧的多语言内容**

因此，不能把“所有展示文本都塞进本地 locale”理解成当前方案，也不能把已有后端字典再复制一份到 locale 里当正式来源。

## 8. key 命名与 namespace 设计原则

### 8.1 命名原则

key 命名必须表达业务语义和归属层次，而不是表达“这段文本出现在页面第几个位置”。

推荐结构：

- 跨模块通用：`common.*`
- 模块级共享：`finance.withdrawOrder.*`
- 页面专属：`finance.withdrawOrder.dashboard.*`

不推荐：

- `text1`
- `btn1`
- `cardA`
- 仅按页面视觉位置命名的 key

### 8.2 复用优先级

新增文案时按以下顺序判断：

1. 当前模块是否已有同语义 key
2. 当前业务域是否已有同语义 key
3. 当前页面是否真的是独有语义

### 8.3 现状问题与治理方向

当前 `common.json` 已存在历史命名混杂问题，例如：

- 同时存在 `create_time` 和 `createTime`
- 同时存在 `delete_confirm` 和 `deleteConfirm`
- 同时存在 `action_success` 和 `operation_success`

这说明现状不是“命名天然整洁”，而是更需要把新增规则文档化：

- 新增 key 默认使用 camelCase
- 不再继续扩散新的 snake_case
- 不为同一语义制造多套别名

## 10. 当前已知限制与不合理点

这套方案已经成型，但并不完美。以下问题必须作为“现状”被记录，而不是被误写成已解决能力。

### 10.1 `switchLocale()` 不会自动刷新后端字典缓存

当前切语言只负责前端 locale namespace 重载，不会自动重拉：

- `common` 字典
- `group` 字典
- `platform` 字典
- `v1` 字典
- `dynamic` 字典

因此，字典 label 是否跟随变化，取决于页面是否在切语言后主动重查。

### 10.2 切语言不会立即刷新页面标题

当前 `document.title` 只在路由 `afterEach` 中更新。若用户留在当前页直接切语言而不跳路由，标题可能保持旧语言。

### 10.3 首次登录后的 `v1` 字典加载链路不一致

应用启动时若已有 token，会加载 `v1Dictionary`；但从登录页新登录成功后的当前链路并不会补拉 `v1Dictionary`，导致首次登录后的运行态与刷新后的运行态不完全一致。

### 10.4 `i18n:check` 不是完整校验

当前脚本只覆盖：

- `zh/en`
- 静态 key 调用

它不覆盖：

- `vi/id`
- 动态拼接 key
- 运行时生成 key
- 字典与 fallback 的来源判断

### 10.5 业务 namespace 的默认语言 fallback 不能被误解

当前默认语言 `zh` 的 `BOOT_NAMESPACES` 会在启动阶段就位，但业务 namespace 是否已经在默认语言下加载，取决于用户是否访问过对应业务域。因此，不能把 `fallbackLocale = zh` 误解成“任何业务 namespace 都天然有完整默认语言兜底”。

## 11. 这篇方案文档之后怎么用

后续若有人问“这套系统的多语言方案是什么”，应先看本文，再决定下一层入口：

- 需要接具体模块：看 `模块内多语言接入AI提示词模板.md`
- 需要看运行时和接入方式：看 `国际化接入与运行时机制说明.md`
- 需要裁定 key、字典、fallback 边界：看 `FE-STD-008-国际化与字典边界规范.md`

这样，`module-i18n-onboarding-sop` 才不是悬空的执行壳，而是建立在可追溯的方案、规则和实现文档之上。
