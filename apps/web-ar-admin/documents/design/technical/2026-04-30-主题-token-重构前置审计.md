# 1. 本轮改造目标（已冻结方向）

## 1.1 目标

- 主题系统由统一的全局 theme state 与 token layer 驱动。
- 主题配置面板直接消费全局 token 定义或 token registry，而不是散落的硬编码色板。
- 全局 token 不只覆盖颜色，还覆盖：
  - `color`
  - `bg`
  - `text`
  - `border`
  - `radius`
  - `space`
  - `shadow`
  - `z-index`
  - `motion`
- Naive UI、Tailwind、品牌层通过 adapter 层统一消费同一套 token。
- `App.vue` 中散落的主题运行时逻辑需要收敛到独立的主题控制层。

## 1.2 明确约束

- 不修改 Naive UI 自身的 `--n-*` 命名。
- 允许新增规范化的全局 token 命名。
- 旧 `--primary-*` 不再新增；相关 bridge 已在当前代码中退役。
- 主题主状态不能再同时由多套真相源驱动。
- VueUse 可以用于主题运行时 hook，但不能引入第二套主题真相源或第二套持久化链路。

## 1.3 非目标

- 本轮不追求一步删光所有旧样式变量。
- 本轮不要求一次性清理所有页面内硬编码视觉值。
- 本轮不重写 Naive UI 组件体系。
- 本轮不做与主题无关的页面视觉重构。

# 2. 当前系统已确认现状

## 2.1 主题真相源并未收敛

| 主题能力 | 当前来源 | 现状问题 |
| --- | --- | --- |
| 运行时主色 | `src/store/modules/designSetting.ts` 中的 `designStore.appTheme` | 与 `appearance.themeColor` 并存，形成双真相源 |
| 外观配置 | `src/store/modules/projectSetting.ts` 中的 `appearance` | 结构更合理，但运行时主色又被旁路覆盖 |
| 主题配置面板 | `src/layout/components/Header/ProjectSetting.vue` | 选择主色时同时写两个 store |
| 运行时合并外观 | `src/App.vue` | `effectiveAppearance.themeColor` 被强制替换成 `designStore.appTheme` |
| 预设 | `src/theme/naive/presets.ts` | 当前只改 `appearance`，与运行时主色可能分叉 |

### 已确认事实

- `ProjectSetting.vue` 已改为只写 `projectStore.appearance.themeColor`。
- `App.vue` 的主题运行时控制已抽离到 `useThemeController`，不再覆盖 `appearance.themeColor`。
- `projectStore.applyPreset()` 继续只更新正式 `appearance / layout / general`，不再需要同步旧主题源。
- `src/store/index.ts` 当前只继续持久化 `PROJECT_SETTING`；`DESIGN_SETTING` 仅用于旧缓存回填。

### 当前结论

- 主题主链路的“双真相源”风险已完成收敛。
- 本文档当前的价值，主要转为记录为什么要保留兼容桥，以及收口阶段仍需遵守的边界。

| 层级 | 当前文件 | 现状 |

# 3. 实施计划冻结前必须完成的审计工作流

## 3.1 工作流 A：主题真相源收敛审计

### 目的

- 决定唯一主题主状态源。
- 明确 `designStore`、`projectStore.appearance`、preset、brand palette 的最终职责。

### 必做项

- 列出当前所有能修改主题的入口。
- 列出当前所有直接读取主题状态而不经过 token 层的文件。
- 冻结目标口径：
  - 谁是唯一 source of truth
  - 谁是静态配置
  - 谁是运行时派生结果

### 产出

- 一张“主题状态职责表”
- 一张“退役字段清单”

## 3.2 工作流 B：消费面盘点审计

### 目的

- 获得完整迁移影响面，避免后续重命名时漏改。

### 必做项

- 扫描并分类所有主题消费点：
  - `--primary-*`
  - `--n-*`
  - 新 `--app-*`
  - 直接读 store
  - 硬编码主色/灰阶/圆角/阴影
- 按文件级输出影响清单。
- 标注“公共组件”与“业务页面”的优先级。

### 产出

- 一份“主题消费影响清单”
- 一份“高风险公共组件名单”

## 3.3 工作流 C：硬编码视觉值审计

### 目的

- 明确哪些值应迁移为 token，哪些值可暂时保留。

### 必做项

- 扫描主题相关的硬编码：
  - 主色
  - 灰阶文本
  - 背景色
  - 边框色
  - 圆角
  - 阴影
  - 间距
- 区分：
  - 属于主题系统的值
  - 属于页面局部视觉表达的值
  - 属于第三方组件适配值

### 产出

- 一份“候选 token 值清单”
- 一份“可延后清理清单”

## 3.5 工作流 E：持久化与迁移兼容审计

### 目的

- 确保本地缓存、preset、旧变量迁移不会引发存量用户状态异常。

### 必做项

# 7. 关于“保证没有 bug”的现实边界

- 本次改造无法承诺绝对零 bug。
- 可承诺的是：
  - 不在状态源未冻结前开改
  - 不在兼容桥未准备好前删旧名
  - 不在验证矩阵未冻结前推进大规模迁移
  - 不在公共组件未回归前批量修改业务页面

### 本文档对应的质量目标

- 把风险从“执行时发现”前移到“计划冻结前发现”。
- 把“主题改造”从一次性大爆改，转成可验证、可回滚、可分阶段推进的工程改造。
