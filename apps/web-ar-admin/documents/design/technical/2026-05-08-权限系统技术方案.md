# 权限系统技术方案（vben 迁移版）

## 1. 权限码来源

### 1.1 数据流

```
POST /admin/login → { token }
        ↓
POST /Home/GetSysUserInfo → { menus: MenuTreeBaseRsp[] }
        ↓
appUserStore.getPermissionCodes
  递归遍历 menus 树，提取每个节点的 authCode，统一转小写
        ↓
accessStore.setAccessCodes(codes)
  写入 vben 的 accessStore，全局可读
```

### 1.2 menus 树结构

```typescript
interface MenuTreeBaseRsp {
  id: number;
  name: string;
  authCode: string; // 权限码，如 "SystemManage:SysUserPage:UserPage:Add"
  menuType: MenuTypeEnum;
  parentId: number;
  children?: MenuTreeBaseRsp[];
}
```

所有权限码**统一转小写**后存入 `accessStore.accessCodes`，消费方比较时也用小写。

### 1.3 本地开发豁免

`fetchUserInfo` 中检测到本地 IP（localhost / 127.0.0.1 / 192.168.x / 10.x / 172.16-31.x）时，自动向 codes 中追加 `'localhost'`，确保本地开发不需要配置权限。

## 2. 访问模式选型：Frontend Mode

### 2.1 选型结论

使用 vben 的 **`accessMode: 'frontend'`**，不使用 backend mode。

### 2.2 选型理由

| 对比项 | Frontend Mode | Backend Mode |
| --- | --- | --- |
| 路由定义 | 前端静态定义，meta.authority 标注权限码 | 后端返回完整路由树 |
| 后端要求 | 无额外接口需求 | 需新增 `/menu/all` 返回 vben 格式路由结构 |
| 老项目兼容 | 老项目也是静态路由 + 前端过滤，模式一致 | 老项目后端不返回路由结构，需适配层 |
| 迁移成本 | 低：路由 meta 字段改名即可 | 高：需后端配合，联调成本大 |

老项目本身就是 frontend mode（静态路由 + 权限过滤），迁移到 vben 是同类换底，无需改变后端协议。

## 3. 三个层面的实现

### 3.1 路由级（页面入口控制）

vben `generateRoutesByFrontend` 用 `meta.authority` 过滤路由，逻辑是 **any-match（OR）**：

```typescript
// packages/utils/src/helpers/generate-routes-frontend.ts
function hasAuthority(route: RouteRecordRaw, access: string[]) {
  const authority = route.meta?.authority;
  if (!authority) return true; // 无 authority = 所有人可访问
  return access.some((value) => authority.includes(value));
}
```

迁移时路由定义：

```typescript
// src/router/routes/modules/system.ts
{
  path: '/system/user',
  meta: {
    authority: ['systemmanage:sysuserpage:userpage'],  // 小写，对应老项目 meta.permissions
    title: '用户管理',
    icon: 'lucide:users',
  },
  component: () => import('#/views/system/user/index.vue'),
}
```

**老项目 → 新项目字段对应：**

| 老项目                               | 新项目（vben）                       |
| ------------------------------------ | ------------------------------------ |
| `meta.permissions: ['Code']`         | `meta.authority: ['code']`（小写）   |
| `userStore.permissions`              | `accessStore.accessCodes`            |
| `src/router/generator.ts` filterTree | vben 内部 `generateRoutesByFrontend` |

### 3.2 按钮级（v-permission 指令）

指令签名与老项目**完全兼容**，写法不需要改：

```html
<!-- 简写：无权限时移除 DOM（默认） -->
<n-button v-permission="['SystemManage:SysUserPage:UserPage:Add']"
  >新增</n-button
>

<!-- 对象写法：无权限时禁用（保留 DOM） -->
<n-button
  v-permission="{ action: ['SystemManage:SysUserPage:UserPage:Export'], effect: 'disabled' }"
>
  导出
</n-button>
```

实现读 `accessStore.accessCodes`，与老项目读 `userStore.permissions` 的逻辑等价。

**注意：** 权限码传入时大小写均可，指令内部转小写后比较。

### 3.3 JS 侧（usePermission hook）

```typescript
import { usePermission } from '#/hooks/web/usePermission';

const { hasPermission, hasEveryPermission } = usePermission();

// OR 逻辑：有任一权限码即为 true
const canView = hasPermission(['SystemManage:SysUserPage:UserPage']);

// AND 逻辑：必须全部具备
const canFullAccess = hasEveryPermission([
  'SystemManage:SysUserPage:UserPage',
  'SystemManage:SysUserPage:UserPage:Add',
]);

// 条件渲染
<div v-if="canView">...</div>

// 动态 tab 过滤
const tabs = allTabs.filter(tab => hasPermission(tab.requiredCodes));
```

**老项目 → 新项目 hook 对应：**

| 老项目方法 | 新项目方法 | 语义 |
| --- | --- | --- |
| `hasPermission(['A','B'])` | `hasPermission(['A','B'])` | OR，有任一即通过 ✅ |
| `hasEveryPermission(['A','B'])` | `hasEveryPermission(['A','B'])` | AND，全部具备才通过 ✅ |
| `hasSomePermission(['A','B'])` | `hasPermission(['A','B'])` | OR 别名，直接用 hasPermission |
| `hasFirstPermission(['A','B'])` | 无对应 | 极少用，遇到时手动实现 |

## 4. 迁移时如何查权限码

1. 在老项目找到对应页面的路由定义，读 `meta.permissions`
2. 或在老项目页面代码中搜索 `v-permission`、`hasPermission`，找到使用的码
3. 码的大小写在新项目里不敏感，但建议**保持原始大小写**写入 `meta.authority` 和组件，阅读更清晰

老项目路由示例：

```typescript
// feat1-ar_platform_admin/src/router/asyncRoutes.ts
{
  path: '/system/user',
  meta: { permissions: ['SystemManage:SysUserPage:UserPage'] },
}
```

→ 新项目路由：

```typescript
{
  path: '/system/user',
  meta: { authority: ['SystemManage:SysUserPage:UserPage'] },
}
```

## 6. 不在此方案范围内

- **数据权限**（商户只看自己的数据）：后端接口层保证，前端不处理
- **动态路由（backend mode）**：已决策不使用，后端不返回路由结构
- **多租户权限隔离**：tenant 切换逻辑在 `appUserStore.activeTenantId`，独立于权限码体系
