# 0. 文档目标

- 为当前后台建立一套“全局 theme state -> 全局 token -> Naive UI / Tailwind / 品牌层”的统一主题架构。
- 收敛当前双主题真相源、散落的运行时逻辑、分裂的变量命名和未接入 token 的 Tailwind。
- 为后续《主题 token 重构实施计划》提供已冻结的设计前提，避免实施阶段反复改口。

> 本文档以上一份前置审计文档为输入： `documents/design/technical/2026-04-30-主题-token-重构前置审计.md`

## 0.1 当前落地状态（2026-04-30）

- 本文档中的核心设计已完成第一轮代码落地：
  - 全局主题唯一主状态源已切到 `projectStore.appearance`
  - `resolvedTheme -> theme tokens -> Naive adapter / CSS vars bridge` 链路已上线
  - `useThemeController` 已接管根层主题同步、`dark` class、CSS vars、favicon 更新
  - `Tailwind 4` 已通过 `@theme inline` 接入第一批 token 映射
- 尚未完全收尾的部分：
  - `src/components` / `src/views` 中已无真实 `--primary-*` 业务消费，`--primary-*` 运行时 bridge 已退役
  - `--app-*` bridge 已删除，运行时仅保留正式 token 输出
  - Header / tenant 这类壳层浮层已接入 token，但仍可能存在少量局部视觉细节可继续收敛
  - `designSetting` store 与 `DESIGN_SETTING` 常量已删除，旧缓存不再参与运行时主题解析
  - 全仓 `typecheck` 未清零，当前阻塞来自既有 TS 基线问题，而非本轮主题链路本身

# 1. 设计目标与边界

## 1.1 目标

- 全局主题只保留一套主状态源。
- 全局 token 层采用“原子 token + 语义 token + legacy alias”三层输出。
- 主题配置面板直接消费 token registry 或 theme config，而不是散落的硬编码色板。
- Naive UI、Tailwind、品牌层、离散 API、表格工具层统一消费主题输出。
- `App.vue` 中的主题运行时逻辑收敛到独立主题控制层。

## 1.2 非目标

- 本轮不重写 Naive UI 组件库本身。
- 本轮不一次性清理全部业务页面硬编码视觉值。
- 本轮不强制所有页面立刻迁移到新 token。
- 本轮不为了主题重构而改动无关业务逻辑。

## 1.3 冻结约束

- 不改 Naive UI 的 `--n-*` 命名。
- 不再新增旧 `--primary-*`；该 bridge 已在当前落地代码中退役。
- 不新增第二套主题持久化。
- 主题运行时 hook 可以使用 VueUse，但不能成为新的主题真相源。

# 2. 核心问题定义

## 2.1 当前问题

### P1. 主题真相源重复

- 当前运行时主色同时存在于：
  - `designStore.appTheme`
  - `projectStore.appearance.themeColor`
- `App.vue` 还会把 `appearance.themeColor` 再覆盖成 `designStore.appTheme`。

### P2. token 层太薄

- 当前 `src/theme/naive/css-vars.ts` 主要只输出主色和圆角。
- 还没有形成稳定的全局语义 token 层。

### P3. adapter 层和 token 决策层混杂

- `theme-overrides.ts` 既在做 Naive adapter，又在做大量视觉值决策。
- `App.vue` 既是应用根组件，又承担了主题运行时控制职责。

### P4. Tailwind 没接入主题 token

- Tailwind 4 已接入工程，但尚未消费项目 token。
- 结果是页面层会继续在 Tailwind、less、内联样式之间各写各的。

### P5. 消费面命名分裂

- 当前同时存在：
  - `themeColor`
  - `appTheme`
  - `--app-primary-*`
  - `--primary-*`
  - `--n-*`
- 这些命名表达的层级和职责并不清晰，容易造成误用。

## 2.2 目标问题陈述

本次重构要解决的不是“如何换一套变量名”，而是：

- 如何建立一套唯一主题状态源。
- 如何把主题状态变成稳定可消费的 token。
- 如何让 Naive UI、Tailwind、品牌层从同一套 token 出发。
- 如何在保留兼容桥的前提下渐进迁移，而不是一次性打爆全站。

# 3. 目标架构

## 3.1 分层模型

```text
Theme State Layer
  -> Resolved Theme Layer
  -> Token Output Layer
  -> Adapter Layer
  -> Consumer Layer
```

### Theme State Layer

- 唯一真相源：
  - `projectStore.appearance.themeColor`
  - `projectStore.appearance.themeScheme`
  - `projectStore.appearance.borderRadius`
  - `projectStore.appearance.compactMode`
- 主题相关布局输入：
  - `projectStore.layout.navigationTheme`
  - `projectStore.layout.navigationMode`

### Resolved Theme Layer

- 负责把原始 store 状态解析为运行时可用主题：
  - 实际深浅模式
  - 品牌主色派生结果
  - 密度档位
  - 圆角档位

### Token Output Layer

- 输出三层变量：
  - 原子 token
  - 语义 token
  - legacy alias token

### Adapter Layer

- Naive adapter
- Tailwind adapter
- Brand adapter
- Runtime DOM adapter

### Consumer Layer

- 页面样式
- 公共组件
- Naive 组件
- 登录页 / Logo / favicon
- 工具层

## 3.2 单向数据流

```text
ProjectSetting UI
  -> projectStore.appearance/layout
  -> resolveTheme()
  -> createThemeTokens()
  -> {
       applyDomTheme(),
       createNaiveThemeOverrides(),
       exposeTailwindThemeVars(),
       applyBrandTheme()
     }
  -> pages / components
```

## 3.3 冻结结论

- `designStore.appTheme` 不再作为最终运行时主题主色。
- `ProjectSetting.vue` 不再直接双写主题状态。
- `App.vue` 不再长期持有完整主题控制逻辑。

# 4. 主题状态模型设计

## 4.1 唯一主题状态源

建议冻结为：

```ts
projectStore.appearance = {
  themeColor: string,
  themeScheme: 'light' | 'dark' | 'auto',
  compactMode: boolean,
  borderRadius: number,
};
```

## 4.2 历史兼容状态

`designStore` 只保留以下两种可能职责之一：

### 方案 A：完全收敛

- `designStore.appTheme` 退役
- `designStore.appThemeList` 迁到正式 `themeConfig`

### 方案 B：迁移期兼容

- `designStore.appTheme` 仅作为旧缓存迁移来源
- 不再作为新逻辑长期读取入口
- 迁移期结束后删除

### 推荐

- 采用 **方案 B** 过渡，再在后续阶段完全收敛。
- 原因：当前已有直接读取点位，直接删会扩大改造面。

## 4.3 Resolved Theme 结构

建议增加运行时解析结构：

```ts
interface ResolvedTheme {
  colorMode: 'light' | 'dark';
  requestedScheme: 'light' | 'dark' | 'auto';
  themeColor: string;
  borderRadius: number;
  compactMode: boolean;
  navigationTheme: 'light' | 'dark';
  brandPalette: BrandPalette;
}
```

### 设计原则

- store 保存“用户意图”
- resolved theme 保存“运行时结果”
- token 层只消费 resolved theme，不直接消费多个 store 分支

# 5. Token 体系设计

## 5.1 输出分层

### 第一层：原子 token

示例：

```css
--color-brand-500
--color-brand-600
--color-neutral-0
--color-neutral-50
--color-neutral-200
--color-neutral-700
--space-1
--space-2
--space-3
--radius-sm
--radius-md
--radius-lg
--shadow-sm
--shadow-md
--z-dropdown
--z-modal
--motion-fast
--motion-normal
```

### 第二层：语义 token

示例：

```css
--ui-accent-solid
--ui-accent-hover
--ui-accent-soft
--ui-bg-page
--ui-bg-card
--ui-bg-elevated
--ui-text-primary
--ui-text-secondary
--ui-text-tertiary
--ui-text-inverse
--ui-border-default
--ui-border-strong
--ui-border-focus
--ui-fill-hover
--ui-fill-active
--ui-radius-control
--ui-radius-card
```

### 第三层：legacy alias

用于平滑迁移：

```css
--primary-color
--primary-color-hover
--primary-color-active
--primary-color-pressed
--primary-color-suppl
--primary-color-bg
--primary-color-bg-hover
```

## 5.2 命名规则

### 原子 token 命名规则

- 只表达值域，不表达业务语义
- 示例：
  - `--color-brand-500`
  - `--space-2`
  - `--radius-md`

### 语义 token 命名规则

- 只表达用途，不表达色相
- 示例：
  - `--ui-bg-card`
  - `--ui-text-secondary`
  - `--ui-border-default`

### Legacy alias 命名规则

- 保持历史名称不变
- 只做映射桥，不作为新代码命名入口

## 5.3 兼容策略

- 新代码优先消费 `--ui-*`
- 可复用基础层可在必要时消费原子 token
- 旧代码短期继续消费 `--primary-*`
- Naive UI 内部变量 `--n-*` 保持原样

## 5.4 关于 `--primary-color-suppl`

- 当前仓库对 `--primary-color-suppl` 的语义并不稳定。
- 部分旧代码把它当“强调补充色”，部分代码把它当“强调浅底色”。
- 设计阶段不直接重定义其历史语义。
- 迁移期策略：
  - 先保持旧行为可用
  - 新代码全部改用明确的 `--ui-accent-soft` / `--ui-accent-solid`

# 6. Adapter 设计

## 6.1 Naive Adapter

职责：

- 从统一 token 或 resolved theme 生成 `GlobalThemeOverrides`
- 不再承担全局 token 决策中心角色

建议：

- `src/theme/naive/theme-overrides.ts` 保留为 adapter 文件
- 但其输入从 `ProjectAppearanceSetting` 升级为 `ResolvedTheme + ThemeTokens`

## 6.2 Tailwind Adapter

职责：

- 让 Tailwind 4 utility 可以消费项目 token
- 让布局仍使用原子类，但颜色、圆角、阴影、间距来自全局 token

建议：

- 继续使用现有 `src/styles/tailwind.css`
- 在该层引入 token 暴露，而不是新造独立主题系统
- Tailwind 不持有主题状态，只消费 CSS vars

## 6.3 Brand Adapter

职责：

- Login 页品牌背景
- Logo / BrandPlate
- favicon
- 后续品牌渐变资产

建议：

- 保留 `getBrandPalette()` 这一派生能力
- 但它的入参改为统一 `resolvedTheme.themeColor`
- 不再直接从页面或组件内部读取 `designStore.appTheme`

## 6.4 Runtime DOM Adapter

职责：

- 同步根节点 `data-theme` / `class`
- 同步 CSS vars
- 触发 favicon 更新

建议：

- 从 `App.vue` 抽离到独立 hook 或 controller

# 7. VueUse 使用边界

## 7.1 可以使用

- `usePreferredDark`
- `useColorMode`
- `useCssVar`（仅适合局部场景）

## 7.2 冻结建议

### 必用

- `usePreferredDark`

用途：

- 替换当前手写 `window.matchMedia('(prefers-color-scheme: dark)')`
- 作为 `themeScheme === 'auto'` 时的系统偏好输入

### 可用但不作为第一优先

- `useColorMode`

用途边界：

- 仅作为 DOM 主题标记的 runtime adapter 候选
- 不能成为真正主题状态源
- 如采用，必须关闭或绕过默认持久化

建议口径：

- 若采用 `useColorMode`，必须满足：
  - `storageKey: null` 或等效禁用持久化
  - 或显式绑定到现有 store/ref
  - 不能再生成第二套 `localStorage` 数据

### 不建议作为主题主链路方案

- `useStorage`

原因：

- 仓库已经通过 Pinia 统一持久化
- 再引入 `useStorage` 只会增加第二套缓存源

## 7.3 结论

- 本次设计推荐：
  - 用 `usePreferredDark` 处理系统深色偏好
  - 主题主状态继续由 Pinia 保存
  - `useColorMode` 仅在需要统一 DOM 标记行为时评估是否引入

# 8. `App.vue` 抽离方案

## 8.1 当前问题

`App.vue` 当前混合了：

- Naive provider 挂载
- 主题深浅判断
- CSS vars 同步
- `dark` class 切换
- favicon 同步
- theme overrides 创建

这使它同时承担了应用根组件和主题运行时控制器的职责。

## 8.2 目标边界

### `App.vue` 保留

- `NConfigProvider`
- 全局 provider 组合
- 对 theme controller 输出结果的消费

### 主题控制层承担

- 解析 resolved theme
- 同步 DOM theme 标记
- 输出 CSS vars
- 触发品牌层更新
- 生成 Naive adapter 输入

## 8.3 建议形态

建议新增独立主题运行时模块，例如：

```text
src/theme/runtime/
  theme-controller.ts
  theme-resolver.ts
  theme-tokens.ts
  theme-legacy-alias.ts
  useThemeController.ts
```

### 说明

- 这里是职责建议，不是最终文件级实施清单。
- 实际文件拆分在实施计划中再冻结。

# 9. 主题配置面板设计口径

## 9.1 当前问题

- 当前色板直接依赖 `appThemeList`
- 当前点选主题色时双写 `designStore` 和 `projectStore`

## 9.2 目标口径

- 配置面板只操作唯一主题状态源
- 色板数据来自 `themeConfig` 或 `token registry`
- preset 切换必须与主色保持一致，不允许再形成两套主色结果

## 9.3 未来扩展口径

主题面板后续可继续接入：

- 主题模式
- 密度模式
- 语义色板预设
- 品牌色板预设

但前提仍然是统一走同一条主题主链路。

# 10. 迁移策略设计

## 10.1 总原则

- 先加新层，不先删旧层
- 先收敛状态，再收敛变量，再迁消费面
- 先迁基础设施和公共组件，再迁业务页面

## 10.2 迁移顺序

### 阶段 1

- 收敛主题主状态源
- 抽出 theme controller

### 阶段 2

- 建立 token registry
- 建立语义 token 与 legacy alias

### 阶段 3

- 接 Naive adapter
- 接 Tailwind adapter
- 接 Brand adapter

### 阶段 4

- 迁高风险公共组件

### 阶段 5

- 迁关键业务页面

### 阶段 6

- 清理历史旧名与残余硬编码

## 11.3 回滚原则

- `designStore.appTheme` 退场前必须确认 direct reads 已收敛
- 公共组件未回归前不得批量迁移业务页面

# 12. 设计冻结结论

## 已冻结

- 唯一主题主状态源应收敛到 `projectStore.appearance`
- 全局 token 采用“三层输出”：
  - atomic
  - semantic
  - legacy alias
- Naive UI 不改 `--n-*`
- Tailwind 4 走“消费 token”，不是独立主题系统
- VueUse 可以接入，但只做 runtime adapter，不做状态真相源
- `App.vue` 的主题运行时逻辑要抽离

## 待实施计划细化

- 最终文件级拆分
- 每阶段改动顺序
- 每阶段验证命令与回归动作
- 旧缓存迁移是否使用懒迁移还是显式迁移
