# Codex Harness 产品定义

## 1. 文档定位

本文档用于定义当前产品的本质、边界、目标与版本阶段。

本文档重点回答的问题是：

- 当前产品本质是什么
- 长期产品本质是什么
- 当前为什么坚持 `Codex-first`
- 当前阶段到底在做什么
- 哪些能力属于核心目标
- 哪些能力当前明确不做

本文档不展开：

- 具体系统分层
- 具体任务执行模型
- 具体角色与 agent 编排
- 具体 skill 接入细节
- 具体 monorepo package 设计

这些内容分别由后续架构、执行模型与 packaging 文档展开。

---

## 2. 长期产品本质

长期来看，产品不应被定义为 `Codex Harness` 本身，而应被定义为：

**Harness：一个面向 AI 工程协作的复杂需求执行骨架。**

它的核心目标不是提供更花哨的聊天体验，也不是做一个通用多模型研究平台，而是把项目中的复杂需求，从模糊自然语言输入，转化为：

- 可理解
- 可拆解
- 可对象化
- 可确认
- 可持续推进
- 可恢复
- 可收口
- 可沉淀为正式文档产物

的正式执行系统。

换句话说，长期产品本质不是围绕某一个模型，而是围绕：

**复杂需求如何被正式执行。**

---

## 3. 当前版本本质

当前版本仍然采用 `Codex Harness` 这一名称，是因为第一版发行形态明确选择：

**先以 Codex 作为唯一宿主与唯一模型内核，把单模型复杂需求执行能力打稳。**

因此，当前版本更准确的定义是：

**Harness 的第一版 Codex 适配发行版。**

当前版本的目标不是证明“产品只属于 Codex”，而是先在 Codex 上验证以下能力是否能稳定成立：

- 复杂需求拆解
- 长任务拆批次
- 执行闸门控制
- 多任务编排
- 文档产物化
- AI 协作治理

所以：

- `Harness` 是长期产品本体
- `Codex Harness` 是当前第一版产品形态

---

## 4. 为什么当前是 Codex-first

当前阶段先只围绕 `Codex` 构建。

原因不是产品能力不足，而是当前最关键的问题不是“支持更多模型”，而是：

**先把单模型复杂需求拆解与执行能力打稳。**

在当前系统里，复杂需求能否被稳定拆解，直接取决于以下要素是否协同：

- 模型本身
- Codex 配置
- Profiles
- 角色提示词
- Skills
- 项目规则
- 文档治理

因此，当前产品路线不是：

- 先追求多模型扩展
- 先做模型编排平台
- 先做通用 agent market

而是：

- 先把 `Codex` 变成稳定宿主
- 先把单模型工作流做成骨架
- 先把复杂需求拆解与执行能力做稳

只有这一层稳定之后，后续扩展更多宿主与模型才有意义。

---

## 5. 当前产品的一句话定义

**Codex Harness 是 Harness 的第一版 Codex 适配发行版，用于把需求、讨论、PRD 与任务上下文转化为正式任务对象、正式执行链路与正式文档产物。**

---

## 6. 长期产品的一句话定义

**Harness 是一个宿主无关的复杂需求执行骨架，用于把 AI 协作从聊天讨论升级为对象化、阶段化、可恢复、可持续推进的正式执行系统。**

---

## 7. 解决什么问题

当前产品主要解决以下问题：

### 7.1 复杂需求只停留在聊天层

传统对话式 AI 很容易停在：

- 聊懂了
- 说清楚了
- 给出计划了

但没有进入真正可持续执行的系统状态。

当前产品要解决的是：

**复杂需求如何从聊天讨论，进入正式执行。**

### 7.2 需求拆解不稳定

复杂需求往往会出现：

- 边界模糊
- 拆分过粗
- 拆分过细
- 执行单元不稳定
- 任务切分依赖临时上下文记忆

当前产品要解决的是：

**把复杂需求稳定拆解为可执行最小单元。**

### 7.3 长任务无法持续推进

复杂需求往往不是单轮完成的，而是：

- 多阶段
- 多批次
- 多任务
- 多子 agent

当前产品要解决的是：

**在确认执行后，让任务以运行态持续推进，而不是反复回到从头解释需求。**

### 7.4 文档不是正式产物

在很多项目里，文档只是补充说明，不是系统产物。

当前产品要解决的是：

- 需求文档产物化
- 设计文档产物化
- 交付文档产物化
- 演进文档产物化

也就是让文档成为执行系统的一部分。

### 7.5 AI 协作规则不稳定

如果规则只存在于对话中，就会出现：

- 会话切换后丢失
- 优先级不清
- 新会话重复解释
- 行为漂移

当前产品要解决的是：

**把 AI 协作从临时对话约束升级为项目默认规则。**

---

## 8. 核心产品目标

当前阶段，产品的核心目标应收敛为以下几项。

### 8.1 把复杂需求对象化

系统必须能把：

- 一句话需求
- PRD
- 多轮讨论结果

转成正式任务对象，而不是停留在聊天上下文。

### 8.2 把复杂需求拆解为可执行最小单元

系统必须支持：

- 复杂任务拆分
- 长任务拆分
- 批次划分
- 子任务划分
- 执行单元划分

### 8.3 建立执行闸门

`开始执行` 必须成为唯一正式执行授权口令。

在进入执行态之前，任务必须已经达到：

- 目标明确
- 边界明确
- 依赖明确
- 拆解完成
- 可执行最小单元成立

### 8.4 让任务在确认后进入持续推进状态

任务一旦进入正式执行态，系统应尽量支持：

- 无中断推进
- blocked / resume
- 多批次推进
- 多任务编排
- 主 agent 调度子 agent

### 8.5 让文档成为正式产物

执行结果不仅是代码，还应包括：

- 需求文档
- 设计文档
- 交付文档
- 演进文档

### 8.6 让 AI 协作规则成为默认制度

系统应通过：

- `AGENTS.md`
- 文档规范
- AI 协作规范
- Skills
- 配置

形成稳定的项目级协作约束。

---

## 9. 当前阶段目标

当前阶段不是全面产品化，也不是立刻做多宿主平台，而是：

**先构建稳定的单模型 Codex 工作骨架与第一版适配器。**

当前阶段重点应放在：

- Codex 最佳配置整合
- `plan / dev / review` profiles 稳定
- `Planner / Implementer / Reviewer` 角色稳定
- `superpowers` skill workflow 接入
- `开始执行` 执行闸门固化
- 文档治理与 AI 协作规范联动
- 复杂任务对象化与执行主链路稳定
- `Harness core + adapter-codex` 边界初步成立

这一阶段的关键词是：

- 稳定
- 收敛
- 骨架成立
- 适配器成立

---

## 10. 非目标

当前产品明确不做：

- 通用多模型 orchestration 平台
- DeerFlow 式 graph execution engine
- 通用 deep research 平台
- 报告 / 播客 / PPT 生成平台
- 纯 prompt/skill 仓库
- 单纯 coding assistant
- 一开始就做完整的多宿主生态
- 一开始就做很重的服务端控制面

这些都不属于当前阶段的优先方向。

---

## 11. 用户接入后的预期能力

用户将当前版本接入自己的项目后，预期应获得以下能力：

- 一个稳定的 Codex 工作骨架
- 一个受规则约束的 AI 协作方式
- 一套复杂需求拆解与执行主链路
- 一套文档产物化能力
- 一套复杂任务对象化、批次化、持续推进能力

也就是说，用户接入的不是“一个模型”，而是：

**一套让 Codex 在项目里稳定拆解复杂需求、持续执行并沉淀文档的工作系统。**

---

## 12. 与 Harness 内核的关系

当前文档所定义的是产品层与发行层视角。

其中：

- `Harness` 是长期产品核心
- `Codex Harness` 是当前发行形态
- `harness-system` 目录中的内容，继续承担执行内核、任务对象、生命周期、runtime control plane、plugin attach / resolution 与 diagnostics 的底层技术依据

因此：

- 产品总叙事不再只依赖 `harness-system`
- `harness-system` 继续保留为底层实现与内核设计依据

---

## 13. 一句话总结

当前最准确的产品判断是：

**Harness 是长期产品本体，Codex Harness 是其第一版 Codex 适配发行版；当前阶段的任务不是做大平台，而是先把单模型复杂需求执行骨架打稳。**
