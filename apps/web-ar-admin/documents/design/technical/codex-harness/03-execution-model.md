# Codex Harness 执行模型

## 1. 文档定位

本文档用于定义当前发行版的执行模型。

本文档重点回答的问题是：

- 复杂需求如何进入正式执行链路
- 任务对象如何组织
- `开始执行` 为什么是执行闸门
- 什么是可执行最小单元
- 长任务与复杂任务如何拆成批次
- 多任务与多子 agent 如何进入正式编排
- “无中断执行”在本系统中到底是什么意思
- 哪些执行能力属于长期 `Harness core`
- 哪些执行能力当前由 `adapter-codex` 承载

本文档不展开：

- 产品定义与边界
- 三层总架构
- 角色与运行时 agent 映射
- skill workflow 接入细节

这些内容由其他文档承接。

---

## 2. 执行模型的核心目标

执行模型不是为了描述“AI 会做什么”，而是为了描述：

**复杂需求如何从自然语言输入，进入正式任务执行状态。**

它的核心目标包括：

- 把需求对象化
- 把复杂任务拆解为批次和执行单元
- 建立执行闸门
- 让任务在确认后进入持续推进状态
- 让 blocked / resume 成为正式协议
- 让执行结果最终形成代码产物与文档产物

换句话说，执行模型不是 prompt 规则，而是系统运行骨架。

---

## 3. 执行模型的宿主无关性

长期来看，执行模型不应被绑定死在 `Codex` 上。

以下能力应尽量属于长期 `Harness core`：

- Requirement / Task / Batch / Work Item / Execution Unit
- 生命周期阶段
- 执行闸门
- blocked / resume
- 批次推进
- 多任务编排抽象
- diagnostics 升级闭环抽象

这些能力描述的是：

**复杂需求如何被正式执行。**

它们不依赖某一个宿主。

当前之所以以 `Codex` 承载这些能力，是因为：

- 第一版只做 `adapter-codex`
- 当前所有 prompt、profiles、skills 与接入方式都优先围绕 `Codex` 验证

所以应区分：

- 执行模型本身：长期应宿主无关
- 当前执行承载：第一版由 `adapter-codex` 实现

---

## 4. 输入如何进入执行模型

系统的输入可以是：

- 一句话需求
- PRD 文档
- 多轮讨论结果
- Bug 描述
- 架构改造目标
- 文档补全需求

这些输入在刚进入系统时，都还不是正式任务。

它们只属于：

- 原始输入
- 候选需求
- 待澄清问题

因此，执行模型的第一步永远不是直接执行，而是：

**将输入收敛为正式任务对象。**

---

## 5. 任务对象模型

复杂需求要能够被持续执行，必须先被对象化。

当前执行模型建议至少稳定以下对象层级：

### 5.1 Requirement

用于承载：

- 原始需求
- PRD
- 多轮讨论后的目标表达
- 需求背景和约束

这是系统中的“问题来源层”。

### 5.2 Task

用于承载：

- 已被确认的正式任务
- 明确边界后的工作目标
- 进入生命周期流转的主对象

这是系统中的“正式执行对象层”。

### 5.3 Batch

用于承载：

- 长任务和复杂任务的阶段性批次
- 一轮可推进的任务集合
- 同一阶段下的一组有组织的工作项

这是系统中的“长任务切片层”。

### 5.4 Work Item

用于承载：

- 某一批次中的子任务
- 某个明确工作块
- 可被主 agent 分派的执行单元集合

这是系统中的“任务拆解层”。

### 5.5 Execution Unit

用于承载：

- 真正可执行的最小单元
- 已经达到执行阈值的工作项
- 可交给执行子 agent 落地的具体对象

这是系统中的“最小执行层”。

---

## 6. 生命周期模型

任务对象一旦形成，就进入正式生命周期。

当前建议的主生命周期为：

1. `requirement-analysis`
2. `boundary`
3. `planning`
4. `execution`
5. `verification`
6. `submission`
7. `closure`

必要时可保留对象入口层，如：

- `task-card-entry`
- `boundary-start`
- `formal-task`

但必须注意：

- 对象入口层不是生命周期主 phase
- phase 负责执行阶段语义
- 对象层负责承载任务实体

---

## 7. 执行闸门：开始执行

## 7.1 `开始执行` 的角色

`开始执行` 不是普通对话指令，而是系统中最关键的执行授权口令。

它的意义是：

- 在此之前，系统处于理解、澄清、拆解、确认阶段
- 在此之后，系统才进入正式执行阶段

因此：

**`开始执行` 是执行闸门，不是普通口令。**

## 7.2 为什么必须有执行闸门

如果没有执行闸门，系统会退化成：

- 一边讨论一边执行
- 需求没清晰就开始改代码
- 长任务在错误边界上持续推进
- 主 agent 不断在理解与执行之间来回切换

所以，执行闸门的本质是：

**把“讨论态”和“执行态”彻底分开。**

## 7.3 什么情况下允许进入执行态

只有当任务已经收敛成可执行最小单元时，才允许进入执行态。

最低要求应包括：

- 目标明确
- 非目标明确
- 影响范围明确
- 当前批次明确
- 当前 work item 明确
- 当前 execution unit 明确
- 依赖和阻塞项已识别
- 验证方式已定义

未达到这些条件，不应接受 `开始执行`。

---

## 8. 可执行最小单元

系统不允许把“一个大需求”直接当成执行对象。

真正进入执行态的，必须是：

**可执行最小单元（Execution Unit）**

它的基本特征包括：

- 边界清晰
- 输入清晰
- 输出清晰
- 依赖清晰
- 可验证
- 可交付

如果一个工作块仍然需要继续大范围澄清边界，就说明它还不是最小执行单元。

这也是为什么系统必须先做：

- 分析
- 拆解
- 批次划分
- 确认

然后才允许执行。

---

## 9. 长任务与复杂任务拆分

## 9.1 为什么必须拆分

复杂需求通常具有以下特征：

- 涉及多个模块
- 涉及多个阶段
- 涉及多个角色
- 无法在单轮中安全完成
- 执行中会出现自然中断点

如果不拆，系统会出现：

- 任务过大无法稳定推进
- 子 agent 分工困难
- 验证和提交无法收口
- 状态恢复成本极高

因此，复杂需求必须进入批次化拆分。

## 9.2 Batch 的作用

`Batch` 用于把长任务切成若干轮可推进的阶段性集合。

每个 Batch 应回答：

- 当前这一轮做什么
- 这一轮不做什么
- 包含哪些 Work Item
- 完成后进入什么下一个状态

## 9.3 Work Item 的作用

`Work Item` 用于表达 Batch 内的子任务块。

它是：

- 批次内的工作容器
- 主 agent 的任务分派单位
- Execution Unit 的上层承载层

## 9.4 Execution Unit 的作用

`Execution Unit` 才是真正进入执行的最小对象。

一旦某个 Work Item 还无法进一步形成 Execution Unit，就不应直接进入正式执行。

---

## 10. 多任务编排

复杂需求在拆解完成之后，不是“线性一路到底”，而是进入：

- 多批次
- 多 Work Item
- 多 Execution Unit
- 多子 agent

的正式编排状态。

这里的“多任务编排”不是单纯并发，而是：

- 哪些任务必须先做
- 哪些任务可以并行
- 哪些任务可以被执行子 agent 接管
- 哪些任务需要查询子 agent 辅助
- 哪些任务需要等上一个 Batch 完成再推进

也就是说，多任务编排的目标不是“同时做很多事”，而是：

**在任务拆解完成并确认后，由运行态承接任务推进，而不是再次退回聊天理解。**

---

## 11. 无中断执行是什么意思

“无中断执行”并不等于永远不需要人工参与。

在本系统里，它更准确的含义是：

- 任务一旦通过执行闸门进入正式执行态
- 后续推进不再依赖重复解释需求
- 系统可通过对象状态、生命周期状态和运行态记录持续推进
- 即使发生 blocked，也可以通过正式协议恢复，而不是从头来过

所以，“无中断执行”的核心不是完全自动，而是：

**执行进入正式状态后，系统能够靠对象、阶段和运行态持续承接。**

---

## 12. 当前版本中的 Codex 承载方式

虽然执行模型长期应宿主无关，但当前第一版的承载方式明确是 `Codex`。

这意味着当前：

- 任务分析主要由 Codex 的 `Planner` 语义承担
- 任务执行主要由 Codex 的 `Implementer` 语义承担
- 验证与收口主要由 Codex 的 `Reviewer` 语义承担
- 主 agent / 查询子 agent / 执行子 agent 的运行时能力，先通过 `adapter-codex` 承接
- `superpowers` 先作为 `adapter-codex` 的 workflow layer 接入

所以：

- 执行模型本身不属于 Codex
- 当前执行模型的第一版产品化承载属于 `adapter-codex`

---

## 13. 一句话总结

**执行模型长期属于 `Harness core`，当前第一版由 `adapter-codex` 承载；它的核心不是让 AI 直接开干，而是先把复杂需求收敛为可执行最小单元，再通过执行闸门进入可持续推进的正式运行态。**
