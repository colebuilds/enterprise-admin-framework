# i18n 技术实现说明

## 1. 文档定位

本文解释两个问题：

## 2. 为什么不直接用 vue-i18n

vue-i18n 是一个通用国际化库，它提供了翻译核心，但没有解决以下应用级问题：

### 2.1 locale 文件的组织和动态加载

**问题：** vue-i18n 创建时需要你自己决定怎么注入消息，没有约定的文件扫描方式。

**vben 的解法：** `loadLocalesMapFromDir(regexp, import.meta.glob(...))` — 用 Vite 的 `import.meta.glob` 静态分析扫描所有 JSON 文件，按 locale 目录分组，构造每个 locale 的合并异步加载器。

```ts
// 调用方只写这一行，目录下所有 JSON 文件自动被纳入
const localesMap = loadLocalesMapFromDir(
  /\.\/langs\/([^/]+)\/(.*)\.json$/,
  import.meta.glob('./langs/**/*.json'),
);
```

好处：新增 JSON 文件不需要修改 loader 代码，目录即约定。

### 2.2 app 层消息的扩展注入点

**问题：** 单一 vue-i18n 实例需要同时承载框架消息（vben 自带）和业务消息（各 app 的），但 @vben/locales 不知道业务 app 的文件在哪。

**vben 的解法：** `setupI18n(app, { loadMessages })` 接受一个回调钩子。每次切换 locale 时，`loadLocaleMessages` 先加载框架层消息，再 `mergeLocaleMessage` 合并业务 app 通过 `loadMessages` 返回的消息。

```ts
async function loadLocaleMessages(lang) {
  // 框架层
  const message = await localesMap[lang]?.();
  i18n.global.setLocaleMessage(lang, message.default);

  // 应用扩展点
  const mergeMessage = await loadMessages(lang);  // ← 业务 app 提供
  i18n.global.mergeLocaleMessage(lang, mergeMessage);
}
```

### 2.3 locale 切换的幂等性

**问题：** 用户多次切换到同一个语言，或者在 Vue 组件树中多个地方都调用 locale 相关代码时，如何避免重复加载？

**vben 的解法：** `loadLocaleMessages` 的第一行检查当前 locale，相同则直接 return：

```ts
async function loadLocaleMessages(lang) {
  if (unref(i18n.global.locale) === lang) {
    return setI18nLanguage(lang);  // 只更新 html 属性，不重新加载消息
  }
  // ...加载逻辑
}
```

### 2.4 UI Kit 组件的轻量 locale

**问题：** vben 的 UI Kit 内部有一些弹框和组件（Alert、Confirm 等）需要"取消"/"确认"这类通用词汇，但这些组件不在 Vue 组件上下文中（它们通过命令式 API 调用），无法直接用 `useI18n()`。

**vben 的解法：** `useSimpleLocale()` — 用 `createSharedComposable` 创建一个轻量的共享响应式 locale。只包含极少的通用词汇（cancel/confirm/submit 等），不依赖 vue-i18n，i18n 切换时会同步调用 `setSimpleLocale(locale)`。

## 3. 为什么不直接用 @vben/locales（不加扩展）

如果直接使用 @vben/locales，存在以下问题：

### 3.1 类型只支持 2 个 locale

`SupportedLanguagesType = 'en-US' | 'zh-CN'` — vben 核心包硬编码。

AR admin **当前仅支持这两种语言**，与 vben 类型完全对齐，无需 `as any` 绕过。如将来需要扩展语言，才需要在 app 层添加 `as any` 绕过此约束。

### 3.2 SUPPORT_LANGUAGES 常量只有 2 个条目

`packages/constants/src/core.ts` 中的 `SUPPORT_LANGUAGES` 只有 `zh-CN` / `en-US`，`LanguageToggle.vue` 使用这个常量渲染语言菜单。

当前 AR admin 只支持这两种语言，与常量完全对齐，无需扩展。

### 3.3 NaiveUI 需要额外的 locale 映射

@vben/locales 只负责 i18n 消息，不处理 UI 组件库的 locale。NaiveUI 有自己的 locale 对象（`zhCN`, `enUS`, `viVN`, `idID`），需要 app 层手工映射。

**当前解法：** `app.vue` 维护静态映射表 `NAIVE_LOCALE_MAP`，响应式联动 `preferences.app.locale`。

## 4. 消息合并机制详解

### 4.1 vben 框架层的消息结构

`packages/locales/src/langs/zh-CN/` 下的 JSON 文件内容是**扁平的 key-value 对**（不嵌套）：

```json
// ui.json
{
  "cancel": "取消",
  "confirm": "确认"
}
```

`loadLocalesMapFromDir` 将这些文件的内容以 **`{ filename: content }` 的形式嵌套**：

```ts
messages['ui'] = (await uiJson()).default;
messages['common'] = (await commonJson()).default;
// ...
return { default: messages };
```

最终注入 i18n 的消息树：

```json
{
  "ui": { "cancel": "取消", "confirm": "确认" },
  "common": { ... },
  "authentication": { ... }
}
```

### 4.2 app 层消息通过 mergeLocaleMessage 合并

app 的 `loadMessages` 通过同样的 `loadLocalesMapFromDir` 扫描 `src/locales/langs/`，返回 `{ page: {...}, menu: {...} }` 结构，再 `mergeLocaleMessage` 合并进同一 locale 的消息树。

最终 `i18n.global.t('page.title')` 能命中 app 层的消息，`$t('ui.cancel')` 能命中 vben 层的消息，二者共存于同一 vue-i18n 实例。

## 5. 与老项目实现的差异说明

| 特性 | 老项目方案 | 新项目方案 |
|---|---|---|
| locale 代码 | 短码 `zh` / `en` / `vi` / `id` | BCP47 `zh-CN` / `en-US` / `vi-VN` / `id-ID` |
| 加载粒度 | namespace 级（按需加载） | locale 级（切换时全量加载） |
| BOOT_NAMESPACES | 有（`common + menu + components`）| 无（首次切换 locale 时全量合并）|
| 路由守卫与 i18n 的耦合 | 有（路由进入前加载对应 namespace）| 无 |
| 切语言方式 | `switchLocale(locale)` — 重放已加载 namespace | `loadLocaleMessages(locale)` — 直接合并全部 |
| 误报警告静默 | 有（切语言时临时关闭 missingWarn）| vben 内置 missingWarn handler |
| API 语言码 | `resolveSupportedApiLanguage()` 转短码 | `resolveRequestLanguage()` 在 `signBody` 内实时读取，BCP47 → 短码 |
| 切语言副作用 | `useLang.changeLang`：调 ChangeLanguage + 重拉所有字典 store | `useLocaleSync`：`invalidateDicts()` + `api.system.changeLanguage()` |
| locale 持久化 | 手动 `storage.set(CURRENT_LANG, locale)` | `@vben/preferences` 统一管理 |

**为什么新项目不延续老项目的 namespace 懒加载？**

老项目总 locale 资源约 700KB，namespace 懒加载有明显收益（首屏只加载 7% 资源）。

新项目在迁移初期 locale 资源体量还很小，全量加载是合理的。如果将来业务 locale 文件增大到性能可感知的程度，可以再引入 namespace 分层，届时参考老项目方案和 `2026-05-01-前端国际化方案.md`。

## 6. app 层的两个扩展点

### 6.1 resolveRequestLanguage（请求语言码）

位于 `src/api/request.ts`，在 `signBody` 函数内调用：

```ts
function resolveRequestLanguage(): string {
  return preferences.app.locale.split('-')[0] ?? 'zh'; // 'zh-CN' → 'zh'
}
```

`signBody` 每次调用时实时读取，不需要事件订阅，语言切换后下一个请求自动带新语言码。

### 6.2 useLocaleSync（切语言副作用）

位于 `src/composables/use-locale-sync.ts`，在 `App.vue` setup 中调用一次：

```ts
export function useLocaleSync() {
  let skipInitial = true;
  watch(
    () => preferences.app.locale,
    async (locale, prevLocale) => {
      if (skipInitial) { skipInitial = false; return; }
      if (locale === prevLocale) return;
      invalidateDicts();                                        // 字典 cache 失效
      if (useAccessStore().accessToken) {
        api.system.changeLanguage().catch(() => {});            // 同步后端，fire-and-forget
      }
    },
  );
}
```

vben 有两个语言切换入口（LanguageToggle + PreferencesDrawer），两者都通过 `updatePreferences` 写 `preferences.app.locale`，一个 watcher 全部覆盖。
