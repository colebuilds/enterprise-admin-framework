# 项目调试入口指南

## 1. 目标

提供一条可执行、可复用的调试入口路径，帮助在页面异常、请求异常、状态异常、构建异常出现时快速找到首个排查入口，避免“工具乱试、层级乱跳、凭感觉改代码”的低质量排查方式。

本文档回答“先看什么、先用什么工具、先从哪一层下手”，不替代完整的调试规范，也不替代 Bug 修复与回归验证流程。

涉及调试原则、问题分型和工具职责时，应进一步遵守：

- `FE-QUALITY-001-调试规范.md`

## 2. 适用范围

本指南适用于：

- `apps/*` 下的页面、组件、状态、请求和构建问题排查
- `packages/*` 下共享包、工具包、基础设施包的调试定位
- AI 与人工共同参与的联调、排查、验证场景

本指南主要覆盖：

- 页面异常
- 请求异常
- 状态异常
- 构建异常
- 浏览器链路异常
- 模块协作异常

本指南不直接覆盖：

- Bug 修复动作本身
- 回归验证层级选择
- Mock 策略完整规范

## 3. 使用前提

开始调试前，至少应先明确：

- 现象是什么
- 预期是什么
- 是否稳定复现
- 最近是否有相关改动
- 问题属于页面、请求、状态、逻辑还是构建层

如果连问题类别都无法判断，应先收集最小证据，再决定工具入口。

## 4. 标准操作步骤

### 4.1 先判断异常类型

第一步不是直接开工具，而是先判断问题落在哪一层。

- 页面异常
- 页面展示不对
- 组件没渲染出来
- 样式错位
- 请求异常
- 请求没发出
- 请求参数不对
- 响应解析错误
- 状态异常
- 页面状态不更新
- store 数据漂移
- 派生状态不一致
- 构建异常
- 启动失败
- 打包失败
- 资源解析失败

### 4.2 页面异常优先看浏览器层

首个入口优先级通常是：

1. `Chrome DevTools`
2. `Vue Devtools`
3. `vite-plugin-vue-inspector`

推荐顺序：

- 先看 DOM、控制台、样式和 Network
- 再看组件树、props、store、route
- 需要定位源码来源时再用 inspector

### 4.3 请求异常先看请求链

请求异常的首个入口通常是：

1. `Chrome DevTools`
2. `FE-API-003-请求参数与响应数据处理规范.md`
3. `FE-API-005-接口Mock与联调协作规范.md`

排查顺序建议：

- 先确认请求是否真的发出
- 再确认参数、header、路由、token、租户上下文是否正确
- 再确认响应结构是否符合预期
- 再确认是否在边界层被错误映射

如果问题已经进入真实接口联调、宿主联调或 bridge 协作阶段，继续参考 `documents/guides/integration/接口与宿主联调指南.md`，本指南只负责“先从哪里排查”。

### 4.4 状态异常先看状态来源

状态异常的首个入口通常是：

1. `Vue Devtools`
2. `FE-ARCH-003-状态管理规范.md`
3. `FE-DATA-002-状态数据归属与持久化规范.md`

排查顺序建议：

- 先确认状态来自组件、页面、全局还是服务端
- 再确认是否发生了复制状态、派生状态、持久化状态冲突
- 再确认是否有跨页面共享但未定义清理规则的问题

### 4.5 构建异常先看工程入口

构建异常的首个入口通常是：

1. 构建命令输出
2. `package.json` 的脚本
3. `vite` / workspace 配置
4. 相关基础设施规范

排查顺序建议：

- 先确认错误发生在启动、编译、打包还是资源解析阶段
- 再确认是否和入口目录、环境变量、包依赖或 workspace 配置有关
- 再确认是否是共享包导出或路径别名问题

### 4.6 纯逻辑问题优先用单元验证

如果问题更像纯逻辑、映射、转换、状态计算错误，应优先：

1. 用 `Vitest` 复现
2. 缩小到最小模块
3. 再决定是否需要浏览器层验证

### 4.7 页面链路问题优先用 Playwright

如果问题涉及完整页面流程、关键交互链路或 smoke 验证，应优先：

1. 用 `Playwright` 复现
2. 再结合 `Chrome DevTools` 看请求与 DOM
3. 必要时回到 `Vue Devtools` 看组件与状态

## 5. 常见场景

### 5.1 页面空白

优先看：

- `Chrome DevTools` 控制台报错
- Network 是否有失败请求
- `Vue Devtools` 是否命中正确组件

### 5.2 接口返回正常但页面不更新

优先看：

- `Vue Devtools` 里的状态流转
- store 是否被正确更新
- `FE-ARCH-003-状态管理规范.md`
- `FE-DATA-002-状态数据归属与持久化规范.md`

### 5.3 请求参数不对

优先看：

- `Chrome DevTools` 的 Network 请求详情
- 请求参数构造是否符合 `FE-API-003-请求参数与响应数据处理规范.md`
- DTO 与边界映射是否正确

### 5.4 构建失败

优先看：

- 构建日志
- 入口路径与别名
- workspace 包导出
- 相关基础设施规则

### 5.5 独立 app 能跑，集成到主框架后异常

优先看：

- 租户上下文是否正确
- 运行时配置装配是否正确
- bridge / 宿主能力是否到位
- `FE-INFRA-006-多租户运行时配置装配规范.md`

## 6. 常见错误

- 不先分型就同时打开所有工具
- 页面问题不看浏览器层，先盲修代码
- 请求问题不看 Network，直接改业务逻辑
- 状态问题不先看来源，直接改 store
- 构建问题不先看脚本和配置，直接大范围改代码
- 发现问题后不沉淀为测试、修复或回归资产
