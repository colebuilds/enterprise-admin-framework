# 工程治理文档使用建议

## 1. 文档定位

本文档用于说明工程治理相关文档在实际协作中的使用方式，帮助团队在面对新的工程治理需求时，快速判断是先写标准、先写接入手册，还是先补充一份更小的说明文档。

本文档只提供使用建议，不替代 `documents/standards/engineering/` 中的正式规范，也不替代 `documents/guides/engineering/` 下的具体接入手册。

## 2. 适用场景

适合以下情况：

- 需要新增工程治理相关说明，但暂时还不确定是否应直接上升为正式标准
- 需要先把工程治理思路讲清楚，再决定后续是否补接入手册或标准正文
- 需要在多个工程治理文档之间做选型，避免一开始就写重
- 需要给 AI 一个更明确的文档落点参考，减少凭感觉决定目录的情况

## 3. 使用建议

### 3.1 先判断文档目的

如果文档回答的是“应该怎样做”，通常更适合落到标准层。  
如果文档回答的是“具体怎么接入、怎么迁移、怎么落地”，通常更适合落到接入手册层。  
如果文档只是帮助梳理思路、给出临时建议或补充说明，则可先作为更小的工程治理说明文档使用。

### 3.2 先小后大

工程治理内容如果还不稳定，建议先写成较小的说明，再逐步演进为：

1. 标准
2. 接入手册
3. 需要时再补充专门的迁移说明或决策记录

这样可以避免一开始就把所有细节写进同一份大文档。

### 3.3 先定目录，再定内容

编写工程治理文档前，建议先按 `documents/README.md` 确认落点，再决定正文结构。  
如果文档属于正式文档，还应同步检查 frontmatter、`updated` 和最近一级 `README.md` 挂载。

### 3.4 保持标准与手册分层

标准负责规则本身，手册负责接入与迁移方式，使用建议类文档负责帮助判断该把内容放到哪一层。  
不要把三类职责混写在同一份文档里。

## 4. 常见注意事项

- 不要把使用建议写成正式规则正文
- 不要把临时判断写成长期制度
- 不要因为文档很小就忽略 frontmatter、`updated` 和挂载检查
- 不要把“建议”与“已生效规范”混为一谈
- 不要为了省事直接把说明文档塞进错误的目录

## 6. 渐进迁移建议

建议按以下顺序处理新的工程治理内容：

1. 先判断它是标准、手册还是说明建议
2. 再决定落点目录
3. 再补齐 frontmatter 和挂载
4. 最后再考虑是否需要上升为正式规范

## 7. 一句话总结

工程治理文档的使用原则不是“一律写成最完整的规范”，而是先判断内容层级，再把它放进最合适的目录，让标准、手册和说明各自承担清晰职责。
