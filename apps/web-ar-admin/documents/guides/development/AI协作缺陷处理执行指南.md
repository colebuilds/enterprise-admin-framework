# AI协作缺陷处理执行指南

## 1. 文档目的

本文档用于说明在当前项目中，如何使用 AI 按既定 SOP 处理一轮缺陷任务。

如果你现在还没决定“该点哪个 skill、该怎么开口”，先看：

- `documents/guides/development/AI技能调用一页口令卡.md`
- `documents/guides/development/AI技能调用速查手册.md`

本文档回答的问题是：

- 一轮 Bug 处理应该如何启动
- AI 在执行过程中会输出什么
- 人在什么阶段接手
- 人工本地验证、SIT 验证和禅道回填如何衔接
- 如何判断这套 SOP 在实际使用中是否顺畅

本文档是执行手册，不是正式规则真源。

正式规则以以下文档为准：

- `FE-QUALITY-006-Bug处理闭环规范.md`
- `FE-AI-008-AI-Bug处理协作规范.md`

## 2. 适用场景

适用于：

- 已从禅道拿到一轮 Bug 列表
- 需要 AI 协助完成清单化、详情分析、方案卡、实施计划和后续修复路由
- 由人工负责本地最终确认、SIT 验证与禅道回填的协作模式

不适用于：

- 纯需求开发
- 无 `bug_id` 的临时探索性问题
- 只做只读查询、不进入修复流程的场景

## 3. 使用前提

执行本指南前，建议确认以下前提：

### 3.1 项目规则已存在

仓库中已具备以下文件：

- `AGENTS.md`
- `documents/README.md`
- `documents/standards/quality/FE-QUALITY-006-Bug处理闭环规范.md`
- `documents/standards/ai-collaboration/FE-AI-008-AI-Bug处理协作规范.md`

### 3.2 本地执行层可用

如果启用了项目私有 skill，应具备：

- `/.repo-local-skills/`
- 对应的 Bug 处理私有 skill
- 私有 skill 已按规范引用正式规则，而不是替代正式规则

### 3.3 当前项目默认约定已知

当前项目默认约定如下：

- Bug 来源：禅道 `我的 Bug`
- 基础分支：`main`
- Bug 修复分支：`fix/<task_slug>`
- 主线同步策略：默认 `merge`
- 默认提交策略：按批次统一 commit
- 禅道状态更新：人工执行
- SIT 验证：人工执行

## 4. 一次标准使用流程

### 4.1 第一步：启动一轮 Bug 处理

人工输入一条启动指令，要求 AI 按 Bug SOP 进入标准处理链路。

推荐示例：

- `按 bug SOP 处理我当前禅道 bug 列表`
- `开始这一轮 bug 修复，先整理清单`
- `继续处理当前这批 bug`

启动后，AI 应先做：

- 读取当前 Bug 列表
- 生成处理前清单
- 进入 bug 详情页做深化分析
- 给出需求理解结论与拟修复方案
- 生成 Bug 修复方案卡
- 标记待测试补充项
- 给出分类与批次建议

### 4.2 第二步：详情深化分析

在判断某条 bug 是否“信息不足”之前，AI 应先完成详情深化分析。

推荐至少检查：

- bug 详情页正文
- 截图、附件、录屏
- 历史记录里的补充说明
- 当前项目里的实际页面、字段、接口与代码结构
- 若当前项目证据仍不足，再查看原型项目 `/Users/cola/Documents/gs/ar-saas` 的对应功能实现；查看前先同步最新代码，优先使用 `git pull --ff-only`

这一阶段的目标不是立刻修复，而是先判断：

- 标题和详情是否能映射到当前项目中的实际对象
- 截图是否已经提供了足够的期望行为或数值口径
- 所谓“缺信息”是不是只是列表摘要太短

只有在这轮分析后仍无法明确关键信息时，才应把该 bug 放入待测试补充清单。

### 4.3 第三步：人工先审需求理解与拟修复方案

在 AI 正式开始改代码前，人工可先核对两项内容：

- AI 是否正确理解了 bug 需求
- AI 提出的拟修复方案是否存在偏差、过改或漏改

这一步尤其适用于：

- 测试描述可能不准确
- 标题和详情存在偏差
- 设计图与现项目字段不完全一致
- 容易把 bug 修复误做成新增功能
- 实现方案涉及数据来源策略切换、固定集合替代动态列表、业务口径固化等较大变更
- 当前实现本身也合理，测试建议代表的是另一种产品口径

### 4.4 第四步：确认本轮处理范围

AI 输出处理前清单后，人工确认：

- 哪些 Bug 纳入本轮
- 哪些 Bug 暂不处理
- 哪些 Bug 因信息不足先挂起
- 哪些 Bug 虽然信息足够，但需要产品进一步确认实现口径
- 哪些 Bug 需要并单处理，哪些需要拆批次

该阶段结束后，AI 应更新：

- 处理前清单
- 待测试补充清单
- 本轮处理批次建议

### 4.5 第五步：进入 PIR 规划阶段

当某条 Bug 或某个批次已经具备：

- 足够证据
- 明确需求理解
- 可审查的修复方案卡

则不应直接开始改代码，而应先进入 `pir-planner`。

这一阶段的目标是把方案卡转成正式实施计划。

### 4.6 第六步：进入 PIR 实施阶段

当实施计划已批准后，再进入 `pir-implementer` 执行实际修改。

### 4.7 第七步：进入 PIR 评审阶段

当实现完成后，再进入 `pir-reviewer` 做独立评审。

### 4.8 第八步：开始正式处理

在范围确认后，AI 应：

- 完成分类与批次决策
- 切回 `main`
- 同步最新 `main`
- 从最新 `main` 创建本轮 `fix/<task_slug>` 分支
- 逐条进入复现、定位、修复

默认情况下，不要求人工预先切好分支。

### 4.9 第九步：人工接力业务确认

当 AI 在以下场景遇到不确定项时，应暂停等待人工确认：

- Bug 预期行为不明确
- 是否应并单存在争议
- 根因不明确且继续修改风险较高
- 当前分支状态不安全
- 分支拆分策略不明确
- 当前实现和测试建议都合理，需进一步由产品确认最终口径

人工在该阶段负责给出业务口径或处理方向。

### 4.10 第十步：AI 本地验证

AI 修复后，应先完成本地验证，包括：

- 原问题路径验证
- 同页或同模块最小必要冒烟验证
- 必要检查命令执行结果

AI 验证通过后，应将相关 Bug 状态更新为：

- `已修复待人工本地验证`

### 4.11 第十一步：人工本地验证

人工在本地执行页面操作，确认业务表现是否符合预期。

该阶段人工需要明确给出结论：

- 哪些 Bug 本地验证通过
- 哪些 Bug 本地验证未通过
- 哪些 Bug 需要进一步调整

AI 应根据人工反馈更新清单，而不是默认把本地验证视为通过。

### 4.12 第十二步：进入提交准备

当同一批次 Bug 均通过人工本地验证后，AI 才进入正式提交准备。

默认规则：

- 同一批次、同一模块、同一分支内的 Bug 统一整理提交

如果存在以下情况，可改为拆 commit 或拆分支：

- 跨模块
- 风险差异明显
- 人明确要求“一条 Bug 一个 commit”

如果当前仓库存在本地提交口径 skill（如 `commit-local`），则在真正执行 commit 或 amend commit message 前，先切换到该本地 skill 生成提交信息。提交阶段的本地约束不要反向写回共享 `.codex/skills/*`。

### 4.13 第十三步：生成处理后交付清单

在准备提交或提交完成后，AI 应输出处理后交付清单，用于：

- 人工核对修复范围
- SIT 验证
- 禅道回填

### 4.14 第十四步：SIT 验证与禅道回填

人工根据处理后交付清单：

- 上 SIT 验证
- 根据验证结果回填禅道状态和备注

AI 默认不直接执行禅道写入操作，仅提供回填建议内容。

## 5. AI 在过程中必须输出的内容

一轮标准 Bug 处理过程中，AI 至少应输出以下内容：

### 5.1 处理前清单

用于确认本轮处理范围，与禅道列表逐条对应。

### 5.2 待测试补充清单

用于承接信息不足但不阻塞整轮的 Bug。

### 5.3 本轮处理批次建议

用于说明哪些 Bug 同批处理，哪些需要拆开。

### 5.4 Bug 修复方案卡

用于承接详情分析后的实施级方案，供人工审查和 PIR 规划使用。

### 5.5 当前工作分支信息

用于说明本轮修复在哪个分支中执行。

### 5.6 处理中跟踪清单

用于说明每条 Bug 当前的处理状态、验证状态和阻塞状态。

### 5.7 处理后交付清单

用于人工本地验证、SIT 验证和禅道回填。

所有清单都必须满足：

- 保留 `bug_id`
- 保留 `bug_title`
- 顺序尽量和禅道一致
- 未处理项、阻塞项、待补充项不得消失

## 6. 人的接力点

在这套协作流程里，人工必须明确接手以下节点：

### 6.1 确认本轮处理范围

确认是否纳入本轮、是否拆批次、是否并单。

### 6.2 协调测试补充信息

对信息不足的 Bug，统一收集并反馈测试补充。

### 6.3 确认业务预期

当 AI 无法明确预期行为时，人工负责给出口径。

### 6.4 执行人工本地验证

在 AI 本地验证完成后，人工必须实际验证。

### 6.5 决定是否允许提交

人工本地验证未通过时，不应进入正式提交。

### 6.6 执行 SIT 验证

SIT 验证由人工负责，AI 只提供交付清单和建议关注点。

### 6.7 操作禅道回填

禅道状态流转、备注、关闭等动作由人工执行。

## 7. 推荐提示词用法

### 7.1 启动整轮处理

适用于从禅道当前列表开始一轮新的处理。

示例：

`按 bug SOP 处理我当前禅道 bug 列表，先生成处理前清单和待补充项。`

### 7.2 为 batch 生成方案卡

`请按 bug SOP 为 batch-1 生成 bug 修复方案卡。`

### 7.3 进入 PIR 规划

`请基于已审过的 bug 修复方案卡进入 PIR 规划。`

### 7.4 按计划进入 PIR 实施

`请按已批准计划进入 PIR 实施。`

### 7.5 对这一轮结果做 PIR 评审

`请对这轮 bug 修复结果进入 PIR review。`

### 7.6 继续当前轮次

适用于同一轮处理中断后继续。

示例：

`继续当前这轮 bug 处理，按已有清单推进下一步。`

### 7.7 人工验证后继续

适用于你已完成本地人工验证，需要 AI 接着整理提交和交付。

示例：

`我已经完成本地人工验证，请根据结果更新处理后清单并准备提交。`

### 7.8 按本地提交规范提交

适用于当前批次已经 ready to commit，且你希望 commit message 按本地私有口径生成。

示例：

`请按 commit-local 规则整理并提交这批代码。`

### 7.9 只输出交付清单

适用于修复已完成，只需要交付物。

示例：

`请按 bug SOP 输出当前这轮 bug 的处理后交付清单。`

## 8. 常见协作模式

### 8.1 整轮处理模式

适用于从列表开始，一次推进整轮处理。

### 8.2 单条 Bug 深挖模式

适用于复杂问题、高风险问题或必须单独处理的问题。

### 8.3 分批次推进模式

适用于同一轮中存在多个模块或信息完整度差异明显的 Bug。

### 8.4 人工验证后续跑模式

适用于 AI 已完成修复和本地验证，等待人工确认后继续。

## 9. 常见异常处理

### 9.1 Bug 描述不清怎么办

AI 不应直接因为标题或摘要短就挂起。应先进入详情页、看截图或附件，并结合当前项目做一轮映射分析；只有在这轮分析后仍无法判断，才标记为 `待测试补充`。

### 9.2 Bug 无法复现怎么办

AI 应保留该 Bug 在清单中，并标记为 `阻塞` 或 `待进一步确认`，不得静默跳过。

### 9.3 测试给了明确改动意见，但当前实现也合理怎么办

这类问题不应简单归类为信息不足。AI 应将其标记为 `待产品确认`，输出当前实现合理性说明、测试建议口径、拟修改实现策略和待确认问题点，等待产品或业务口径确认后再决定是否实施。

### 9.4 几条 Bug 看起来像同一个问题怎么办

AI 可建议并单处理，但应先明确记录批次建议，并由人工确认。

### 9.5 当前分支不干净怎么办

AI 可以先继续完成清单化、信息审查和批次建议，但在进入分支准备阶段前必须停下，先反馈当前分支和工作区状态。默认做法是先把工作区处理干净，再从最新 `main` 继续推进。

### 9.6 人工本地验证未通过怎么办

AI 不应进入正式提交，而应回到修复链路继续处理，并更新处理中跟踪清单。

### 9.7 SIT 验证失败怎么办

人工应将失败结果反馈给 AI；AI 应基于处理后清单回到问题定位和修复阶段重新推进。

## 10. 如何判断这套 SOP 是否好用

以下问题可作为可用性检查项：

- 启动时是否只需要一句简短提示词
- AI 是否会先输出清单，而不是直接修改代码
- 信息不足的 Bug 是否会被单独挂起，不阻塞整轮
- AI 是否能从最新 `main` 正确准备分支
- 处理后交付清单是否能和禅道逐条对账
- 人工本地验证、SIT 和禅道回填是否有明确交接点
- 中途中断后是否能从已有清单继续，而不需要重讲全部上下文

如果以上多项无法满足，说明 SOP 仍需继续迭代。
