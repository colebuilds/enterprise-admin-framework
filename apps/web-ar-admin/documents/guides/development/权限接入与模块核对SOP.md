# 权限接入与模块核对SOP

## 1. 文档目的

本手册用于统一回答当前项目里“权限码怎么从接口层落到前端页面”和“如何系统排查一个模块的权限接线是否完整”这两类问题。

这份文档重点解决的是：

- 后端或权限树里已经有权限码，但前端没有消费
- 前端消费了错误层级的权限码
- 页面只接了页面入口权限，没有接按钮、操作列、批量操作等子权限
- UI 做了隐藏或禁用，但点击 handler 没有做兜底
- UI 已经做了权限控制，但数据请求、预加载、自动刷新或轮询没有同步收口
- 同模块内不同页面的权限口径不一致

规则层面的页面通用规范，仍以 `docs/page-development-guide.md` 为准；本手册重点解释“当前仓库权限链路怎么跑”和“模块级权限核对应该怎么做”。

## 2. 当前仓库权限运行时链路

### 2.1 权限数据源

当前前端权限的源头是用户信息接口返回的菜单树。

核心入口：

- `src/store/modules/user.ts`

当前实现要点：

- 前端会把后端返回的 `menus` 做树拍平
- 读取每个节点的 `authCode`
- 统一转小写后写入 `userStore.permissions`
- 本地开发域名场景会额外补一个 `localhost` 权限

这意味着当前仓库里几乎所有权限判断，本质上都在消费同一份拍平后的权限码集合。

### 2.2 路由与菜单过滤

核心入口：

- `src/router/generator.ts`
- `src/router/guards.ts`

当前实现要点：

- 页面路由权限写在路由 `meta.permissions`
- 动态路由生成时，会用当前用户权限过滤 `asyncRoutes`
- 多个权限码按“命中任一项即可”的 OR 语义处理
- 如果用户手输一个受控路由但没有权限，守卫会跳转到 `403`

所以路由层权限主要回答的是：

- 这个页面入口能不能进
- 这个菜单要不要显示

### 2.3 页面内权限消费入口

当前仓库常见的权限消费方式分为以下几类：

1. 路由 / 页面入口：`meta.permissions`
2. 静态按钮：`v-permission`
3. 动态区块 / tab / layout：`hasPermission()`
4. 表格行操作：`auth`
5. 批量操作：`batchActions.auth`
6. 组件内部禁用态：`permission` prop
7. 高风险动作：handler 内二次权限兜底

相关实现入口：

- `src/hooks/web/usePermission.ts`
- `src/directives/permission.ts`
- `src/components/Table/src/createActionColumn.ts`
- `src/components/Table/src/components/TableAction.vue`
- `src/components/ProComponents/ProCrudTable/ProCrudTable.vue`
- `src/components/ConfirmSwitch/index.vue`

### 2.4 当前仓库的关键语义

执行模块核对时，至少要记住以下语义：

- 权限判断统一按小写比较
- 多个权限码通常是 OR 关系，不是 AND 关系
- `v-permission` 默认是“无权限直接移除 DOM”
- 表格操作列不仅会看 `auth`，还会叠加 `ifShow`
- 批量操作和组件内开关并不是只靠 `v-permission`

## 3. 前置门槛与执行状态

### 3.1 前置阻塞

以下任一情况成立时，本 SOP 必须进入 `前置阻塞`，不能降级继续：

- 没有真实权限树返回体
- 没有可用 `token` 去拉真实权限树
- 没有可登录的账号 / 密码 / 环境信息去获取真实权限树

进入 `前置阻塞` 后，AI 必须明确提示用户补齐以下至少一项：

1. 真实 `menus/authCode/menuType/children` 返回体
2. 可用 `token`
3. 可登录的账号 / 密码 / 环境信息

后台角色勾选截图、口头描述、页面截图只能作为辅助证据，不能替代真实权限树数据。

### 3.2 权威核对

`权威核对` 指的是：

- 已有代码
- 且已经拿到真实权限树返回体
  - 可以是用户直接提供
  - 也可以是基于可用 `token`
  - 或基于可登录账号 / 密码 / 环境信息现取

只有进入 `权威核对`，才允许输出：

- 已确认漏接
- 已确认错接
- 已确认部分接入
- 已确认权限层级使用错误

### 3.3 为什么不能再做静态降级

只看代码时，前端最多只能确认“当前页面写了哪些权限判断”，但无法可靠确认：

- 后端到底有没有定义这个权限码
- 这个权限码是不是目标账号实际可见
- 某个按钮应不应该显示、禁用，还是仅依赖接口拦截
- 同一个模块里不同角色的真实菜单树结构是否一致

因此，这套 SOP 不再支持“没有真实权限树时继续做静态预审”的降级路径。

## 4. 模块级权限核对流程

### 4.1 先圈定模块范围

至少明确一个真实目标：

- 模块目录，例如 `src/views/finance/withdrawOrder/dashboard`
- 页面文件，例如 `src/views/finance/withdrawOrder/index.vue`
- 或明确的业务页面名称

如果用户没有给路径，但上下文能明确模块，也可以继续。

### 4.2 收集前置数据

建议优先收集以下信息：

1. 模块代码路径
2. 路由声明文件
3. 真实权限树返回体 `menus/authCode/menuType/children`
4. 可用 `token`，可用于拉到目标账号真实权限树
5. 或可登录账号 / 密码 / 环境信息
6. 后台角色勾选截图（辅助）
7. 期望行为与当前异常描述
8. 若要验证行为，至少准备一组有限权限账号

如果第 3、4、5 项都不具备，本 SOP 必须中断并提示用户配合补数据。

### 4.3 先盘点权限码

盘点时至少区分以下层级：

- 页面入口权限
- 查询列表权限
- 操作权限
- 导出权限
- 日志权限
- 详情权限
- 批量操作权限

如果拿到了真实权限树，还要明确 `menuType` 语义，避免把：

- 模块
- 页面
- 按钮 / 操作

混成同一层处理。

### 4.4 再盘点页面交互点

一个模块内至少检查以下交互点：

- 顶层路由与 tab
- 查询、重置、刷新
- 普通按钮
- 表格操作列
- 批量操作
- 开关类组件
- 弹窗入口
- 弹窗内部操作
- 卡片点击
- 二级 tab
- 动态下拉项
- handler 内真实触发接口的高风险动作
- `onMounted`、`watch`、轮询、自动刷新、默认查询、provide/inject 上下文初始化等自动数据请求入口

### 4.5 建立映射表

每个交互点都应落成一条映射：

| 交互点   | 期望权限码    | 当前消费方式 | 当前表现 | 结论     |
| -------- | ------------- | ------------ | -------- | -------- |
| 批量转派 | `xxx:Operate` | 未接         | 始终显示 | 疑似漏接 |

对涉及接口请求的交互点，还要额外确认：

- UI 不可见或不可操作时，对应请求是否仍会在 `onMounted`、`watch`、轮询、自动刷新或预加载阶段触发

其中“结论”字段默认按权威核对口径输出：

- 已确认漏接
- 已确认错接
- 已确认部分接入
- 已确认缺少 handler 兜底

### 4.6 问题分类口径

建议统一用以下分类，不要笼统写“权限有问题”：

1. 有权限码，但前端完全未消费
2. 有消费，但用错层级
3. 页面只消费了一部分
4. UI 做了隐藏 / 禁用，但 handler 没做兜底
5. UI 已做权限控制，但数据请求权限没有同步，仍会越权触发接口
6. 前端使用了不存在或过时的权限码
7. 同模块内不同页面权限口径不一致

## 5. 当前仓库推荐处理方式

### 5.1 路由与页面入口

页面能否进入、菜单能否显示，优先使用：

- `meta.permissions`

不要把按钮权限直接塞到路由入口层，也不要用一个 `GetPageList` 权限兜住整页所有操作。

### 5.2 静态按钮

普通按钮优先使用：

- `v-permission`

默认语义是无权限直接移除。只有产品明确要求“保留按钮但禁用”时，才考虑使用禁用效果或组件级禁用方案。

### 5.2A UI 与数据权限联动

权限核对不能只停留在“UI 有没有隐藏 / 禁用”。

对任何受控区域，还要同步检查：

- `onMounted`
- `watch`
- 轮询 / 自动刷新
- 默认列表查询
- provide/inject 上下文初始化
- 跨 tab 或卡片跳转触发的预加载

如果 UI 已经不可见，但这些数据请求仍会执行，应视为明确权限问题，而不是“只是体验问题”。

### 5.3 动态结构

以下结构优先在数据构造阶段过滤，而不是渲染后补 `v-if`：

- tab
- 操作列数组
- 批量操作数组
- 动态按钮组
- 动态卡片项

对应常见方式：

- `hasPermission()`
- `auth`
- `batchActions.auth`

### 5.4 表格操作列

表格操作列要同时区分两层：

1. 权限是否允许显示
2. 业务条件是否允许显示

也就是同时关注：

- `auth`
- `ifShow`

不能只写 `ifShow`，也不能只写 `auth` 后忽略业务条件。

### 5.5 组件内禁用态

像状态开关、确认开关这类组件，优先看组件本身是否支持：

- `permission` prop

不要把这类场景全部退化成外围 `v-if`。

### 5.6 高风险操作

对于真实发起接口、改状态、批量处理、转派、审核等动作，只做 UI 层隐藏不够。

建议至少满足两层：

1. UI 层不展示或禁用
2. handler 内再次做权限兜底，避免旧状态、缓存状态或异常交互绕过前端显示层

## 6. 常见误判与注意事项

### 6.1 不要把页面权限和查询权限混为一层

常见误判是把：

- 页面可进入
- 页面可查询
- 页面可操作

都绑定到一个 `GetPageList` 或“查询列表”权限上。

这会导致页面入口控制正常，但按钮和操作列全部失控。

### 6.2 不要只搜 `v-permission`

当前仓库的权限消费并不只在模板指令里。

如果只搜 `v-permission`，很容易漏掉：

- `hasPermission()`
- `auth`
- `batchActions.auth`
- `permission` prop
- handler 内兜底

### 6.2A 不要只看 UI 显示结果

当前仓库里，很多数据请求并不一定挂在按钮点击上，还可能出现在：

- `setup` 期间的共享上下文初始化
- `onMounted`
- `watch`
- 轮询 / 自动刷新
- 表格默认首屏查询

所以“页面上看不到按钮了”不等于“权限已经接对了”，还要确认无权限时接口是否真的不再请求。

### 6.3 不要只修截图暴露点

一个模块里，如果某个 tab 暴露出权限漏接，通常要顺带检查：

- 相邻 tab
- 同页批量操作
- 详情弹窗
- 二级 tab
- 卡片入口

否则很容易修掉一个点，漏掉同模块同模式的其它入口。

### 6.4 没有真实权限树就直接停

如果没有真实权限树，也拿不到可用 `token` 或账号 / 密码 / 环境信息，本 SOP 必须停在“前置阻塞”。

这不是保守建议，而是执行门槛。

## 7. 输出与文档落点

### 7.1 盘点输出建议

模块权限核对至少回报：

1. 本次执行状态：`前置阻塞` 或 `权威核对`
2. 真实权限树数据来源
3. 模块权限码清单
4. 页面交互点清单
5. 权限码与交互点映射表
6. 问题分类结果
7. 建议修复方式
8. 是否存在“UI 权限与数据请求权限不同步”

### 7.2 文档落点

长期可复用的权限说明、模块级样板、接入规则说明，默认落到：

- `documents/guides/development/`

单次排查记录、会话型执行清单、临时核对结果，默认落到：

- `/.local-work/`

### 7.3 与本地 skill 的关系

本手册是正式的人类与 AI 共用规则源。

本地私有 skill：

- `/.repo-local-skills/module-permission-onboarding-sop/SKILL.md`

负责把这份正式规则路由成一次可执行流程。

如果两者发生冲突，应优先修正文档和 skill，使它们重新一致，而不是长期容忍双口径。

## 8. 日常使用方式

### 8.1 一句话 AI 提示词

如果只想直接复制模板，不想从本手册手动摘句子，优先查看：

- `documents/guides/development/权限核对AI提示词模板.md`

推荐直接使用下面这句，把 `[模块/页面路径]` 换成真实目标：

```text
请按本仓库权限 SOP 为 [模块/页面路径] 做模块 / 页面权限核对。要求：先检查真实 `menus/authCode/menuType/children` 数据是否就绪；如果没有真实权限树返回体，也拿不到可用 token 或账号 / 密码 / 环境信息去拉数据，直接中断并告诉我还缺什么，不允许降级成静态预审；拿到真实权限树后，再盘点模块已有权限码与 menuType 层级，继续核对页面里的路由、Tab、按钮、表格操作列、批量操作、开关类组件、弹窗入口、高风险 handler，以及 `onMounted`、`watch`、预加载、自动刷新、轮询、默认查询这类自动数据请求入口，建立“权限码 -> 消费方式 -> 当前表现 -> 结论”的映射表；检查权限消费时不要只搜 v-permission，还要覆盖 hasPermission、auth、batchActions.auth、permission prop、handler 二次校验，并明确判断是否存在 UI 权限与数据请求权限不同步；保持结论收敛，不要顺手重构无关代码。请先给出受影响文件清单和执行步骤，再开始排查；完成后汇报执行状态、真实权限树数据来源、权限码清单、交互点清单、映射表、问题分类结果、建议修复方式、风险点，以及是否同步更新了相关文档。
```

### 8.2 本地执行模板

如果要把一次排查沉淀成本地执行记录，优先复用：

- `/.local-work/permission-audits/_template-前置阻塞.md`
- `/.local-work/permission-audits/_template-权威核对.md`

前者用于缺少真实权限树、需要向用户追要账号 / 密码 / token / 返回体的阻塞记录；后者用于已经拿到真实权限树后的最终问题单。
