# AI子任务下发示例

## 1. 目标

本文档用于说明：当主代理决定拆分复杂任务时，应该如何给子代理下发清晰、可执行、可收口的任务。

如果你现在还没决定“该先看哪个入口、要不要拆子任务”，先看：

- `documents/guides/development/AI技能调用一页口令卡.md`
- `documents/guides/development/AI技能调用速查手册.md`

---

## 2. 适用场景

适用于以下情况：

- 任务涉及多个模块或多个目录
- 可以并行做只读盘点
- 存在明确的文件边界可分配
- 主代理需要降低长时间卡住的风险

不适用于以下情况：

- 只改 1 到 2 个文件
- 单一关键决策尚未形成
- 多个子任务必然频繁修改同一文件

---

## 3. 下发前主代理必须先做什么

在给子代理下发任务前，主代理至少先明确：

- 总目标是什么
- 本次不包含什么
- 哪些文件允许改
- 哪些文件禁止改
- 共享入口文件由谁负责
- 子代理是只读盘点，还是进入实际实施

若任务尚未收到“明确授权语义”（不强依赖固定口令），则子代理只能做只读分析，不得进入实际落盘阶段。

---

## 4. 标准下发模板

推荐使用以下模板：

```text
目标：
负责文件：
禁止修改：
任务类型：只读盘点 / 实施修改 / 校验收口
输出内容：
发现以下情况必须返回主代理：
```

推荐的“必须返回主代理”场景包括：

- 需要修改计划外文件
- 发现边界与其他子任务冲突
- 需要新增命令或新增权限
- 发现本任务本质上变成架构级决策

---

## 5. 最小示例

假设目标是整理 `packages/utils/`，并按职责拆成 `time/`、`amount/`、`core/`。

### 5.1 主代理保留项

- 统一目标和边界
- 决定最终目录结构
- 保留总导出文件与总 README
- 最终汇总结果

### 5.2 子代理 A：时间工具盘点

```text
目标：盘点 packages/utils 中应归入 time/ 的工具
负责文件：packages/utils/** 中与日期、时间、duration、timezone 相关文件
禁止修改：总导出文件、amount 相关文件、core 相关文件
任务类型：只读盘点
输出内容：候选文件清单、重复能力、建议归类、风险点
发现以下情况必须返回主代理：边界不清、需要改文件、和 amount/core 发生重叠
```

### 5.3 子代理 B：金额工具盘点

```text
目标：盘点 packages/utils 中应归入 amount/ 的工具
负责文件：packages/utils/** 中与金额、精度、格式化、货币相关文件
禁止修改：总导出文件、time 相关文件、core 相关文件
任务类型：只读盘点
输出内容：候选文件清单、金额语义边界、精度风险
发现以下情况必须返回主代理：边界不清、需要改文件、和 time/core 发生重叠
```

### 5.4 子代理 C：引用影响分析

```text
目标：分析 packages/utils 拆分后可能影响的 import
负责文件：所有引用 packages/utils 的调用方，仅做只读检查
禁止修改：任何实现文件
任务类型：只读盘点
输出内容：高风险 import 清单、迁移顺序建议、最容易出错的入口
发现以下情况必须返回主代理：发现共享入口必须同步改动、影响范围超出预期
```

### 5.5 主代理收口

主代理收到结果后再决定：

- 最终目录边界
- 哪些文件进入实施阶段
- 哪些文件继续由主代理保留
- 是否需要进入 `skills/ai-change-control/SKILL.md` 的实际修改流程

---

## 6. 常见错误

- 一上来就让多个子代理同时改同一个入口文件
- 在边界未明确前直接进入并发实施
- 让子代理自己决定是否扩大范围
- 没有要求子代理用结构化格式回报

---

## 7. 与 FE-AI-007 协议对齐（推荐）

复杂任务建议先按 `FE-AI-007-AI分阶段执行与连续交付协议` 建立阶段，再在阶段内下发子任务。

最小实践：

- 一个子任务对应一个执行单元（EU）
- 子任务完成后必须给出验证证据
- 并行子任务收敛后再统一阶段验收
- 阶段验收通过后再做一次阶段提交

## 8. 最终原则

好的子任务下发，不是把大任务切碎，而是：

- 先定边界
- 再分文件所有权
- 先并发盘点
- 后并发实施
- 最终仍由主代理统一收口
