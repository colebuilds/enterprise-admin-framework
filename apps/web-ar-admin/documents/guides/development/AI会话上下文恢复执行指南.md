# AI会话上下文恢复执行指南

## 1. 文档目的

本文档用于说明：

**当一个任务需要跨 session 继续时，应该如何生成交接快照、如何恢复现场、如何继续进入后续流程。**

如果你现在还没决定“该先看哪个入口、该怎么继续当前任务”，先看：

- `documents/guides/development/AI技能调用一页口令卡.md`
- `documents/guides/development/AI技能调用速查手册.md`

本文档只讲使用方式，不作为规则真源。规则真源由 `FE-AI-009-AI会话上下文恢复规范.md` 负责。

## 2. 最简单的使用方式

如果只想记住一套最小动作，记下面这条就够了：

### 切 session 前

1. 在稳定检查点生成交接快照
2. 更新 `/.local-work/session-context/LATEST_PATH.txt`
3. 结束当前 session

### 新 session 恢复时

1. 读取 `/.local-work/session-context/LATEST_PATH.txt`
2. 读取它指向的快照文件
3. 执行 `git status --short`
4. 执行 `git rev-parse --abbrev-ref HEAD`
5. 根据快照里的“待办与下一步”继续

## 3. 什么时候该生成交接快照

建议只在**稳定检查点**生成。

推荐时机：

- 需求结论已经确认
- 方案已经确认
- 实施计划已经批准
- 一个 batch 已经完成
- 准备切 session 前

不建议在这些时候生成：

- 代码刚改到一半
- 需求还在摇摆
- 当前方案还没审过
- 还不知道下一步该做什么

判断标准很简单：

**如果此刻能清楚写出“下一步是什么”，就适合生成快照。**

## 4. 普通需求怎么用

### 4.1 切 session 前

可以直接说：

- `生成当前任务交接快照`
- `更新当前任务恢复快照`

AI 应生成一份快照，并写清：

- 当前任务目标
- 当前分支
- 已完成事项
- 当前改动文件
- 已做校验
- 下一步动作

### 4.2 新 session 恢复时

可以直接说：

- `先读取最新交接快照，恢复当前任务现场`

AI 应先做：

1. 读取 `LATEST_PATH.txt`
2. 读取目标快照
3. 查看 `git status --short`
4. 查看当前分支
5. 再根据快照继续任务

## 5. Bug 任务怎么用

Bug 任务恢复的顺序是：

**先恢复通用任务现场，再进入 `bug-sop`。**

### 5.1 切 session 前

当 bug 任务已经稳定到以下任一状态时，适合生成快照：

- intake 已完成
- batch 已确认
- 方案卡已完成
- 一个 batch 已完成
- 准备切 session 前

### 5.2 新 session 恢复时

可以说：

- `先恢复当前任务，再按 bug SOP 继续`
- 或者：
- `按 bug SOP 继续当前任务，先读取最新交接快照`

AI 应先恢复：

- 当前 bug 范围
- 当前 batch
- 当前是否有待补充项
- 当前是否已有方案卡
- 当前是否已进入 PIR

然后再进入 `bug-sop`。

## 6. PIR 阶段怎么用

PIR 不负责恢复现场，只负责继续执行某个阶段。

### 6.1 切 session 前

如果当前任务已经进入：

- `pir-planner`
- `pir-implementer`
- `pir-reviewer`

那么交接快照里必须明确写：

- 当前 PIR 阶段
- 当前已批准计划
- 当前阻塞点
- 下一步入口 skill

### 6.2 新 session 恢复时

可以说：

- `先恢复当前任务，再继续 PIR 阶段`

AI 应先恢复：

- 当前处于哪个 PIR 阶段
- 是否已有已批准计划
- 是否允许继续实施
- 当前是否应转入 review 或 commit

然后再进入：

- `pir-planner`
- `pir-implementer`
- `pir-reviewer`

## 7. 提交阶段怎么用

提交阶段也建立在通用恢复底座之上。

### 7.1 切 session 前

如果任务已经 ready to commit，快照里要明确写：

- 当前是否允许提交
- 当前建议提交范围
- 当前验证是否通过
- 当前应进入哪个提交 skill

### 7.2 新 session 恢复时

恢复后，如果任务已经到提交阶段，再进入：

- `commit-local`

可以说：

- `先恢复当前任务，再按 commit-local 规则提交`

## 8. 快照里必须写什么

一份通用交接快照，最少要写这 8 项：

1. 会话目标
2. 当前工作区状态
3. 本轮已完成事项
4. 本轮改动文件清单
5. 校验与结果
6. 待办与下一步
7. 关联文档
8. `/new` 后恢复步骤

如果是 bug / PIR / 调试任务，再补对应的场景附加块。

## 9. 推荐触发语句

### 9.1 生成快照

- `生成当前任务交接快照`
- `更新当前任务恢复快照`

### 9.2 恢复任务

- `先读取最新交接快照，恢复当前任务现场`
- `恢复当前任务上下文`

### 9.3 恢复 bug 任务

- `先恢复当前任务，再按 bug SOP 继续`

### 9.4 恢复 PIR 阶段

- `先恢复当前任务，再继续 PIR 阶段`

### 9.5 恢复后提交

- `先恢复当前任务，再按 commit-local 规则提交`

## 10. 常见错误

### 10.1 不生成快照，直接切 session

结果是下次只能翻旧聊天，恢复效率低。

### 10.2 在不稳定阶段生成快照

结果是快照只记录了半成品状态，下一次还是接不上。

### 10.3 把 `bug-sop` 当恢复入口

这是分层错误。正确顺序是：

- 先恢复任务
- 再进入 `bug-sop`

### 10.4 只写背景，不写下一步

没有“待办与下一步”的快照，恢复价值很低。

### 10.5 一个快照混多个任务

会导致任务边界不清，恢复后还得重新拆。

## 11. 如何判断这套机制是否好用

如果恢复机制是好用的，新 session 应该满足：

- 不需要翻整段旧聊天
- 能立刻知道当前任务目标
- 能立刻知道当前分支和工作区状态
- 能立刻知道上次停在哪一步
- 能立刻知道下一步进入哪个流程

如果这些做不到，就说明恢复设计还不够好。

## 12. 当前推荐工作流

建议统一按下面这条链使用：

1. 任务做到稳定检查点
2. 生成交接快照
3. 更新 `LATEST_PATH.txt`
4. 新 session 先恢复通用现场
5. 再按任务类型进入：
   - `bug-sop`
   - `pir-planner`
   - `pir-implementer`
   - `pir-reviewer`
   - `commit-local`
