# 权限核对 AI 提示词模板

## 1. 目标

本文档用于提供一套可直接复制给 AI 的提示词模板，减少“模块已有权限码，但前端没有正确消费”这类任务里反复补充上下文的成本。

适用目标：

- 为一个模块做权限权威核对
- 盘点一个模块有哪些权限码
- 检查权限码是否已经在路由、Tab、按钮、表格操作列、批量操作、弹窗和 handler 里正确消费

如果你现在还没决定“该点哪个 skill、该怎么开口”，先看：

- `documents/guides/development/AI技能调用一页口令卡.md`

完整规则与背景说明，以：

- `documents/guides/development/权限接入与模块核对SOP.md`

为准。

---

## 2. 默认使用方式

推荐直接把下面模板中的 `[模块/页面路径]` 替换成真实目标后发送给 AI。

优先填写：

- 模块目录路径，例如 `src/views/finance/withdrawOrder/dashboard`
- 或页面文件路径，例如 `src/views/finance/withdrawOrder/index.vue`

如果任务跨多个页面，但仍属于一个业务模块，优先写模块目录，不要只写业务名。

## 3. 最终标准提示词

```text
请按本仓库权限 SOP 为 [模块/页面路径] 做模块 / 页面权限核对。要求：先检查真实 `menus/authCode/menuType/children` 数据是否就绪；如果没有真实权限树返回体，也拿不到可用 token 或账号 / 密码 / 环境信息去拉数据，直接中断并告诉我还缺什么，不允许降级成静态预审；拿到真实权限树后，再盘点模块已有权限码与 menuType 层级，继续核对页面里的路由、Tab、按钮、表格操作列、批量操作、开关类组件、弹窗入口和高风险 handler，建立“权限码 -> 消费方式 -> 当前表现 -> 结论”的映射表；检查权限消费时不要只搜 v-permission，还要覆盖 hasPermission、auth、batchActions.auth、permission prop、handler 二次校验，以及 `onMounted`、`watch`、预加载、自动刷新、轮询、默认查询这类自动数据请求入口，明确检查是否存在 UI 权限与数据请求权限不同步；保持结论收敛，不要顺手重构无关代码。请先给出受影响文件清单和执行步骤，再开始排查；完成后汇报执行状态、真实权限树数据来源、权限码清单、交互点清单、映射表、问题分类结果、建议修复方式、风险点，以及是否同步更新了相关文档。
```

## 4. 极短版提示词

```text
按仓库权限 SOP 核对 [模块/页面路径] 的权限消费：先确认是否拿到了真实权限树；没有返回体、token 或账号 / 密码时就中断并告诉我缺什么；拿到数据后再查路由、Tab、按钮、操作列、批量操作、弹窗、handler，以及 `onMounted` / `watch` / 预加载 / 自动刷新这类自动请求入口，并给出映射表、修复建议和风险。
```

适用场景：

- 你已经确认过模块边界
- 你不需要 AI 再重复解释基本规则
- 你只想快速发起一轮标准权限排查

## 5. 可选追加要求

根据任务场景，可以在标准提示词后追加下面任意一条。

### 5.1 缺数据时必须中断

```text
如果你拿不到真实权限树返回体，也拿不到可用 token 或账号 / 密码 / 环境信息，请直接中断，并明确告诉我还缺哪一项数据，不要降级成静态预审继续做。
```

### 5.2 直接做权威核对

```text
我会提供真实 menus/authCode/menuType/children 返回体，或者提供可用 token / 账号密码让你去拉数据；请按权威核对模式输出最终问题单，不要停留在静态猜测。
```

### 5.3 先盘点再实施

```text
先只做盘点，不要立即改代码；先输出权限码清单、页面交互点清单、映射表和风险点，等我确认后再实施。
```

### 5.4 强制补 handler 兜底判断

```text
如果发现高风险动作只做了 UI 隐藏或禁用，请把 handler 二次权限兜底也纳入建议修复范围。
```

### 5.4A 强制检查 UI 与数据权限联动

```text
如果发现 UI 已经做了权限控制，请继续检查对应的数据请求是否也被同一权限收口，尤其是 onMounted、watch、预加载、自动刷新、轮询和默认列表查询；不要只因“按钮不显示了”就判定权限已接对。
```

### 5.5 只允许最小改动

```text
保持改动收敛，不要顺手重构无关代码，不要改变现有业务交互和样式结构。
```

## 6. 直接示例

### 6.1 dashboard 前置阻塞

```text
请按本仓库权限 SOP 为 src/views/finance/withdrawOrder/dashboard 做模块 / 页面权限核对。当前没有真实权限树返回体，也没有可用 token 或账号 / 密码 / 环境信息，因此请直接中断，并明确告诉我还缺哪一项数据，不能降级成静态预审继续做。等数据补齐后，再继续做权威核对。
```

### 6.2 dashboard 权威核对

```text
请按本仓库权限 SOP 为 src/views/finance/withdrawOrder/dashboard 做模块 / 页面权限权威核对。我会提供真实 menus/authCode/menuType/children 返回体，或者提供可用 token / 账号密码让你去拉数据；请基于这些证据盘点模块权限码，并核对页面里的路由、Tab、按钮、表格操作列、批量操作、弹窗、高风险 handler，以及 onMounted、watch、预加载、自动刷新、轮询、默认查询这类自动数据请求入口是否都正确消费；建立“权限码 -> 消费方式 -> 当前表现 -> 结论”的映射表，并输出已确认漏接、已确认错接、已确认部分接、缺少 handler 兜底，以及 UI 权限与数据请求权限不同步的问题单。请先给出受影响文件清单和执行步骤，再开始排查；完成后汇报执行状态、真实权限树数据来源、最终问题分类、建议修复顺序、验证建议和文档更新情况。
```
