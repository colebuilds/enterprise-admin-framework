# 项目上手路径指南

## 1. 目标

为新成员和 AI 提供一条最短、最稳、最不容易误读的项目上手路径，帮助进入仓库后先建立正确的项目认知，再逐步进入规范、参考资料和具体实现，不要一上来就全量扫文件。

本文档回答“第一次进入这个项目应该先看什么、后看什么、理解顺序是什么”，不重复 `reference` 的完整内容，也不重复总入口文档本身的全部正文。

## 2. 适用范围

本指南适用于：

- 第一次进入本仓库的新成员
- 第一次接触本仓库的 AI
- 从其他项目迁移过来的协作者
- 需要快速建立项目认知的临时协作者

本指南主要覆盖：

- 进入项目后的阅读顺序
- 文档优先级
- 先理解什么、后理解什么
- 哪些内容不应一开始就展开

本指南不直接覆盖：

- 具体开发规范正文
- 具体联调、测试或排障流程
- 具体代码实现细节

## 3. 使用前提

开始上手前，先接受两个前提：

- 这个项目不是单一站点工程，而是一个具有多租户、多主题、多语言、多风格和多终端协作特征的 monorepo
- 不同目录里的文档不是同等优先级，必须先按入口和主题顺序阅读

如果没有先建立这两个前提，后面读到的细节很容易被误解成最终架构或唯一做法。

## 4. 标准操作步骤

### 4.1 先看项目总入口

第一步先看：

- [`AGENTS.md`](../../../AGENTS.md)
- [`documents/README.md`](../../README.md)

目的不是一次读完所有细节，而是先确认：

- 这个仓库的文档体系怎么分层
- 哪些目录是入口，哪些是正文
- 哪些文档属于当前必须遵守的规则

### 4.2 再看文档规范与协作规范

建议第二步阅读：

- [`documents/standards/collaboration/文档编写与元数据规范.md`](../../standards/collaboration/文档编写与元数据规范.md)
- [`documents/standards/ai-collaboration/FE-AI-000-AI协作开发规范.md`](../../standards/ai-collaboration/README.md)
- [`documents/standards/ai-collaboration/FE-AI-001-AI变更控制协议.md`](../../standards/ai-collaboration/FE-AI-001-AI变更控制协议.md)
- [`documents/standards/ai-collaboration/FE-AI-003-AI子任务协作与主代理协调规范.md`](../../standards/ai-collaboration/FE-AI-003-AI子任务协作与主代理协调规范.md)

这一组主要帮助你理解：

- 文档怎么写
- AI 怎么协作
- 什么时候能改、什么时候不能改
- 复杂任务怎么拆分和收口

### 4.3 先看 reference，再看 standards

建议第三步先看参考文档，再看规范文档：

- [`documents/reference/FE-REF-001-项目概览与技术栈地图.md`](../../reference/README.md)
- [`documents/reference/FE-REF-002-业务维度与变体体系说明.md`](../../reference/README.md)
- [`documents/reference/FE-REF-003-终端形态与运行模式说明.md`](../../reference/README.md)
- [`documents/reference/FE-REF-004-当前工程结构与模块地图.md`](../../reference/README.md)
- [`documents/reference/FE-REF-005-多租户目标架构与边缘识别说明.md`](../../reference/README.md)
- [`documents/reference/FE-REF-006-现状到目标架构的演进说明.md`](../../reference/README.md)

这一组先帮助你建立“项目现在是什么、目标是什么、正在怎么演进”的认知。

在理解项目现状之后，再按主题进入规范层：

- `documents/standards/coding/`
- `documents/standards/architecture/`
- `documents/standards/api/`
- `documents/standards/data/`
- `documents/standards/ui/`
- `documents/standards/infrastructure/`
- `documents/standards/quality/`

### 4.4 再看 guides，最后进入具体执行

在已经理解现状和规则后，再看操作型手册：

- `documents/guides/testing/项目测试执行指南.md`
- `documents/guides/debugging/项目调试入口指南.md`
- `documents/guides/integration/接口与宿主联调指南.md`

如果你要开始真正改动，建议按这个顺序串联：

1. 看规则和背景
2. 看对应规范
3. 看对应操作指南
4. 再进入具体代码或文档任务

如果已经不是“第一次进入项目”，而是已经拿到具体开发任务，继续参考 `documents/guides/development/规范落地执行指南.md`，本指南只负责首次进入项目时的阅读顺序与理解路径。

## 5. 常见场景

### 5.1 新成员第一次进入项目

建议顺序：

1. `AGENTS.md`
2. `documents/README.md`
3. `documents/reference/*`
4. `documents/standards/collaboration/*`
5. `documents/standards/ai-collaboration/*`
6. 具体标准目录
7. 对应 `guides/*`

### 5.2 AI 第一次进入项目

建议顺序：

1. `AGENTS.md`
2. `documents/README.md`
3. `documents/standards/collaboration/文档编写与元数据规范.md`
4. `documents/standards/ai-collaboration/*`
5. `documents/reference/*`
6. 相关 `standards/*`
7. 相关 `guides/*`

### 5.3 需要快速接手一个具体任务

建议顺序：

1. 先看任务相关目录的 `README.md`
2. 再看对应 `standards`
3. 再看对应 `guides`
4. 最后回到代码或正文

### 5.4 只想快速理解项目大方向

建议优先看：

- `documents/reference/FE-REF-001-项目概览与技术栈地图.md`
- `documents/reference/FE-REF-002-业务维度与变体体系说明.md`
- `documents/reference/FE-REF-005-多租户目标架构与边缘识别说明.md`
- `documents/reference/FE-REF-006-现状到目标架构的演进说明.md`

## 6. 常见错误

- 一上来就全量扫描所有文档
- 先看正文，不先看入口
- 把 `reference` 当成规范
- 把 `standards` 当成项目现状说明
- 把 `guides` 当成规则正文
- 忽略 `AGENTS.md` 里的入口与授权要求
