# Harness System 全量测试方案

## 1. 目标

本方案用于为当前 Harness System 设计一批可重复执行的全量测试用例，重点验证：

- 对象链路是否闭环
- 主控架构是否稳定
- 口令行为是否一致
- 状态、转场、编排是否符合既定规则
- 运行时记录是否足以支撑恢复与复盘

本方案先解决“测什么、怎么测、先测哪批”，不直接实现自动化测试平台。

## 2. 版本绑定

- 当前适用版本：`0.1.0`
- 适用范围：
  - 当前 README 中已声明的最小可运行能力
  - 当前已落地口令与状态规则
- 不适用范围：
  - 尚未进入 README 或技术方案的未来能力
  - 仅在讨论中出现、但未形成正式规则的想法

### 2.1 版本变更时的用例处理规则

- 若功能语义未变，仅实现增强：
  - 旧用例继续适用
- 若功能语义已变：
  - 旧用例应标注为需修订或废弃
- 若新增正式能力：
  - 必须新增对应测试用例
- 不允许让测试方案长期脱离当前版本语义

### 2.2 用例级版本绑定规则

- 除非单条用例另有说明，本文所有 `HN-*` 用例默认适用于：`0.1.0`
- 若某条用例属于：
  - 观察性用例
  - 已知待实现差距
  - 后续版本能力必须在该用例中单独标注：
  - `适用版本`
  - 或 `版本说明`
- 版本升级后，不得只更新文档头部版本号而不检查单条用例适用性

## 3. 测试范围

当前阶段覆盖以下 6 个测试分组：

1. 查询层测试
2. 议题层测试
3. 任务层测试
4. 批次层测试
5. 主控架构测试
6. 汇总与回归策略

不包含：

- 业务代码功能测试
- 自动化测试框架实现
- 浏览器级 E2E 平台建设
- 多仓库协同测试

## 4. 当前功能覆盖矩阵

| 对象层 | 当前功能 | 当前口令 / 能力 | 覆盖状态 |
| --- | --- | --- | --- |
| 查询 | 高价值查询过滤、最近查询展示 | `最近查询` | 已覆盖，待观察展示优先级 |
| 议题 | 自动进入议题层、议题集合与单对象查看 | `议题列表`、`议题详情`、`升级为任务` | 部分覆盖 |
| 任务 | 任务态识别、总览、集合、单对象查看 | `当前状态`、`任务列表`、`任务详情` | 已覆盖，待补边界 |
| 转场 | 显式切换与回到、继续只恢复当前阶段 | `切换到任务`、`回到任务`、`继续` | 已覆盖，待补边界 |
| 批次 | 多任务编排判断与当前批次查看 | `是否适合多任务编排`、`当前批次` | 已覆盖，待补时机边界 |
| 主控架构 | 主控、子执行、子查询边界 | 主控前台响应、查询型子代理上限 | 部分覆盖 |
| 执行授权 | 开始、停止、提交结果 | `开始执行`、`停止执行`、`提交结果` | 已覆盖 |

说明：

- `部分覆盖` 表示已有主链路用例，但边界、反向或版本语义仍缺失
- `待补覆盖` 表示当前方案尚未把该能力独立收成明确用例

## 5. 执行原则

- 先测主链路，再测边界条件
- 先测查看与转场，再测编排与恢复
- 先保留人工可执行性，再决定哪些适合自动化
- 测试用例命名保持稳定，避免频繁重编号

## 6. 查询 / 议题 / 任务判定边界

### 6.1 查询

满足以下特征时，按查询处理：

- 只需要当下回答
- 回答后不需要继续跟进
- 不进入任务流也没有明显损失

### 6.2 议题

满足以下特征时，优先进入议题层：

- 当前不一定立刻执行
- 后续还需要回来处理
- 明显不能丢
- 与系统、规则、架构、稳定性、方案演进直接相关
- 还未明确到足以直接进入任务卡

### 6.3 任务

满足以下特征时，优先进入任务层：

- 已经明确要做
- 已经有清晰动作或产出物
- 已经可以进入边界 / 计划
- 不再只是“先记住这个问题”

### 6.4 判定边界测试目标

- 回答完即结束：查询
- 先记住、后面还要处理：议题
- 现在就应正式推进：任务

## 7. 优先级划分

### 7.1 P0 冒烟集

必须优先覆盖：

- 查询是否正确进入 `最近查询`
- 高价值输入是否能形成议题或直接进入任务卡
- 任务卡形成后，`当前状态` / `任务列表` 是否进入任务态
- `继续` 是否只恢复当前阶段
- `开始执行` 后是否进入执行锁定
- `当前批次` 在有/无批次时是否表现正确

### 7.1A Harness 内核最小自动回归基线

自 package-ready 前收敛阶段开始，Harness 内核至少应具备一组可重复执行的最小自动回归。

当前 V1 基线建议先覆盖：

- runtime state load / save
- plugin assembly
- plugin runtime resolution
- diagnostics 升级链关键节点

当前建议入口：

- `pnpm harness:test`

当前定位：

- 这不是完整测试平台
- 而是 package-ready 前最小自动回归基线

### 7.2 P1 主链路回归集

建议在每轮系统改动后回归：

- 查询 -> 议题 -> 任务
- 任务 -> 批次
- 插问 -> 不切任务
- 显式切换 -> 真转场
- 执行中切换 -> 被拒绝
- `skill_hit` 与 runtime 写入

### 7.3 P2 边界与异常集

在主链路稳定后补测：

- 无目标对象时的拒绝提示
- 已升级对象的重复操作
- 查询型子代理上限
- 批次 blocked / skipped 场景
- 状态恢复不一致

## 8. 测试分组与用例

## 8.1 查询层测试

### HN-QUERY-001 最近查询默认只展示高价值查询

- 目标：
  - 验证 `最近查询` 不回显流程口令、控制口令和查看口令
- 前置：
  - 会话中包含内容型查询和流程口令混合输入
- 操作：
  1. 连续输入 2 个内容型查询
  2. 连续输入 `继续`、`当前状态`、`任务列表`
  3. 查询 `最近查询`
- 预期：
  - 只显示高价值内容型查询
  - 不显示 `继续`、`当前状态`、`任务列表`

### HN-QUERY-002 最近查询默认返回最近 5 条

- 目标：
  - 验证默认展示数量为 5
- 前置：
  - 已有至少 6 条高价值内容型查询
- 操作：
  1. 查询 `最近查询`
- 预期：
  - 默认返回最近 5 条
  - 顺序从近到远

### HN-QUERY-003 子查询型问题优先进入最近查询

- 目标：
  - 验证长查询或需旁路分析的问题具有更高展示优先级
- 前置：
  - 会话内存在普通短问和长查询
- 操作：
  1. 提交一个普通短问
  2. 提交一个需要深分析的长查询
  3. 查询 `最近查询`
- 预期：
  - 长查询保留在最近查询中
  - 短问不一定进入展示

### HN-QUERY-004 最近查询空态正确提示

- 目标：
  - 验证没有可展示查询时的行为
- 前置：
  - 当前会话尚未形成高价值内容型查询
- 操作：
  1. 查询 `最近查询`
- 预期：
  - 明确提示当前没有可显示内容
  - 不误把流程口令当作查询结果

### HN-QUERY-005 当前状态、任务列表等查看口令不进入最近查询

- 目标：
  - 验证查看类口令不会污染最近查询
- 前置：
  - 会话中混入真实内容型查询与查看类口令
- 操作：
  1. 输入 1 条内容型查询
  2. 执行 `当前状态`、`任务列表`、`当前批次`
  3. 查询 `最近查询`
- 预期：
  - 只保留内容型查询
  - 不展示查看类口令

### HN-QUERY-007 sidecar-query 在最近查询中的展示优先级

- 适用版本：
  - `0.1.0`
- 版本说明：
  - 当前版本将该项视为展示策略观察项
  - 不把“稳定高优先级保留”写成 `0.1.0` 已完全收敛能力

- 目标：
  - 验证 sidecar-query 结束后，其对应长查询在 `最近查询` 中的展示优先级是否符合当前观察预期
- 前置：
  - 已有至少 1 条普通内容型查询
  - 已触发 1 条 sidecar-query 型长查询
- 操作：
  1. 完成一条 sidecar-query 型长查询
  2. 再补充若干普通内容型查询
  3. 查询 `最近查询`
- 预期：
  - sidecar-query 对应长查询应被优先关注并记录观察结果
  - 普通内容型查询按近到远排序
  - 若出现被后续普通查询快速挤掉的情况，应记录为 `0.1.0` 展示策略待收敛项，而不直接判定为主链路回归失败

## 8.2 议题层测试

### HN-TOPIC-001 高价值输入自动形成议题

- 目标：
  - 验证明确待处理需求会进入议题层
- 前置：
  - 当前无激活任务
- 操作：
  1. 输入一个明确但尚未进入任务卡的待处理事项
  2. 查询 `议题列表`
- 预期：
  - 新输入被识别为议题
  - `议题列表` 可见

### HN-TOPIC-002 议题详情显示单议题最小信息

- 目标：
  - 验证 `议题详情` 显示范围正确
- 前置：
  - 已有至少一个议题
- 操作：
  1. 查询 `议题详情：<议题名/编号>`
- 预期：
  - 显示议题名称、状态、来源、判断依据、升级建议
  - 不切换成任务详情

### HN-TOPIC-003 议题不存在时正确拒绝

- 目标：
  - 验证找不到议题时的行为
- 操作：
  1. 查询 `议题详情：不存在的议题`
- 预期：
  - 明确提示未找到
  - 明确说明“本次命令不会被执行”

### HN-TOPIC-004 议题升级后从议题列表主项转出

- 目标：
  - 验证 `升级为任务` 后议题层行为
- 前置：
  - 已有目标议题
- 操作：
  1. 执行 `升级为任务：<议题名/编号>`
  2. 查询 `议题列表`
- 预期：
  - 原议题不再作为主项出现
  - 不产生重复议题

### HN-TOPIC-005 低价值输入不应误判为议题

- 目标：
  - 验证普通短问、流程口令与查看口令不会误入议题层
- 前置：
  - 当前无激活任务
- 操作：
  1. 连续输入普通短问与流程口令
  2. 查询 `议题列表`
- 预期：
  - 不产生误判议题
  - 议题列表仍为空或仅保留真实高价值事项

### HN-TOPIC-006 高价值输入必须形成可读取议题实例

- 目标：
  - 验证高价值输入不只被内部理解，还必须形成可读取议题实例
- 前置：
  - 当前没有同名议题
- 操作：
  1. 输入一条明确与系统演进直接相关、且后续需跟进的问题
  2. 查询 `议题列表`
  3. 查询 `议题详情：<议题名/编号>`
- 预期：
  - 议题列表不为空
  - 新议题具备名称、状态、来源摘要、判断依据、升级建议
  - 议题详情可以读取该实例

### HN-TOPIC-007 明确要做的事项不应被长期滞留在议题层

- 适用版本：
  - `0.1.0`
- 版本说明：
  - 当前版本重点验证“系统是否已明确判定它更接近任务”
  - 不把“自动形成任务卡”作为 `0.1.0` 的既有实现前提

- 目标：
  - 验证“已明确要做”的输入不会被误当成普通议题长期搁置
- 操作：
  1. 输入一条目标和产出都已明确的事项
  2. 观察系统是否把它判定为更接近任务
- 预期：
  - 应明确判定其更接近任务而非普通议题
  - 若当前版本仍需人工进入任务卡，应在测试结果中标注为 `0.1.0` 已知差距
  - 不应把该行为误记为“当前版本已支持自动建任务卡”

## 8.3 任务层测试

### HN-TASK-001 任务卡形成后进入任务态

- 目标：
  - 验证正式任务卡形成后即为激活任务
- 操作：
  1. 形成任务卡草案
  2. 查询 `当前状态`
  3. 查询 `任务列表`
- 预期：
  - `当前状态` 不再显示空闲中
  - `任务列表` 不再显示无任务

### HN-TASK-002 任务详情显示单任务最小信息

- 目标：
  - 验证 `任务详情` 范围正确
- 前置：
  - 已有目标任务
- 操作：
  1. 查询 `任务详情：<任务名/编号>`
- 预期：
  - 显示任务编号、标题、类型、协作方式、阶段、执行状态、已完成项、阻塞点、下一步
  - 不切换为批次详情

### HN-TASK-003 继续只恢复当前阶段

- 目标：
  - 验证 `继续` 不自动跨阶段
- 前置：
  - 任务已停在边界中或计划中
- 操作：
  1. 中途插问
  2. 输入 `继续`
- 预期：
  - 返回原阶段
  - 不从边界直接跳到计划

### HN-TASK-004 切换到任务与回到任务

- 目标：
  - 验证显式转场口令
- 前置：
  - 存在多个未执行任务
- 操作：
  1. 执行 `切换到任务：<任务名/编号>`
  2. 执行 `回到任务：<任务名/编号>`
- 预期：
  - 未执行阶段可切换
  - 回到已存在任务
  - 不生成新任务

### HN-TASK-005 执行中转场被拒绝

- 目标：
  - 验证执行阶段转场限制
- 前置：
  - 当前任务已收到 `开始执行`
- 操作：
  1. 执行 `切换到任务：<任务名/编号>`
  2. 执行 `回到任务：<任务名/编号>`
- 预期：
  - 被拒绝
  - 提示当前任务仍在执行中
  - 提示如需转场，先 `停止执行`

### HN-TASK-006 任务卡最小结构完整

- 目标：
  - 验证正式任务卡最小结构足以支撑任务态识别
- 前置：
  - 新形成一个正式任务卡
- 操作：
  1. 检查任务卡是否包含任务标题、任务类型、当前阶段、协作方式
  2. 查询 `当前状态`
  3. 查询 `任务列表`
- 预期：
  - 当前状态可进入任务态
  - 任务列表可识别并展示该任务
  - 不因字段缺失退回空闲态

### HN-TASK-007 升级为任务后进入任务卡起点而不自动执行

- 目标：
  - 验证议题升级为任务后的起始状态
- 前置：
  - 已有一个可升级议题
- 操作：
  1. 执行 `升级为任务：<议题名/编号>`
  2. 查询 `任务列表`
  3. 查询 `当前状态`
- 预期：
  - 新任务进入任务列表
  - 处于已建卡、归类中、边界中或计划中
  - 不会自动进入执行中

### HN-TASK-009 被消息打断后继续恢复原执行点

- 适用版本：
  - `0.1.0`
- 版本说明：
  - 当前版本已明确稳定的是“继续恢复当前阶段，不跨阶段推进”
  - “恢复到原执行点”在 `0.1.0` 中仍属于增强目标，不作为既有实现前提

- 目标：
  - 验证普通消息不会终止当前任务，且 `继续` 至少能恢复到原执行阶段
- 前置：
  - 当前任务已进入执行阶段
  - 执行推进到可识别的中间点
- 操作：
  1. 中途发送一条普通查询或普通消息
  2. 查询 `当前状态`
  3. 输入 `继续`
- 预期：
  - 普通消息不会终止当前任务
  - 当前任务仍保持为原执行任务
  - `继续` 后恢复到原执行阶段，而不是跳到其他阶段或其他任务
  - 若当前版本还不能精确恢复到原执行点，应记录为 `0.1.0` 增强目标，而不直接判定为主链路回归失败

## 8.4 批次层测试

### HN-BATCH-001 是否适合多任务编排的时机判断

- 目标：
  - 验证该口令仅在分析后、边界前可执行
- 操作：
  1. 在正确时机执行
  2. 在错误时机执行
- 预期：
  - 正确时机给出拆分建议
  - 错误时机明确说明“本次命令不会被执行”

### HN-BATCH-002 当前批次正确显示最小批次信息

- 目标：
  - 验证 `当前批次` 显示范围
- 前置：
  - 已有激活批次
- 操作：
  1. 查询 `当前批次`
- 预期：
  - 显示当前任务、下个任务、已完成任务、blocked 任务、每个任务状态和协作方式

### HN-BATCH-003 无批次时正确拒绝

- 目标：
  - 验证无批次时的行为
- 操作：
  1. 在无激活批次时查询 `当前批次`
- 预期：
  - 明确提示没有激活批次
  - 明确说明“本次命令不会被执行”

### HN-BATCH-004 blocked / skipped 状态可追溯

- 目标：
  - 验证批次中阻塞与跳过状态可被记录和回看
- 前置：
  - 当前已有激活批次
- 操作：
  1. 制造一个最小 blocked 场景
  2. 将该任务标记为 skipped
  3. 查询 `当前批次`
- 预期：
  - blocked / skipped 记录可追溯
  - 不影响其他任务继续推进

### HN-BATCH-005 批次只在开始执行后激活

- 目标：
  - 验证“适合多任务编排”与“已有激活批次”是两件事
- 操作：
  1. 在计划中确认任务适合多任务编排
  2. 查询 `当前状态`
  3. 不执行 `开始执行`
- 预期：
  - 可以判断适合多任务编排
  - 但当前仍没有激活批次
  - 只有 `开始执行` 后才进入批次态

### HN-BATCH-006 是否适合多任务编排在生成边界前可正常执行

- 目标：
  - 验证 `是否适合多任务编排` 在正确时机可正常执行
  - 验证“命令可执行”与“结论是否适合”必须分开判断
- 前置：
  - 当前任务已形成
  - 尚未执行 `生成边界`
- 操作：
  1. 在当前阶段执行 `是否适合多任务编排`
  2. 观察命令是否被正常处理
  3. 观察最终判断结果
- 预期：
  - 命令本身应正常执行
  - 系统可返回“适合”或“不适合”的判断结果
  - 即使最终结论为“不适合”，也不能误报成“本次命令不会被执行”

## 8.5 主控架构测试

### HN-CONTROL-001 插问不切任务

- 目标：
  - 验证插问只作为旁路处理
- 前置：
  - 已有激活任务
- 操作：
  1. 中途提出不相关短问题
  2. 查询 `当前状态`
- 预期：
  - 当前主任务不变
  - 插问不会自动形成转场

### HN-CONTROL-002 无关长问题走旁路处理

- 目标：
  - 验证主控不被无关长问题直接拖住
- 前置：
  - 已有激活任务
- 操作：
  1. 中途提出无关长问题
- 预期：
  - 不覆盖当前任务
  - 按旁路问题处理

### HN-CONTROL-003 查询型子代理上限

- 目标：
  - 验证查询型子代理同时最多 1 个
- 前置：
  - 已有无关长查询正在处理
- 操作：
  1. 再提交第二个无关长查询
- 预期：
  - 不再启动新的查询型子代理
  - 提示等待、替换或停止当前查询任务

### HN-CONTROL-003A 单子查询完整生命周期

- 目标：
  - 验证单子查询从触发到结束的完整生命周期
- 前置：
  - 已有激活任务
- 操作：
  1. 中途提交一个无关长问题
  2. 观察是否进入旁路查询
  3. 查询结束后再查看 `最近查询`
- 预期：
  - 不覆盖当前任务
  - 旁路查询状态可解释
  - 长查询在最近查询中具有更高保留优先级

### HN-CONTROL-004 批次执行中的前台限制

- 目标：
  - 验证批次执行中不接受新的执行任务
- 前置：
  - 当前处于批次执行或主控调度中
- 操作：
  1. 发起新的执行类命令
- 预期：
  - 被拒绝
  - 明确说明“本次命令不会被执行”

### HN-CONTROL-005 开始执行与停止执行的授权边界

- 目标：
  - 验证执行授权和停止授权的硬边界
- 操作：
  1. 在未开始执行前尝试推动落盘动作
  2. 执行 `开始执行`
  3. 执行 `停止执行`
- 预期：
  - 未开始执行前不得落盘
  - `开始执行` 后进入执行阶段
  - `停止执行` 后立即停止推进并回报当前状态

### HN-CONTROL-006 普通消息打断不等于终止任务

- 目标：
  - 验证普通消息只会打断连续推进节奏，不会被误判成任务终止
- 前置：
  - 当前任务已进入执行阶段
- 操作：
  1. 在执行中发送一条普通消息
  2. 查询 `当前状态`
  3. 不执行 `停止执行`
- 预期：
  - 当前任务仍保持执行中
  - 不会被误记为已收口或已停止
  - 只有显式执行 `停止执行`，才算真正终止任务

## 8.6 执行授权测试

### HN-AUTH-001 未收到开始执行前不得落盘

- 目标：
  - 验证非授权口令不会触发真正落盘修改
- 前置：
  - 当前任务已形成
  - 尚未收到 `开始执行`
- 操作：
  1. 输入 `继续`、`继续执行`、`按方案做` 等非授权表达
  2. 观察系统行为
- 预期：
  - 只能继续分析、边界、计划或解释
  - 不得进入真实落盘修改
  - 不得把非授权表达误当成执行授权

### HN-AUTH-002 停止执行后立即生效

- 目标：
  - 验证 `停止执行` 属于最高优先级停止口令
- 前置：
  - 当前任务已进入执行阶段
- 操作：
  1. 执行 `停止执行`
  2. 观察系统是否继续推进
  3. 查询当前状态
- 预期：
  - 停止立即生效
  - 不继续落盘修改
  - 明确回报停止时的当前状态与已完成情况

## 9. 后续自动化建议

当前测试方案先以人工可执行为主，后续优先考虑自动化的部分：

- 查询 / 议题 / 任务 / 批次的输出结构校验
- 错误时机的拒绝行为校验
- 状态归属与阶段推进校验

不建议过早自动化的部分：

- 主控与子查询的真实交互体验
- 长查询响应质量
- 复杂上下文切换的可理解性

## 10. 完成标准

当以下条件满足时，可认为当前 Harness 测试方案初步可用：

- 查询、议题、任务、批次四层对象都有对应测试
- 主控架构有独立测试组
- 有 P0 冒烟集
- 有完整回归集
- 用例可由人稳定复现
- 用例与当前版本基线有明确绑定
