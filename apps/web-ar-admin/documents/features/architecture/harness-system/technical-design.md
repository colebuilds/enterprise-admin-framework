# Harness System 技术方案（最小版）

## 1. 设计目标

本方案只解决最小可运行问题：

- 如何给任务分类
- 什么叫任务完成
- 执行后至少要验证什么

本方案不尝试覆盖完整 runtime 平台、长期状态持久化或自动调度，只补最小运行时与改进收集能力。

## 2. 最小状态流

当前阶段定义如下最小状态流：

1. `intake`
   - 接收任务，不执行
2. `classify`
   - 判断任务类型和风险等级
3. `boundary`
   - 生成任务边界草案
4. `approval`
   - 等待边界确认与 `开始执行`
5. `plan`
   - 输出执行计划
6. `execute`
   - 在已确认边界内执行
7. `verify`
   - 按任务类型完成最小验证
8. `review-ready`
   - 输出结果、验证结论与未覆盖风险

当前阶段不定义 `merge-ready` 的独立工程门禁，只保留 `review-ready` 作为最小交付出口。

## 2.1 最小任务状态记录

当前阶段补充一个最小状态对象，用于支持长任务、架构任务和中途中断任务的接续。

最小状态字段只包含：

- `current_stage`
- `completed_items`
- `current_blocker`
- `next_step`

这些字段只用于说明任务当前走到哪里、哪些项已完成、当前卡点和下一步动作，不扩展为完整任务平台状态系统。

## 2.2 最小证据对象

当前阶段补充一个最小证据对象，用于支撑结果回报、验证结论说明与任务接续。

当前只认可以下最小证据类型：

- 命令输出
- 测试结果
- 手工回归说明
- 文档挂载 / 结构检查
- 已知风险说明

这些证据只要求能够说明“为什么当前可以认为已验证到某个程度”，不扩展为完整审计或自动采集系统。

## 2.3 最小运行时目录

当前阶段补充一个最小运行时目录：

- `.harness-runtime/`

该目录只用于保存运行时快照与反馈记录，不进入正式文档体系，也不要求遵守 `documents/` 下的 frontmatter 与挂载规则。

当前阶段的最小运行时文件包括：

- `current-batch.yaml`
  - 当前激活批次快照
- `current-task.yaml`
  - 当前激活任务快照
- `trial-feedback.yaml`
  - 真实任务试运行反馈
- `manual-validation.yaml`
  - 人工验证记录
- `events.log`
  - 关键检查点事件

为避免频繁变化的运行态数据进入正式提交，当前仓库只跟踪模板文件，实际运行时文件默认应加入忽略规则。

## 2.3.1 最小批次对象

当前阶段补充一个最小批次对象，用于承接“多任务批次”的外层控制面：

- 当前批次是什么
- 当前激活任务是谁
- 下一个任务是谁
- 哪些任务已完成
- 哪些任务已阻塞

最小批次对象只承接：

- `batch_id`
- `title`
- `goal`
- `current_task_id`
- `next_task_id`
- `task_order`
- `completed_tasks`
- `blocked_tasks`
- `notes`
- `tasks`

当前阶段的 `tasks` 条目只要求包含：

- `task_id`
- `title`
- `status`
- `depends_on`
- `agent_mode`
- `summary`

当前阶段的任务状态只支持：

- `pending`
- `in_progress`
- `completed`
- `blocked`
- `skipped`

该对象不扩展为历史批次仓库，不扩展为自动调度器，也不替代单任务的任务卡、边界、计划和验证。

## 2.3.2 系统级角色分层

当前阶段将多 agent 进一步上收为系统级角色分层：

- 主控 agent
  - 常驻前台
  - 负责查询响应、控制响应、批次调度、执行分配和最终收口
- 执行型子代理
  - 负责当前主任务或当前批次任务的实际执行
  - 可多个，但必须由主控直接管理
- 查询型子代理
  - 负责与当前主任务无关的长查询
  - 同时最多 1 个
  - 不改主任务状态

当前阶段的多 agent 协作默认采用：

- 主控 + 多子执行 + 单子查询

而不采用：

- 子执行再套子执行

执行层应保持扁平，避免形成过长通信链路。

## 2.4 最小写入策略

当前阶段不做实时写入，只在检查点写入最小运行时文件。

建议写入时机如下：

1. 任务卡草案基本成形后
2. 收到 `开始执行` 后
3. 关键阶段切换时
4. 准备 `提交结果` 前

当前阶段写入目标只包括：

- 当前批次恢复
- 任务恢复
- 试运行反馈
- 人工验证记录
- 关键事件回放

不扩展为事件流日志平台或完整任务数据库。

## 2.4.1 自动写入执行

当前阶段将最小写入策略进一步收敛为自动写入执行规则。

只允许以下 4 个自动写入检查点：

1. 任务卡草案基本成形后
   - 写入或更新 `current-task.yaml`
2. 收到 `开始执行` 后
   - 更新 `current-task.yaml`
   - 追加 `events.log`
3. 关键阶段切换时
   - 更新 `current-task.yaml`
   - 按需追加 `events.log`
4. `提交结果` 时
   - 写入或更新 `trial-feedback.yaml`
   - 若存在人工验证信息，补充 `manual-validation.yaml`

写入方式如下：

- `current-task.yaml`
  - 覆盖写
- `trial-feedback.yaml`
  - 覆盖写
- `manual-validation.yaml`
  - 覆盖写
- `events.log`
  - 追加写

当前阶段不自动记录每一轮对话，也不记录全量命令输出，只记录恢复和改进分析所必需的最小信息。

## 2.4.2 运行时读取

当前阶段以下场景应优先读取最小运行时文件：

- `当前状态`
  - 优先读取 `current-task.yaml`
- `任务列表`
  - 优先读取 `current-batch.yaml`
  - 再读取 `current-task.yaml`
- `诊断`
  - 可读取 `current-batch.yaml`、`current-task.yaml`、`trial-feedback.yaml` 与 `events.log`

运行时文件只作为最小恢复锚点与反馈锚点，不替代正式规则与任务边界。

## 2.4.3 最小切换规则

当前阶段的批次切换只定义以下最小规则：

1. 当当前任务满足以下条件时，可标记为 `completed`
   - 已有结果回报
   - 已形成最小证据
   - 当前阶段已收口
   - 无当前阻塞
2. 下一个任务应从 `task_order` 中选择第一个：
   - `status = pending`
   - 且 `depends_on` 已全部满足的任务
3. 若任务无法继续，应标记为 `blocked`
   - 并写入 `blocked_tasks`
   - 再由主代理决定跳过、等待或先切换独立任务

当前阶段不定义自动切换，不定义优先级系统，也不定义复杂依赖图。

## 2.4.4 批次执行中的前台限制

当系统已经进入批次执行或主控调度状态时，前台主控只接受：

- 查询类口令
- 控制类口令
- 普通短消息

当前阶段不允许在批次执行中直接进入新的执行型流程。若用户发起新的执行命令，应明确提示：

- 当前处于任务执行或批次执行中
- 本次命令不会被执行
- 若要继续，应等待当前任务收口或先执行 `停止执行`

此限制的目标是让主控始终保持可观察、可中断，而不是在前台再次进入长执行。

## 2.4.5 插问、旁路问题与任务转场

当前阶段进一步区分三类前台输入：

1. 插问
   - 指与当前任务并行出现的临时提问、短讨论或补充问题
   - 默认不改变当前主任务
2. 旁路问题
   - 指与当前主任务无关、且需要较长分析或检索的问题
   - 默认不覆盖当前主任务
   - 若需要深入处理，应进入单子查询代理
3. 显式转场
   - 指用户明确要求：
     - `切换到任务：...`
     - `回到任务：...`
     - `暂停当前任务，进入任务：...`
   - 只有命中显式转场表达时，才允许真正切换当前任务

当前阶段的默认规则如下：

- 新问题默认先按插问或旁路问题处理
- 当前主任务默认保持不变
- 未显式要求切换时，不得自动覆盖当前主任务
- 是否允许真正切换，还取决于当前是否进入执行阶段

该规则的目标是避免在主控架构下因插问或普通消息导致主任务丢失。

当前阶段的任务转场权限定义如下：

- 未执行阶段
  - 允许在命中显式转场表达后切换任务
- 执行阶段
  - 不允许直接切换任务
  - 如需切换，必须先 `停止执行`

其中，`开始执行` 是未执行与执行中的分界点。

## 2.5 改进收集范围

当前阶段的改进收集只覆盖以下检查点：

- 一句话任务进入与最小补问
- 任务卡草案形成
- 任务边界形成
- 执行计划形成
- 验证清单生成
- `重读 AGENTS.md`、`重载技能`、`诊断` 等维护口令触发
- `提交结果` 收口

这些记录主要用于判断：

- 指令是否给出了正确类型的信息
- 任务卡字段是否足够
- 边界和计划是否稳定
- 维护口令是否真有价值
- 当前规则是否还需要继续收敛

## 3. 任务入口分流

当前阶段将任务入口分为两条最小流：

### 3.0 任务卡默认规则

当前阶段，所有进入正式流程的任务都应先形成任务卡。

任务卡是默认任务对象，边界、计划、执行和验证都建立在任务卡之上。  
默认顺序如下：

1. 任务卡
2. 边界
3. 计划
4. 执行
5. 验证

若当前信息不足，应先完成最小补问，再形成任务卡，不应在没有任务卡的情况下直接进入边界、计划或执行。

任务卡还应作为协作方式判断的基础对象。

若任务可能涉及多 agent，任务卡阶段可先保留协作方式初判。若当前信息不足，可暂记：

- `agent_mode: 分析中`

随后应在边界或计划阶段明确：

- `agent_mode`
- `agent_split_reason`

若确定采用多 agent，还应在进入执行前明确：

- `agent_split_plan`

`生成计划` 时至少应说明：

- 当前是否可以使用多 agent
- 若仍不能定，原因是什么

在 `开始执行` 前，`当前状态` 应能看到最终协作方式判断结果。

### 3.1 普通任务流

适用于：

- `business-feature`
- `bugfix`
- `docs-only`
- 边界清晰的小型重构任务

默认流转方式为：

1. 一句话任务
2. 最小补问
3. 任务卡草案
4. 边界
5. 计划
6. 执行

### 3.2 架构任务流

适用于：

- `core-architecture`
- 拆包
- 迁移
- 共享层边界重划
- 平台能力建设

默认流转方式为：

1. 一句话任务
2. 最小补问
3. 架构拆解草案
4. 分阶段任务卡
5. 边界
6. 计划
7. 执行

### 3.3 最小判断线

- 若任务核心难点是“怎么实现”，优先走普通任务流
- 若任务核心难点是“边界、阶段、迁移路径或验证出口”，优先走架构任务流
- 若任务同时存在可并行盘点、可并行实施、文件边界可切分等特征，应在边界或计划阶段判断是否进入 `multi-agent`

对于 `core-architecture` 任务，当前阶段默认不应直接进入实现，而应先完成任务本体拆解与阶段划分。

## 4. 最小任务类型

当前先定义 4 类任务：

### 4.1 `business-feature`

适用场景：

- 新业务能力
- 业务逻辑增强
- 面向用户的功能迭代

### 4.2 `bugfix`

适用场景：

- 缺陷修复
- 异常行为修复
- 回归问题处理

### 4.3 `core-architecture`

适用场景：

- 核心架构演进
- 平台能力建设
- 基础设施或协作机制升级

### 4.4 `docs-only`

适用场景：

- 纯文档任务
- 文档治理
- 索引、规范、说明更新

## 5. 最小完成定义

### 5.1 `business-feature`

至少满足：

- 目标功能已落地
- 必要测试已补或已说明未补原因
- 验证结论明确
- 主要风险与未覆盖项已说明

### 5.2 `bugfix`

至少满足：

- Bug 现象或复现方式已说明
- 修复已完成
- 至少完成一次回归验证
- 未覆盖风险已说明

### 5.3 `core-architecture`

至少满足：

- 关键改动已完成
- 设计说明已形成
- 验证方式已说明
- 影响范围与风险已说明

### 5.4 `docs-only`

至少满足：

- 文档落点正确
- frontmatter 正确
- `updated` 已同步
- 最近一级 README 已挂载

## 6. 最小验证分级

当前阶段只定义 3 个验证层级：

### 6.1 `quick`

适用于低风险或局部任务。  
要求完成最小必要验证，例如局部检查、文档结构核对、相关测试或最小命令验证。

### 6.2 `standard`

适用于默认交付任务。  
要求在 `quick` 基础上补充与影响面对应的验证，并给出明确验证结论。

### 6.3 `high-risk`

适用于核心架构、基础设施、迁移和高影响面任务。  
要求在 `standard` 基础上补充关键路径验证与未覆盖风险说明。

## 6.4 人工验证最小要素

当任务需要人工验证时，当前阶段至少应记录以下内容：

- 触发是否正确
  - 执行某个口令或动作后，是否进入了正确功能
- 应有信息是否齐全
  - 输出是否包含该口令或功能应展示的信息类型
- 是否存在越界行为
  - 是否出现了本不应出现的附带动作或信息
- 是否易于理解
  - 输出是否足够清楚，便于人工快速判断正确性
- 结论
  - `pass` / `partial` / `fail`

当前阶段不单独建设人工验证平台，只要求其能作为最小证据与反馈的一部分被记录。

## 7. 验证失败分流

当前阶段在进入 `verify` 后，如验证未通过，只定义 3 类最小出口：

### 7.1 可修复后重试

适用于：

- 问题局部可控
- 不需要扩大任务边界
- 不需要新增权限或高风险动作

此时可继续在当前任务内修复，并重新执行最小验证。

### 7.2 必须停下并重新确认

适用于：

- 验证失败暴露出新的影响范围
- 需要扩大修改范围
- 需要新增命令权限
- 需要进入新的高风险动作

此时不应直接继续实现，而应先回到边界或计划确认阶段。

### 7.3 可带风险回报但不得宣称完成

适用于：

- 主体工作已完成
- 但验证未完全通过或覆盖不足
- 当前又不适合在本次任务内继续扩展

此时可回报当前结果、未覆盖风险与建议后续项，但不得将任务表述为已完成。

## 8. 任务类型与验证关系

- `business-feature`
  - 默认不低于 `standard`
- `bugfix`
  - 默认不低于 `standard`
- `core-architecture`
  - 默认不低于 `high-risk`
- `docs-only`
  - 默认使用 `quick`

## 9. 与现有体系的挂接原则

本方案只做路由，不复制现有细则。

- 边界与授权仍由 `AGENTS.md` 和 `FE-AI-001/002` 决定
- 多代理协作仍由 `FE-AI-003` 决定
- 质量动作仍应优先引用 `FE-QUALITY-*`

Harness 当前阶段只负责给任务提供统一交付框架。

系统级角色与任务级 `agent_mode` 的关系如下：

- 系统级角色
  - 说明谁负责前台、执行和旁路查询
- 任务级 `agent_mode`
  - 说明当前任务采用 `single-agent` 还是 `multi-agent`

两者不冲突：

- 主控 + 多子执行 + 单子查询
  - 是系统级运行架构
- `single-agent / multi-agent`
  - 是任务级执行策略

同样，任务转场规则也属于系统级运行约束：

- 插问和旁路问题
  - 默认不改变当前主任务
- 显式切换
  - 才改变当前主任务
