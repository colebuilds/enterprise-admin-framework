# Harness runtime 控制面 V1

## 1. 目标

本文档用于定义 `.harness-runtime/` 在 Harness 当前阶段的控制面 V1 设计，使 runtime 不再只是最小记录器，而是能承接任务对象、执行入口、验证状态与总收口状态的最小控制面。

本文档回答的问题是：

- runtime 控制面 V1 的职责是什么
- 哪些对象必须由 runtime 承接
- 哪些信息应以快照承接
- 哪些信息应以事件承接
- 当前 `.harness-runtime/*` 文件在 V1 中各自负责什么
- runtime 如何支撑后续自动拆分、自动执行、自动验收和总 MR 收口

## 2. 为什么 runtime 不能只作为记录器

若 runtime 只用于“记录发生过什么”，则后续系统仍会缺少统一控制面：

- 当前总任务是谁无法稳定恢复
- 当前中任务和可提交单元无法稳定定位
- 执行入口无法稳定区分 `manual_start / auto_start`
- 验证状态和总收口状态无法被统一消费
- skill 之间仍会靠对话上下文而不是统一对象协作

因此，runtime 必须从“最小记录器”升级为“最小控制面”。

## 3. runtime 控制面 V1 的职责

当前阶段，runtime 控制面 V1 负责：

- 承接当前激活的总任务
- 承接当前激活的中任务
- 承接当前激活的可提交单元
- 承接当前拆分方案
- 承接当前执行入口
- 承接当前验证状态
- 承接当前提交状态
- 承接总 MR 准备状态

当前阶段，runtime 仍不负责：

- 自动调度
- 数据库持久化
- 完整依赖图求解
- 多批次自动恢复
- 替代所有 skill 的执行逻辑

## 4. 核心承接对象

### 4.1 总任务对象

runtime 必须能稳定承接当前复杂需求对应的总任务对象，至少支持：

- 当前总任务是谁
- 当前总任务处于哪个阶段
- 当前总任务的总文档在哪里
- 当前总任务是否已进入总收口阶段

### 4.2 中任务对象

runtime 必须能稳定承接当前正在推进的中任务对象，至少支持：

- 当前中任务是谁
- 当前中任务属于哪个总任务
- 当前中任务是分析型还是执行型
- 当前中任务处于哪个阶段

### 4.3 可提交单元对象

runtime 必须能稳定承接当前正在执行或待验证的可提交单元，至少支持：

- 当前单元是谁
- 当前单元属于哪个中任务
- 当前单元的文件边界
- 当前单元的验证状态
- 当前单元是否已提交

### 4.4 拆分方案对象

runtime 必须能承接当前任务的拆分结果，以便系统恢复时仍知道：

- 当前总任务拆成了哪些中任务
- 当前中任务拆成了哪些可提交单元
- 当前拆分方案是否已被确认

### 4.5 收口对象

runtime 必须能承接最小收口信息，以支撑：

- 总验收是否已开始
- 总验收是否已完成
- 总 MR 是否已准备
- 总 MR 是否已提交

## 5. 快照与事件分工

### 5.1 快照负责“当前是什么”

快照适合承接当前状态，例如：

- 当前总任务
- 当前中任务
- 当前可提交单元
- 当前执行入口
- 当前验证阶段

快照回答的是：

- 现在处于什么状态
- 当前激活对象是谁

### 5.2 事件负责“发生过什么”

事件适合承接关键阶段切换，例如：

- 拆分方案确认
- 执行入口命中
- 验证开始
- 验证失败
- 提交完成
- 总验收开始
- 总 MR 准备完成

事件回答的是：

- 发生过哪些关键动作
- 状态为什么会走到当前形态

### 5.3 V1 不追求全事件化

当前阶段不把 runtime 设计成完整事件溯源系统。

V1 以“快照为主，事件为辅”为原则：

- 快照承接当前控制面
- 事件补关键轨迹

## 6. 当前 `.harness-runtime/*` 文件在 V1 中的角色

### 6.1 `current-task.yaml`

在 V1 中，`current-task.yaml` 应从“当前任务快照”升级为：

- 当前激活对象快照的核心入口

至少应能表达：

- 当前总任务
- 当前中任务
- 当前可提交单元
- 当前执行入口
- 当前阶段

### 6.2 `current-batch.yaml`

在 V1 中，`current-batch.yaml` 继续承接批次视角，但需要更明确支持：

- 总任务与中任务的组织关系
- 当前任务顺序
- 已完成 / blocked / 下一任务

### 6.3 `trial-feedback.yaml`

在 V1 中，`trial-feedback.yaml` 继续承接试运行反馈，但其结果应能反哺：

- 当前可提交单元
- 当前中任务
- 总任务验收判断

### 6.4 `manual-validation.yaml`

在 V1 中，`manual-validation.yaml` 继续承接人工验证，但不再只服务单任务场景，还要服务：

- 中任务人工验证
- 总任务总体验收

### 6.5 `events.log`

在 V1 中，`events.log` 继续承接关键检查点事件，但应重点记录：

- 拆分确认
- 执行入口命中
- 验证与提交检查点
- 总验收与总 MR 检查点

## 7. runtime 与其他模型的关系

### 7.1 与任务对象模型的关系

runtime 控制面必须直接承接：

- 总任务
- 中任务
- 可提交单元

否则任务对象模型无法真正进入系统执行层。

### 7.2 与收口模型的关系

runtime 控制面必须承接总任务收口所需的最小状态，例如：

- 总验收状态
- 总 MR 状态
- 子任务提交汇总状态

否则复杂需求虽然有收口规则，但无法被系统持续追踪。

### 7.3 与 skill 的关系

runtime 控制面不替代 skill，但应成为 skill 的统一状态承接面。

这意味着：

- task-flow skill 负责形成任务对象
- orchestration skill 负责形成拆分与协作方式
- runtime-recording skill 负责把这些对象写入控制面

## 8. 当前阶段的最小演进方向

runtime 控制面 V1 的目标不是一次性平台化，而是为后续改造提供最小、稳定的控制面骨架。

建议后续演进顺序为：

1. 先让当前快照文件承接三层任务对象
2. 再让事件文件承接关键检查点
3. 再补总验收与总 MR 状态
4. 再决定是否进入更重的状态存储设计

## 9. 当前阶段总结

Harness runtime 控制面 V1 可以压缩成一句：

它是一个以快照为主、事件为辅的最小控制面，用于承接总任务、中任务、可提交单元、执行入口、验证状态和总收口状态。

若没有这层控制面，后续自动拆分、自动执行、自动验收和总 MR 收口都无法稳定闭环。
