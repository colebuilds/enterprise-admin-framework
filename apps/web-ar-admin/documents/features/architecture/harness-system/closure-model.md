# Harness 总任务收口模型

## 1. 目标

本文档用于定义 Harness 在复杂需求场景下的最终收口方式，明确：

- 总任务何时进入收口阶段
- 总验收对象是什么
- 总 MR 如何组织
- 子任务提交如何汇总到总 MR
- 复杂需求何时可视为真正完成

## 2. 为什么需要单独定义收口模型

复杂需求拆成多个任务后，不能因为所有子任务都“做完了”就默认视为总任务已完成。

若缺少总任务收口模型，会出现以下问题：

- 子任务都提交了，但整体目标未验收
- 文档、验证、MR 关系不统一
- 总任务文档逐步失焦
- 复杂需求缺少最终统一交付出口

因此，总任务必须拥有独立的收口模型，而不能只依赖子任务完成状态。

## 3. 收口对象定义

复杂需求最终收口时，至少需要四类对象共同组成最终交付：

- 总任务
- 总文档
- 总验收
- 总 MR

这四类对象缺一不可。

## 4. 总任务收口阶段

### 4.1 总任务进入收口的前提

总任务进入收口阶段前，至少应满足：

- 拆分方案已明确
- 所有计划内中任务已进入完成或可验收状态
- 所有可提交单元已形成独立提交结果或已明确阻塞原因
- 总任务文档已可支撑最终回顾与验收

### 4.2 总任务收口不是自动等于完成

进入收口阶段只表示：

- 当前可以开始做总体验收
- 当前可以开始准备总 MR

并不表示总任务已经完成。

## 5. 总验收对象

### 5.1 总验收的定位

总验收是复杂需求的最终确认对象，用于回答：

- 原始总目标是否达成
- 子任务结果是否已形成完整交付
- 是否还存在残留风险、未闭环项或挂起项

### 5.2 总验收最小内容

总验收至少应包含：

- 总目标达成情况
- 子任务完成摘要
- 剩余风险
- 未完成项或延期项
- 总体验证结果
- 是否允许进入总 MR

### 5.3 总验收与子任务验证的关系

总验收不替代子任务验证。

子任务验证回答的是：

- 单个任务是否完成

总验收回答的是：

- 这些任务组合起来，是否构成完整交付

## 6. 总 MR 模型

### 6.1 总 MR 的定位

总 MR 是复杂需求最终对外展示与收口的统一交付对象。

总 MR 不是单个子任务 MR 的简单堆叠说明，而应承接：

- 总任务目标
- 总体验收结论
- 关键变更摘要
- 文档更新摘要
- 风险与后续项

### 6.2 当前阶段的默认组织方式

当前阶段默认采用：

- 子任务分别 `commit / push`
- 最终汇总成一个总 MR

这意味着：

- 子任务可以形成独立提交链路
- 总任务最终仍需要一个统一交付入口

### 6.3 总 MR 与子任务提交的关系

总 MR 至少应能关联：

- 已纳入本次复杂需求的提交单元
- 已完成的中任务
- 对应的总任务文档
- 对应的总验收结论

## 7. 子任务提交与总收口关系

### 7.1 子任务提交不是总任务完成判据

子任务完成 `commit / push`，只说明：

- 该子任务已形成提交结果

不说明：

- 总任务已完成
- 总 MR 已可提交
- 总体验收已通过

### 7.2 子任务完成后的总收口动作

当所有计划内子任务已完成提交后，系统仍应继续执行：

- 汇总子任务结果
- 汇总文档更新
- 汇总验证结果
- 生成总验收结论
- 组织总 MR

## 8. 复杂需求完成判定

复杂需求只有在同时满足以下条件时，才可视为真正完成：

- 总任务目标已达成
- 计划内中任务已完成或已显式转出
- 子任务提交结果已完成汇总
- 总文档已更新
- 总验收已完成
- 总 MR 已准备完成或已提交

若缺少上述任一项，则不应把复杂需求视为完成。

## 9. 对后续系统建设的意义

该收口模型将直接指导：

- runtime 控制面的总任务状态设计
- 自动提交流程的总 MR 组织
- 自动验收对象的结构设计
- 文档与提交的总控关系

若收口模型不先明确，后续多任务交付虽然能执行，但无法稳定形成最终闭环。

## 10. 当前阶段总结

Harness 的复杂需求最终收口必须同时承接：

- 总任务
- 总文档
- 总验收
- 总 MR

其中：

- 子任务负责形成局部提交结果
- 总任务负责形成最终交付结果

这两者不能互相替代。
