# Harness runtime 模板 V1

## 1. 目标

本文档用于定义 `.harness-runtime/*` 在当前阶段的模板结构方向，使 runtime 控制面 V1 可以进一步落到可改造的模板层。

本文档回答的问题是：

- 五类 runtime 文件分别承接什么
- 哪些字段来源于总任务 / 中任务 / 可提交单元
- 哪些检查点应写入哪些模板
- 快照文件与事件文件在模板层如何分工

## 2. 设计原则

### 2.1 模板服务于控制面，不服务于临时补丁

模板设计必须服从 runtime 控制面 V1 的目标，不应继续停留在“单任务最小记录器”的旧视角。

### 2.2 字段来源必须可追溯

每个模板字段都应尽量能说明来源于：

- 总任务
- 中任务
- 可提交单元
- 执行入口
- 验证状态
- 收口状态

### 2.3 当前阶段只定义字段方向，不直接进入最终实现

本文档只定义模板职责与字段方向，不直接把模板写死成最终代码或数据库结构。

## 3. 模板分层原则

当前阶段的 `.harness-runtime/*` 应分为两类：

- 快照类模板
- 事件类模板

### 3.1 快照类模板

快照负责承接“当前是什么”。

当前阶段包括：

- `current-task.yaml`
- `current-batch.yaml`
- `trial-feedback.yaml`
- `manual-validation.yaml`

### 3.2 事件类模板

事件负责承接“发生过什么”。

当前阶段包括：

- `events.log`

## 4. 模板职责与字段方向

### 4.1 `current-task.yaml`

#### 角色

`current-task.yaml` 是当前激活对象快照的主入口。

#### 必须承接的对象

- 当前总任务
- 当前中任务
- 当前可提交单元
- 当前执行入口
- 当前阶段

#### 字段方向建议

- 总任务标识与标题
- 中任务标识与类型
- 可提交单元标识与目标
- 当前状态
- 当前执行入口
- 当前文档路径
- 当前验证状态
- 当前提交状态

#### 字段来源

- 总任务对象
- 中任务对象
- 可提交单元对象
- 执行入口判断结果

### 4.2 `current-batch.yaml`

#### 角色

`current-batch.yaml` 负责批次视角与顺序视角。

#### 必须承接的对象

- 当前总任务下的中任务顺序
- 当前任务
- 下一任务
- 已完成任务
- blocked 任务

#### 字段方向建议

- 总任务标识
- 批次标题
- 当前任务标识
- 下一任务标识
- 已完成列表
- blocked 列表
- 每个任务的最小状态摘要

#### 字段来源

- 总任务对象
- 中任务对象
- 拆分方案对象

### 4.3 `trial-feedback.yaml`

#### 角色

`trial-feedback.yaml` 负责承接试运行后的反馈结果，并为总任务验收提供输入。

#### 必须承接的对象

- 当前可提交单元反馈
- 当前中任务反馈
- 总任务收口判断所需反馈摘要

#### 字段方向建议

- 关联任务标识
- 验证结果摘要
- 问题清单
- 风险摘要
- 是否建议继续推进

#### 字段来源

- 可提交单元验证结果
- 中任务验证结果
- 总任务验收前汇总结果

### 4.4 `manual-validation.yaml`

#### 角色

`manual-validation.yaml` 负责承接人工验证结果。

#### 必须承接的对象

- 单个可提交单元的人工验证
- 中任务的人工验证
- 总任务的总体验收结果

#### 字段方向建议

- 验证对象标识
- 验证类型
- 验证结论
- 待确认问题
- 人工备注

#### 字段来源

- 可提交单元
- 中任务
- 总任务验收对象

### 4.5 `events.log`

#### 角色

`events.log` 负责承接关键检查点事件。

#### 必须承接的事件类型

- 拆分方案确认
- 执行入口命中
- 执行开始
- 验证开始 / 失败 / 完成
- 提交完成
- 总验收开始 / 完成
- 总 MR 准备完成 / 提交

#### 字段方向建议

- 时间戳
- 事件类型
- 任务对象标识
- 当前阶段
- 当前状态
- 一句话原因或摘要

#### 字段来源

- 任务对象状态切换
- skill 命中事件
- 提交与验收检查点

## 5. 检查点与模板写入关系

### 5.1 任务卡成形后

建议至少写入：

- `current-task.yaml`
- `current-batch.yaml`（若已进入批次视角）

### 5.2 拆分方案确认后

建议至少写入：

- `current-task.yaml`
- `current-batch.yaml`
- `events.log`

### 5.3 命中执行入口后

建议至少写入：

- `current-task.yaml`
- `events.log`

### 5.4 验证阶段切换后

建议至少写入：

- `current-task.yaml`
- `trial-feedback.yaml`
- `manual-validation.yaml`（如有人工验证）
- `events.log`

### 5.5 总任务进入收口后

建议至少写入：

- `current-task.yaml`
- `trial-feedback.yaml`
- `manual-validation.yaml`
- `events.log`

## 6. 与对象模型和收口模型的关系

### 6.1 与任务对象模型的关系

runtime 模板必须承接：

- 总任务
- 中任务
- 可提交单元

否则三层对象无法真正进入 runtime。

### 6.2 与收口模型的关系

runtime 模板必须能表达：

- 总验收状态
- 子任务提交汇总状态
- 总 MR 准备状态

否则复杂需求无法在 runtime 中形成最终闭环。

## 7. 对后续改造的意义

该文档将直接指导后续：

- `.harness-runtime/templates/*` 文件改造
- `harness-runtime-recording` skill 写入逻辑调整
- 自动执行、自动验收与总 MR 相关检查点设计

若模板方向不先稳定，后续运行时改造会继续在“最小记录”与“控制面”之间摇摆。

## 8. 当前阶段总结

Harness runtime 模板 V1 的核心目标不是把模板写复杂，而是把快照和事件的职责分工稳定下来，让 runtime 真正能承接控制面所需的最小对象与最小状态。
