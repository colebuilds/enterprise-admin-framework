# Harness 系统蓝图与总体目标规范

## 1. 文档目标

本文档用于定义 Harness 系统的总体目标、能力边界、任务模型、自动执行闭环与后续改造方向。

本文档不是单一规则补丁，而是当前阶段的系统总蓝图，用于统一后续：

- 文档规范建设
- AI 协作规则建设
- skill 执行约束建设
- runtime 控制面建设
- 自动渐进式迭代能力建设

## 2. Harness 的系统定位

Harness 不是单一 skill，也不是单一文档规范，而是一套以项目文档、协作规范、skill 和运行时状态为基础的自动研发系统。

它的目标是：

- 理解复杂需求
- 自动拆解复杂任务
- 自动推进任务执行
- 自动生成与修正文档
- 自动验证与提交结果
- 在任务完成后反哺规范、skill 与 runtime

Harness 当前先服务于前端 monorepo H5 项目，但长期目标不局限于前端项目，也可扩展到后端或其他大型工程系统。

## 3. 核心输入模型

Harness 的输入不只限于自然语言需求，而是“需求线索集合”。

常见输入包括：

- 自然语言任务描述
- 项目地址或仓库上下文
- 项目代码
- `documents/`
- `AGENTS.md`
- `skills/*`
- PRD
- 图片、原型、截图
- 其他有助于形成明确需求的材料

系统应先基于这些输入完成：

- 项目理解
- 需求澄清
- 需求收敛

当需求明确后，再进入正式任务拆解与执行链路。

## 4. 任务对象模型

Harness 的任务对象模型至少分为三层：

### 4.1 总任务

用于承接一个复杂需求的总体目标与最终交付。

总任务必须始终有独立文档，否则随着拆分和演进推进，任务边界会逐渐模糊。

总任务至少负责：

- 总目标
- 总边界
- 总拆分方案
- 总文档
- 总验收
- 总汇总 MR

### 4.2 中任务

用于承接总任务下的中间阶段目标。

中任务既可以是：

- 执行型中任务
- 分析型中任务

当复杂需求暂时不具备直接拆分条件时，应先拆为分析型中任务，例如：

- 参考实现分析
- 依赖盘点
- 现状识别
- 架构可行性分析

分析型中任务完成后，可进一步支撑正式任务拆分。

### 4.3 可提交单元

Harness 自动拆解的最小单元是可提交单元。

一个可提交单元应具备：

- 可独立边界
- 可独立验证
- 可独立提交
- 可独立回滚

分析型中任务若形成了闭环结果，也可以作为独立提交单元。

## 5. 复杂需求拆分模型

### 5.1 复杂需求必须在自动执行前完成拆分

复杂需求不能在执行后黑盒拆分。

正确顺序应为：

1. 先分析需求与项目现状
2. 判断是否具备直接拆分条件
3. 生成拆分方案
4. 人工确认拆分方案
5. 再进入自动执行

### 5.2 不具备拆分条件时不得强制拆分

若当前仍缺少拆分依据，不得强行拆分一组伪任务。

典型场景包括：

- 需要先参考其他项目实现
- 需要先理解现有功能
- 需要先盘点依赖关系
- 需要先形成架构可行性判断

此时应先生成分析型中任务，而不是直接强拆执行型任务。

### 5.3 拆分方案必须透明

复杂需求在进入自动执行前，必须先输出可被人工确认的拆分方案。

拆分方案的展示粒度可随任务复杂度动态调整，但必须足够清晰，至少能支撑：

- 任务数量判断
- 边界判断
- 包或模块影响面判断
- 依赖关系判断
- 风险判断
- 是否允许进入自动执行判断

## 6. 自动执行模型

### 6.1 自动执行不是默认黑盒执行

Harness 的自动执行承接的是“已确认拆分后的任务单元”，不是“未经解释的大任务黑盒执行”。

自动执行前，应先完成：

- 任务明确
- 拆分明确
- 边界明确
- 计划明确
- 人工确认复杂任务拆分方案

### 6.2 自动执行默认闭环目标

在大多数正常场景下，Harness 自动迭代的默认目标应是：

- 自动执行
- 自动验证
- 自动 code review
- 自动 commit
- 自动 push
- 自动形成总 MR

### 6.3 自动执行入口分层

执行入口分为：

- 人工入口
- 自动入口

若任务命中自动执行条件，则进入自动执行链路。  
若任务未命中自动执行条件，则回到人工授权链路。

## 7. 验证与 Code Review 模型

### 7.1 验证不只等于测试

Harness 的验证应至少包括：

- 功能验证
- 测试验证
- 边界验证
- 文件范围验证
- 计划一致性验证

### 7.2 自动 code review 的位置

自动 code review 应发生在 `commit` 前，作为提交前自审的一部分。

但它不替代：

- 提交后 review
- MR 层 review
- CI 层验证

### 7.3 提交门槛

任务进入提交前，至少应确认：

- 结果已完成
- 最小验证已完成
- 文件范围符合边界
- 无关文件已排除
- 不依赖尚未完成的人工验证

## 8. 文档反哺模型

### 8.1 文档是系统知识底座

Harness 依赖文档理解项目背景、架构、规范和演进过程。  
因此文档不是静态说明，而是系统运行的知识底座。

### 8.2 出现歧义时默认自动反哺

只要架构实现与文档出现歧义，就应自动反哺回文档，避免文档与架构长期漂移。

默认策略：

- 一般歧义：自动改文档
- 偏差过大：人工接入确认

### 8.3 文档反哺也是任务结果的一部分

任务完成后，系统应自动判断是否需要：

- 新增文档
- 更新文档
- 沉淀规范
- 修正文档

## 9. 人工接入阈值

Harness 的默认方向是大多数场景自动推进，少数高风险场景人工接入。

当前已明确需要人工接入的场景包括：

- 反复验证不通过
- 架构改动大

### 9.1 架构改动分级

建议分为四级：

- `L1`
  - 轻微结构改动
- `L2`
  - 中等架构改动
- `L3`
  - 重大架构改动
- `L4`
  - 架构重写级

当前约定：

- `L1-L2`
  - 可自动推进
- `L3-L4`
  - 必须人工接入

## 10. 总任务收口模型

复杂需求最终收口时，不应只停留在“多个子任务都做完了”。

Harness 要求复杂需求最终至少形成：

- 总任务
- 总文档
- 总验收
- 总汇总 MR

当前约定的大需求交付方式为：

- 子任务分别 `commit / push`
- 最终汇总成一个总 MR

## 11. runtime 控制面目标

当前 `.harness-runtime/` 仍偏最小记录能力，但 Harness 的长期目标并不只是“恢复记录”。

未来 runtime 更应作为控制面，承接：

- 当前总任务
- 当前中任务
- 当前可提交单元
- 拆分方案
- 执行入口
- 验证状态
- 提交状态
- 总 MR 状态

## 12. 当前系统差距清单

### 12.1 保留

当前可继续保留的内容包括：

- `documents/` 文档体系
- `AGENTS.md`
- 已有 AI 协作规范
- 文档治理规则
- skill 分层思路
- `.harness-runtime/` 目录承载方向

### 12.2 重构

当前应优先重构的部分包括：

- 任务流主链路
- 自动执行机制
- 验证系统
- 文档反哺机制

### 12.3 重写

当前建议准备重写的部分包括：

- 任务对象模型
- runtime 控制面模型
- skill 之间的统一协作协议

### 12.4 新增

当前仍需新增的部分包括：

- Harness 总蓝图体系
- 任务对象模型规范
- 总任务 / 中任务 / 可提交单元模型
- 总 MR 收口规则
- 版本号自动更新规则
- 迭代日志规则
- 规范、skill、runtime 的自演进规则

## 13. 自演进目标

Harness 的长期目标不是只完成任务，而是自动连：

- 文档规范
- AI 协作规范
- skill
- runtime

一起演进。

在大多数场景下，Harness 应自动迭代。  
在不确定、冲突、高风险或高等级架构改动场景下，再接入人工讨论。

## 14. 当前阶段总结

Harness 当前已经具备：

- 规则体系雏形
- 文档体系雏形
- skill 分层雏形
- 自动执行入口规则雏形

但距离最终目标仍存在明显差距。

当前阶段最重要的事情不是继续零散补丁，而是：

- 先建立总蓝图
- 再基于蓝图重构任务模型、runtime 控制面和 skill 协议
