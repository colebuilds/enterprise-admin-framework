# 主题 Token 重构实施计划

> **For agentic workers:** REQUIRED SUB-SKILL: Use `superpowers:executing-plans` to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** 收敛当前后台的主题状态、token 输出和 adapter 链路，让全局 theme state 成为唯一真相源，并由统一 token 驱动 Naive UI、Tailwind、品牌层与公共组件。

**Architecture:** 本次实施按 `v1` 收敛范围推进，不一次性铺满全部语义 token。执行顺序固定为“主题状态收敛 -> token 输出层 -> Naive / DOM / Tailwind / Brand adapter -> 公共组件迁移 -> 业务页面跟进”。当前代码已完成主链路落地，`--primary-*` 与 `--app-*` 运行时 bridge 均已退役，`designSetting` store 与 `DESIGN_SETTING` 常量也已删除，不改 `--n-*`。

**Tech Stack:** `Vue 3`、`Pinia`、`Naive UI`、`Tailwind CSS 4`、`TypeScript`、`Vitest`、`@vueuse/core`、`pnpm`

> **Note (2026-04-30 closeout):** 本文档以下 task/step 保留为实施过程记录。若与当前代码状态冲突，以源码现状为准：`create-legacy-theme-vars.*` 已删除，`--primary-*` bridge 已退役，`src/store/modules/designSetting.ts` 与 `DESIGN_SETTING` 已删除。

# 1. 本轮边界

## 1.1 In Scope

1. 收敛唯一主题主状态源到 `projectStore.appearance`
2. 建立 `resolved theme -> token output -> adapter` 链路
3. 输出 `v1` 全局 token
4. 让 `Naive UI`、`Tailwind`、品牌层、离散 API、表格工具层统一消费新链路
5. 迁移第一批公共组件，阻止旧命名继续扩散

## 1.2 Out of Scope

1. 不重命名 `Naive UI` 的 `--n-*`
2. 不在本轮一次性迁移全部业务页面
3. 不因主题改造重构无关视觉结构
4. 不在本轮直接删除全部 `DESIGN_SETTING` 历史缓存

# 3. 实施原则

1. **先收敛状态，再迁消费面**
   - 任何页面迁移前，先完成 `resolved theme`、token 输出和 adapter 收口。
2. **只冻结 `v1` 语义 token**
   - 长期全局公开语义 token 控制在 `40-45` 个左右。
   - 菜单、表格、Naive 细粒度项优先下沉到 adapter。
3. **旧桥先留，防止回归**
   - `--primary-*` 在公共组件迁移完成前必须保留。
4. **测试先行**
   - 先补主题链路测试，再修改全局运行时文件。
5. **每阶段都可回滚**
   - 每个阶段都要以“旧链路仍可工作”为边界，避免一次性切断。
