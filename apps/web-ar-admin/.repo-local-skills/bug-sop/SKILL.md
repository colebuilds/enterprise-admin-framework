# Bug SOP

## 目标

- 把“从 bug 列表进入，到方案卡、实施计划、实现、评审、验证交接和输出回填清单”的整轮缺陷处理流程跑标准化。
- 这份私有 skill 只负责执行顺序和交付约束，不替代正式规则源。

## 适用场景

- 用户让你处理一轮 bug 列表
- 用户提到禅道 bug、我的 Bug、缺陷清单、回归清单
- 任务需要先整理清单，再分类、产出方案卡，并把后续工作路由到 PIR
- 任务需要输出可对账的处理前/处理后清单
- 任务需要人工本地验证、SIT 验证和禅道回填交接

不优先用于：

- 纯需求开发
- 无 `bug_id` 的探索性问题
- 只做只读查询、不进入修复流程
- 单纯讨论流程而不准备执行

## 输入要求

- 至少应有可读取的 bug 列表、一组明确 bug_id，或一段能定位当前缺陷范围的描述
- 若输入不足以形成处理前清单，先要求补充最小范围信息

## 执行步骤

1. 先读取正式规则源：
   - `AGENTS.md`
   - `documents/README.md`
   - `documents/standards/quality/FE-QUALITY-006-Bug处理闭环规范.md`
   - `documents/standards/ai-collaboration/FE-AI-008-AI-Bug处理协作规范.md`
   - `documents/guides/development/AI协作缺陷处理执行指南.md`
   - 若任务描述体现为“继续上一轮 / 跨 session 恢复 / 接着做 batch”，则优先检查 `/.local-work/session-context/LATEST_PATH.txt`，并读取其中指向的最新交接快照
2. 读取 bug list，生成处理前清单。
3. 先进入 bug 详情页，读取正文、截图、附件、录屏和历史记录，并结合当前项目页面、字段、接口和代码结构做一轮映射分析。
4. 若当前项目证据仍不足，可继续查看原型项目 `/Users/cola/Documents/gs/ar-saas` 的对应功能实现；查看前先同步最新代码，优先使用 `git pull --ff-only`。
5. 在完成详情深化分析后，再审查每条 bug 的信息完整度，将仍然信息不足的项移入待测试补充清单，不阻塞其余 bug。
6. 对可处理 bug 做分类与批次决策。
7. 在进入正式修复前，先输出每条可处理 bug 的“需求理解结论”和“拟修复方案”，便于人工审查；若涉及数据来源策略切换、固定集合替代动态列表等较大改动，必须在方案里明确指出。
8. 为每条可处理 bug 或每个已确认批次生成 `Bug 修复方案卡`，至少包含：证据来源、当前实现说明、目标口径说明、需求理解结论、具体修复方案、代码落点、明确不做什么、风险点、待确认项。
9. 若测试提出的改动意见明确，但当前实现本身也合理，且争议点落在产品到底要哪种实现口径，则将该问题标记为 `待产品确认`，输出当前实现合理性说明、测试建议口径、拟修改实现策略和待确认问题点，在确认前不进入正式修复。
10. 对已形成方案卡且边界清楚的问题，将后续工作路由到 `pir-planner`；由 `pir-planner` 负责把方案卡转成实施计划。
11. 当实施计划已批准后，再进入 `pir-implementer` 做实际修改；不要由 `bug-sop` 直接承担完整实现。
12. 当实现完成后，再进入 `pir-reviewer` 做独立评审。
13. 在 PIR 实施与评审完成后，再输出处理后交付清单，供 SIT 验证与禅道回填使用。
14. 若当前批次准备进入正式提交，且仓库存在本地提交口径 skill（如 `commit-local`），则在生成或 amend commit message 前，先路由到该本地 skill；不要把只在本地生效的提交规则写回共享 `.codex/skills/*`。

## 风险与禁止项

- 不要跳过清单化直接修代码
- 不要只凭标题和列表摘要就判定信息不足
- 不要因为单条信息不足阻塞整轮
- 不要把“当前实现也合理、但测试建议要求另一种产品口径”的问题，误当成普通可直接修复 Bug
- 不要默认从旧工作分支直接切新 bug 分支
- 不要默认直接操作禅道写状态
- 不要在人工本地验证前默认进入正式提交
- 不要让未处理 bug 从清单中消失
- 如果当前工作区不干净，可以继续做清单化和分类，但在分支准备前必须暂停并反馈

## 输出要求

- 每轮至少输出：
  - 处理前清单
  - 待测试补充清单
  - 本轮处理批次建议
  - Bug 修复方案卡
  - 处理中跟踪清单
  - 处理后交付清单
  - PIR 路由建议
  - 如进入提交阶段，则补充本地提交 skill 路由建议
- 所有清单都必须：
  - 保留 `bug_id`
  - 保留 `bug_title`
  - 与来源列表可逐条对账
  - 保留未处理项、阻塞项、待补充项
