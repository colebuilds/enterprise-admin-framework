# Branch Init

## 目标

- 在开发开始前，用最小上下文建立当前分支与目标模块的总框架认知。
- 优先识别模块入口、共享能力入口和兼容性风险点，不做无边界全量扫描。
- 为后续实现提供稳定的“先看哪里、再按需展开”的初始化结论卡。

## 适用场景

- 从 `main` 切出新分支后，准备开始开发。
- 新开 session 进入当前分支，需要先熟悉当前代码现场。
- 接手某个具体模块前，需要先看架构入口、公共组件、公共函数和共享能力入口。
- 用户明确提出：
  - `先熟悉项目再开发`
  - `先看公共组件 / 公共函数`
  - `按 branch-init 初始化当前分支`
  - `按 branch-init 初始化某个模块`

## 不适用场景

- 不替代会话快照恢复。
- 不替代 `bug-sop`、PIR 或实施计划。
- 不自动改代码，也不自动生成长篇架构总结。
- 不做全仓库知识灌入或无目标全量扫描。

## 输入要求

- 至少明确一个目标：
  - 当前分支
  - 或目标模块 / 业务域
- 若用户未提供目标模块，则优先按当前分支相对 `main` 的差异热点初始化。

## 执行步骤

1. 读取 `AGENTS.md`，提取硬约束和通用开发准则。
2. 检查当前 git 现场：
   - 当前分支
   - `git status --short`
   - 若当前分支不是 `main`，查看相对 `main` 的主要差异范围与热点目录
3. 只建立入口地图，不展开大段细节：
   - 目标模块入口：路由、页面入口、`shared` / `context` / `useXxxPage`、测试入口
   - 共享能力入口：`src/components/index.ts`、`src/hooks/index.ts`、`src/utils/`、`src/store/modules/`
   - 需要时再读取 `documents/reference/2026-04-30-公共能力入口索引.md`
4. 若一级共享入口未命中，再查看同业务域的相似模块，找局部可复用实现。
5. 输出初始化结论卡，只包含：
   - 当前现场
   - 模块入口
   - 共享能力入口
   - 复用 / 兼容提醒
   - 建议先读的 3-6 个文件
6. 初始化完成后，具体实现细节按任务推进过程按需加载。

## 上下文预算

- 初始化阶段默认不超过 8-12 个文件。
- 优先入口文件，不先全仓库关键词漫游。
- 只输出以下三档判断：
  - `已确认可复用`
  - `疑似可复用，需继续查看实现`
  - `未在入口层发现现成能力`
- 不输出“全仓库不存在可复用能力”这类过度承诺结论。

## 输出要求

- 输出必须简短、结构化，优先用短卡片而不是长文。
- 初始化目标是“建立地图”，不是“灌入所有知识”。
- 若发现潜在共享改动点，必须明确提示兼容性风险。
